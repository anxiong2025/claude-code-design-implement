PART 1 · 认知地基 · 第 3 章

需求到能力：Agent Loop 背后的产品决策

读完这一章，你能
  • 把"帮我改一下代码"这种模糊需求，翻译成一个可执行的能力循环
  • 说清楚"能对话的 AI"和"能干活的 Agent"差在哪一步，以及那一步为什么难
  • 理解 Agent Loop（模型思考→调工具→观察结果→再思考）这个决策，如何决定了产品的能力上限
  • 知道"循环什么时候停"是个产品问题，而不只是技术问题

先看一个再普通不过的场景。你对一个 AI 编程助手说：

"我这个函数报了个空指针，你帮我看看，顺便把相关的测试也补上。"

一句话，但它其实是一连串动作：找到那个函数→读懂它→定位空指针的来源→改代码→找到对应的测试文件→补测试→可能还要跑一遍确认没坏。作为产品经理，你要回答的第一个问题不是"用什么模型"，而是：一个 AI，凭什么能把这一句话拆成这么多步、并且一步步做完？

这一章讲的就是这个"凭什么"。它背后是 AI Agent 产品最核心的一个设计决策——Agent Loop。理解了它，你就理解了这类产品能做什么、不能做什么的边界从何而来。

一、"能对话"和"能干活"，隔着一个循环

很多人对 AI 产品的想象还停留在 ChatGPT 刚出来的样子：你问一句，它答一句。这叫一问一答（single-turn）。它很有用，但有个天花板——它只会"说"，不会"做"。

你让一个纯对话模型"帮我改代码补测试"，它最多给你一段它猜测的代码，因为它根本没看过你的文件。它不知道你的函数长什么样、测试用的什么框架、项目里还有哪些相关文件。它在闭卷答题。

关键区分

一问一答：模型接收输入，生成一段文本，结束。它能"说得对"，但不能"做成事"，因为它无法与真实世界（你的文件、你的终端、你的数据库）交互。

Agent（智能体）：模型不仅能生成文本，还能决定去调用一个工具（比如"读取文件 X"），拿到结果后再继续思考，如此往复，直到把事办完。它能"做成事"。

从"能说"到"能做"，中间隔着的那道坎，就是让模型能够行动、并且能根据行动的结果继续行动。而实现这一点的机制，就是一个循环。

给产品经理的话

当你评估一个 AI 需求时，先问自己一句：这个需求需要 AI "说"就够了，还是需要它"做"？如果只需要说（写文案、答疑、总结），一问一答就够，别过度设计。如果需要做（改文件、查数据、执行操作），你就进入了 Agent 的领域——成本、复杂度、风险都会上一个台阶。这是你要向老板和团队交代清楚的第一件事。

二、Agent Loop：一张图看懂"AI 怎么干活"

Claude Code 处理你那句"帮我改代码"时，内部跑的就是下面这个循环。它不复杂，但每一步都是一个产品决策。

图 3-1 Agent Loop：从需求到完成的一次循环
这个循环里，模型每转一圈都在做同一个判断：任务办完了吗？没办完，就再调一个工具、再看一次结果。办完了，就退出循环、把结果交给你。产品的"智能感"，正来自这个"自己决定下一步做什么"的能力。

把这张图翻译成大白话，Claude Code 面对"帮我改代码补测试"时是这样走的：

  1. 第一轮思考："我得先看看这个函数。"→ 调用【读文件】工具。
  2. 看到文件后再思考："哦，空指针是因为这个参数没判空。我得改这里。"→ 调用【编辑文件】工具。
  3. 改完再思考："需求里还说了补测试。测试文件在哪？"→ 调用【搜索】工具找测试文件。
  4. 找到后再思考："在这，我来加一个针对 null 的测试用例。"→ 再次调用【编辑文件】。
  5. 最后一轮思考："改完了，测试也补了，没有新动作要做了。"→ 退出循环，告诉你："已修复并补充了测试。"
这就是产品的能力上限

注意：这个 Agent 能做的所有事，都是"思考 + 它手上有的工具"的组合。它能改代码，是因为它有【编辑文件】这个工具；它要是没有【跑测试】的工具，它就永远只能"改完假装跑过了"。作为 AI PM，你决定给 Agent 配哪些工具，就等于在决定这个产品的能力边界。这是第 6 章要展开的重点，这里先埋下伏笔。

三、为什么是"循环"，而不是"一张流程图"

你可能会问：既然步骤这么清楚，为什么不干脆写死一个流程——"先读文件、再改、再补测试"——非要搞个循环让模型自己决定？

因为真实需求是无法预先穷举的。用户的下一句可能是"帮我看看这个报错"，也可能是"把这个模块重构成异步的"，还可能是"这段代码为什么这么慢"。你不可能为每一种可能都画一张固定流程图。

决策固定流程 vs. 动态循环
场景
产品要处理开放式、无法预先枚举的用户需求
选项 A
固定流程（工作流）：产品经理预先设计好每一步。可控、可预测，但只能处理设计过的场景，遇到没想到的需求就卡死。
选项 B
动态循环（Agent）：让模型每一步自己决定下一步做什么。灵活、能应对开放需求，但过程不完全可预测，可能绕路、可能失控。
取舍
Claude Code 选了 B——因为"帮人写代码"本质是开放式的。代价是：你放弃了对每一步的精确控制，换来了"能处理没预设过的任务"。
源码印证
主循环用 while(true) 反复调用模型，每轮根据模型返回里有没有 tool_use 决定是继续还是退出
这是一个真实的产品分水岭

"工作流（Workflow）"还是"智能体（Agent）"，是当下每个 AI 产品团队都要吵一架的问题。它没有标准答案，取决于你的场景有多"开放"：需求越封闭、对可预测性要求越高（如金融、医疗的合规流程），越该倾向固定工作流；需求越开放、越需要随机应变（如编程助手、通用研究），越该倾向 Agent 循环。很多成熟产品是两者混用：主干用工作流保证可控，局部用 Agent 处理灵活性。你要做的，是判断你的需求落在光谱的哪一端。

四、"循环什么时候停"，是个被低估的产品问题

循环最危险的地方在于：它可能不停。模型可能陷入"改了又改""反复读同一个文件""绕圈子"的状态。对用户来说，这就是"转圈半天没反应，还烧着钱"。所以"何时停止"绝不只是工程细节，它直接决定用户体验和成本。

一个成熟的 Agent 产品，至少要设计这么几种"刹车"：

设计循环的四种停止条件
正常完成
模型这一轮不再要求调工具，说明它认为任务办完了——这是理想的退出。
用户喊停
用户按下中断（Esc）。Agent 必须能被随时打断——这是"人还在方向盘上"的底线。
触碰上限
轮数、Token、时间任一超过预算，强制停下。防止无限循环烧钱。
出错中止
工具连续报错、或模型请求了不被允许的操作，安全地中止并向用户说明。
给产品经理的话

写这类产品的需求文档时，"停止条件"和"正常流程"一样重要，甚至更重要。你要明确回答：任务卡住了怎么办？用户想中途叫停怎么办？烧了太多钱要不要自动停？这些"异常路径"如果不设计，上线后就是投诉和账单爆炸的来源。一个经验法则是：先设计怎么停，再设计怎么跑。

五、本章小结：一句话需求背后的三个决策

回到开头那句"帮我改代码补测试"。现在你应该能看清，让 AI 把这句话做完，产品团队在背后至少做了三个关键决策：

  • 决策一：用循环，不用一问一答。这让 AI 从"能说"变成"能做"，是 Agent 产品的地基。
  • 决策二：用动态循环，不用固定流程。这让产品能应对开放式需求，代价是放弃了对每一步的精确控制。
  • 决策三：给循环装好刹车。正常完成、用户喊停、触碰上限、出错中止——四种停止条件共同保证产品可用、可控、不烧钱。

这三个决策，共同定义了这个 AI 产品"能做什么、不能做什么、会不会闯祸"。而它们全都发生在你写下第一行代码之前——这正是 AI 产品经理的主战场。
