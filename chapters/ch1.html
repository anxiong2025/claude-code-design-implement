第一章

Agent Loop（核心循环）

本章目标

把一句"用户请求"变成"能自己调工具、自己判断何时收工、最后给出结论"的 Agent。读完这章，你会掌握所有 Agent 系统最底层、也最不可省略的那个结构——循环，并能对照 Claude Code 真实源码写出一个可运行的单 Agent。

在拆源码之前，先建立一个贯穿全书的心智模型。很多人以为 Agent 是某种神秘的"会自己思考的程序"，但当你看进 Claude Code 的核心，会发现它朴素得惊人：

核心心智模型

一个 Agent 就是「一个 while 循环 + 一个消息数组」。循环每转一圈，就问一次模型"下一步做什么"；模型要么调用工具（循环继续），要么给出答案（循环结束）。所谓"智能"，是模型贡献的；Claude Code 这套工程代码的职责，只是把这个循环安全、可靠、可中断地转起来。

下面这张图就是这句话的动画版——点「下一步」，跟着走完一个完整的两轮对话，看左边的控制流怎么循环、右边的 messages[] 怎么逐轮长大。本章接下来的每一节，都是在拆解这张图里的某一步。

本章的五个小节，就是把这句话拆成五个必须回答的工程问题：状态存在哪（1.1）、循环按什么范式转（1.2）、模型的输出怎么实时接住（1.3）、工具结果怎么还给模型（1.4）、循环怎么安全地停下来（1.5）。

1.1 消息列表作为状态载体

先问第一个根本问题：Agent 的"记忆"存在哪里？

这个问题之所以关键，是因为大模型有一个反直觉的特性——它是无状态的。每次调 API 都是一次全新的、互不相识的请求，模型不会"记得"上一轮你们聊了什么。那多轮对话里的上下文靠什么活下去？

Claude Code 的答案朴素到近乎粗暴：用一个数组把所有说过的话都存起来，每轮结束追加新内容，下一轮把整个数组重新丢给模型。记忆不在模型里，而在调用方手里。这个数组，就是源码中 State 对象的 messages 字段（query.ts:205）：

// query.ts:205
type State = {
  messages: Message[]   // ← 完整对话历史，Agent 唯一真正持久的状态
  turnCount: number     // 轮次计数（1.5 节讲）
  // ...还有 8 个字段，全是"恢复机制"的账本，下面解释
}

初学者读源码时最容易被 State 里那一堆字段吓退。这里给一个减负的读法：这 10 个字段里，只有 messages 是"业务状态"，其余 9 个几乎全是"恢复用的临时账本"——记录"上一轮为什么要重试""compact 压缩到哪了""max_tokens 恢复过几次"。它们每轮基本都会重置，属于第二章 Harness 的话题。第一章你只需盯住 messages 这一个字段。

循环每转完一圈，末尾把三类新消息拼进数组，构成下一轮的输入（query.ts:1716）：

// query.ts:1715-1716
const next: State = {
  messages: [
    ...messagesForQuery,   // ① 本轮送入模型的历史
    ...assistantMessages,  // ② 本轮模型返回的内容（含 tool_use）
    ...toolResults,        // ③ 本轮工具执行结果（以 user 角色注入）
  ],
  turnCount: nextTurnCount,
  // ...其余字段重置
}
state = next   // 下一轮迭代从这里读
数据流 · 悬停高亮 下一轮 messages = 三段拼接 — 对照上面 query.ts:1716
每轮循环末尾，三段消息按 [...历史, ...模型输出, ...工具结果] 拼成新数组——这就是 Agent「记忆」增长的全部机制。

一个容易被忽视、但很能说明设计意图的细节：拼进去的是 messagesForQuery，不是 state.messages 原样。messagesForQuery 是原始历史经过预处理（压缩、截断、裁剪超大工具结果）之后的版本。这意味着——"完整历史"和"真正喂给模型的历史"是两回事。第一章我们默认它们相等，把它当"全量追加"来理解；第二章会看到 Harness 如何在这一步动手脚来控制 token。

设计启示

给开发者把"状态"和"计算"彻底分离：模型负责无状态的计算，状态收敛到调用方的一个数组里。这带来三个白捡的好处——可持久化（存下数组就能续上对话，见第八章 Session）、可回放（重放数组就能复现任何一轮）、可裁剪（要省 token，动数组就行）。凡是你自己要设计的 Agent，第一件事就是想清楚"状态数组长什么样"。

给产品经理"AI 记不住上文"从来不是模型缺陷，而是产品有没有把历史完整带上的工程选择。当你评估一个 Agent 产品的上下文能力时，真正该问的是："它的历史数组保留策略是什么？超长了怎么裁？"——这决定了体验的连贯性上限。

1.2 ReAct 框架

有了状态载体，下一个问题是：循环每一圈到底按什么套路转？Claude Code 用的是业界公认的 ReAct 范式。

ReAct（Reasoning + Acting）由 Google 在 2022 年的论文提出，是当今几乎所有主流 Agent 框架的理论底座。它的核心洞见是：不要让模型一口气想完再行动，而是让"想"和"做"严格交替——每做一步，就把结果喂回去，让模型基于真实反馈想下一步。三个交替的动作是：

  • Thought（思考）——模型用自然语言推理当前局面，决定下一步干什么
  • Action（行动）——模型发起一次工具调用（在 Claude API 里就是一个 tool_use block）
  • Observation（观察）——工具执行结果以 tool_result 回注，模型"看到"结果，进入下一轮 Thought

三步一圈，循环往复，直到模型在某一轮的 Thought 里判断"任务完成了"，不再发出 Action 为止。Claude Code 的实现就是一个 while(true)（query.ts:307）——正是章首那张可以点的循环图。每次迭代严格对应一个完整的 Thought → Action → Observation 周期，落到源码是这六步：

  1. 预处理 messages（压缩 / 裁剪，第二章展开）
  2. [Thought] 调用模型、流式接收：text block 是推理文字，tool_use block 是行动决定
  3. 本轮出现 tool_use 了吗？—— 否 就 return 'completed'（循环唯一的正常出口）
  4. [Action] 执行工具
  5. [Observation] 把 tool_result 注入 messages
  6. turnCount++，检查是否超 maxTurns，然后 continue 回到第 2 步

到这里，"何时停"这个问题浮出水面：循环怎么知道该转下一圈，还是该收工？直觉上你会想：看模型返回的 stop_reason 是不是 'tool_use' 不就行了？这正是所有人自己实现时都会踩的坑。Claude Code 源码在这里留了一句醒目的注释（query.ts:554）：

// query.ts:554-556
// Note: stop_reason === 'tool_use' is unreliable -- it's not always set correctly.
// Set during streaming whenever a tool_use block arrives — the sole
// loop-exit signal. If false after streaming, we're done ...

翻译过来：stop_reason 这个字段不可靠，不总是被正确设置，不能拿它当循环判据。Claude Code 的做法是自己维护一个布尔标志 needsFollowUp——在流式接收过程中，只要亲眼解析到一个 tool_use block，就把它置为 true（query.ts:832-834）：

// query.ts:826-835
if (message.type === 'assistant') {
  assistantMessages.push(message)
  const msgToolUseBlocks = message.message.content.filter(
    content => content.type === 'tool_use',
  )
  if (msgToolUseBlocks.length > 0) {
    toolUseBlocks.push(...msgToolUseBlocks)
    needsFollowUp = true   // ← 循环是否继续，只认这个信号
  }
}

"我实际收到了工具调用"是自己观察到的事实，"服务端告诉我它停下是因为工具调用"是外部的声明——Claude Code 选择相信前者。

设计启示

给开发者这是一条能迁移到任何系统的防御式设计原则：控制流的判据，优先用"你亲自观测到的事实"，而不是"上游承诺的元数据"。元数据可能缺失、可能语义漂移、可能因协议升级而失真；而"我这轮到底解析出没有工具调用"是确定的。你自己写 Agent 时，别把循环命脉押在 stop_reason 上。

1.3 流式响应的逐块处理

上一节说 needsFollowUp 是"在流式接收过程中"设置的——这句话背后是一个绕不开的机制：模型的输出不是一次性返回的，而是一条持续到达的流。

调用模型返回的是一条 SSE（Server-Sent Events）流，内容一块一块（block by block）陆续到达。Claude Code 的策略是：每拼装完一个 content block，就立刻把它 yield 出去，绝不等整条 message 收完。为什么要这么做？两个产品级的理由：

  • 体验——用户能看到文字一个字一个字冒出来，而不是盯着转圈等十几秒
  • 可中断——正因为是逐块处理，用户随时按 Ctrl+C，循环能在下一个 block 的边界干净地停下（这正是 1.5 节 aborted_streaming 退出路径的由来）

流式处理的核心是一个状态机，按到达的事件类型逐步拼装每个 block：

for await (const part of stream) {
  switch (part.type) {
    case 'content_block_start':
      // 新 block 开始。tool_use 的 input 先初始化为空字符串，
      // 因为它的参数是流式拼接的 JSON，还没收全
      blocks[part.index] = { ...part.content_block, input: '' }
      break

    case 'content_block_delta':
      // 增量到达，追加到对应 block：
      //   text_delta       → block.text  += delta.text
      //   input_json_delta → block.input += delta.partial_json
      break

    case 'content_block_stop':
      // 一个 block 收全了 → 立刻组装成 AssistantMessage 并 yield
      // query.ts 收到后马上检查它是不是 tool_use，决定 needsFollowUp
      yield assembledMessage
      break
  }
}

这里有两个值得记住的工程细节：

第一，tool_use 的参数是"流式拼接的 JSON 字符串"。模型不会一次性给你一个完整的参数对象，而是把 {"city": "北京"} 这段 JSON 一个片段一个片段地发过来，拼在 input 字符串上。只有等 content_block_stop、整段收全了，才能 JSON.parse 成真正的参数对象。这解释了一个常见困惑："为什么不能边收工具参数边执行？"——因为在收全之前，你手里的参数是不完整、无法解析的。

第二，逐块 yield 意味着一条 assistant message 可能包含多个 block：先来一段 text（模型的推理），再来一个或多个 tool_use（模型的决定）。这也是为什么模型能在一轮里同时发起多个工具调用——下一节就用到这一点。

设计启示

给产品经理"流式输出"不是锦上添花的动效，而是 Agent 可交互性的地基。有了逐块流，才谈得上"实时展示进度""随时打断""看着它一步步干活"这些让 Agent 从黑盒变成协作者的体验。评估 Agent 产品时，"能不能中途喊停、喊停后状态是否干净"是一条硬指标。

1.4 工具调用结果注入

模型发出了 tool_use，Claude Code 执行完工具，拿到了结果——现在轮到最后一个环节：怎么把结果还给模型？

答案里藏着一个初学者常觉得"反直觉"的协议约定：工具结果不是以某种"系统消息"或"函数返回"的身份传回，而是被包装成 tool_result block、以 user（用户）角色追加进消息列表。这是 Anthropic API 的强制协议：每个 tool_use 必须有一个 tool_use_id 完全相同的 tool_result 与之配对，且必须紧跟在那条 assistant 消息之后。

下面是一次完整 ReAct 循环之后，messages 数组的真实形态（顺便演示 1.3 说的"一轮里发起多个工具调用"）：

# 用户要"查北京和上海的天气再比较"，模型一轮里发起两个工具调用

[
  # ── 用户请求
  {"role": "user", "content": "帮我查北京和上海的天气，然后比较一下"},

  # ── [Thought+Action] 模型推理，并同时发起两个 tool_use
  {"role": "assistant", "content": [
    {"type": "text",     "text": "我来同时查询两个城市..."},
    {"type": "tool_use", "id": "tu_01", "name": "get_weather", "input": {"city": "北京"}},
    {"type": "tool_use", "id": "tu_02", "name": "get_weather", "input": {"city": "上海"}}
  ]},

  # ── [Observation] 两个结果以 user 角色注入，靠 tool_use_id 配对
  {"role": "user", "content": [
    {"type": "tool_result", "tool_use_id": "tu_01", "content": "晴，18°C"},
    {"type": "tool_result", "tool_use_id": "tu_02", "content": "多云，22°C"}
  ]},

  # ── 下一轮 Thought：模型基于结果给出最终回答，不再发 tool_use → 循环结束
  {"role": "assistant", "content": [
    {"type": "text", "text": "北京晴 18°C，上海多云 22°C，上海更暖..."}
  ]}
]
动画 · 并发 tool_use ↔ tool_result：靠 id 配对，可以乱序并发 — 小球在跑，两条通道同时进行

把结果放回数组之前，还要过一道 normalizeMessagesForAPI（query.ts:1396）——它负责把 Claude Code 内部的消息格式转成 API 认的"线上格式"：过滤内部专用的虚拟消息、合并相邻的 user 消息（某些后端如 Bedrock 的兼容要求）、清理超大媒体、剥掉内部字段。这层"内外格式转换"你现在不必细究，只需知道：内部消息模型 ≠ API wire 格式，中间隔着一层规整化。

最后点一个上面例子已经埋下的伏笔：同一轮的多个工具调用是并发执行的（read-only 工具走并发，并发上限由环境变量 CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY 控制，默认 10）；而会改动文件/状态的 write 类工具必须串行。为什么这样区分、调度器怎么实现，是第三章"工具系统"的主题。

设计启示

给开发者注意这个协议设计的精妙：用 tool_use_id 做配对，而不依赖顺序。正因如此，多个工具才能乱序并发、谁先返回都不影响正确性——只要 id 对得上就能归位。你设计任何"发起—回执"式的异步接口时，都值得学这一招：用显式 ID 配对，而非隐式顺序。

1.5 maxTurns 熔断与全部退出路径

循环的正常出口只有一个：模型不再发 tool_use（completed）。但"正常"是理想情况——万一模型陷入死循环，一遍遍调工具却始终给不出结论呢？任何 while(true) 都必须配一个熔断闸，否则就是一颗定时炸弹。这就是 maxTurns。

先说清 turnCount 的语义，它容易被误解：它从 1 起步，含义是"即将向模型发起第几次调用"——初始用户请求是 turn 1，第一次工具 follow-up 是 turn 2，以此类推。每当有工具结果、循环准备再转一圈时，在末尾 +1 并检查上限（query.ts:1705）：

// query.ts:1679, 1705-1711
const nextTurnCount = turnCount + 1

if (maxTurns && nextTurnCount > maxTurns) {
  yield createAttachmentMessage({ type: 'max_turns_reached', maxTurns })
  return { reason: 'max_turns', turnCount: nextTurnCount }
}

completed 和 max_turns 只是最常见的两个出口。一个成熟的循环，退出路径远不止"正常结束"和"跑太久"——它要应对中断、报错、上下文超限、Hook 拦截等各种现实。下面是 Claude Code 全部的 Terminal 退出路径：

reason触发条件
completed模型正常完成：本轮无 tool_use，needsFollowUp = false（唯一"正常"出口）
max_turnsnextTurnCount 超过 maxTurns 上限 —— 防死循环的熔断闸
aborted_streaming模型流式响应过程中用户中断（Ctrl+C）
aborted_tools工具执行过程中用户中断
prompt_too_long上下文过长，且所有恢复路径（compact 压缩等）都已耗尽
model_error模型调用抛出无法恢复的异常
image_error图片尺寸/处理错误
blocking_limittoken 总量超过硬性阻断上限（自动压缩关闭时）
stop_hook_prevented / hook_stoppedHook 显式要求停止（第七章）
出口图 · 悬停 循环的全部出口 — 把鼠标移到某条出口上，看它从哪来
一个成熟的循环，出口不止"跑完了"——它要为中断、超限、报错、Hook 拦截各留一条有名字的退路。

与"退出"相对的还有一组"不退出、而是恢复后继续转"的内部分支（源码里叫 Continue transition），例如 max_output_tokens_escalate（输出被截断 → 提高 max_tokens 上限重试）、reactive_compact_retry（上下文爆了 → 紧急压缩后重试）。它们不是本章重点，但值得先埋个印象：Claude Code 的循环里，"出错"绝大多数时候的第一反应不是退出，而是想办法恢复着继续跑。这套恢复机制正是第二章 Harness 的核心价值。

设计启示

给开发者把一堆 if 散落各处，不如像 Claude Code 这样：让循环体的每个出口都返回一个带 reason 的结构化结果。退出路径一旦被显式枚举成一张表，就变得可测试、可监控、可穷尽——你能对着这张表逐条问"这个分支我处理了吗"。

给产品经理Agent 会失败，这是必然，不是意外。产品的成熟度，恰恰体现在"失败得体不体面"：是无声地卡死，还是明确告诉用户"我达到最大步数了/上下文太长了/被你打断了"？上面这张退出路径表，本质上就是一份"Agent 会以哪些方式收场"的产品清单——每一种都对应一句该给用户的话。
