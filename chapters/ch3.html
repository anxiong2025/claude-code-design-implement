第三章

工具系统——Agent 的手

实现目标：Agent 能调 Bash、读写文件等多种工具，且读操作并发、写操作串行，互不干扰。

引言 · 为什么工具系统需要专门一章来讲

第一章讲 Agent Loop 时，工具调用被压缩成一行话："执行工具、拿到结果"。但这句话背后藏着这套架构里最复杂的工程决策之一。

考虑这样一个场景：模型在同一轮里同时发出三个工具调用——Read("src/query.ts")、Grep("isConcurrencySafe")、Edit("src/Tool.ts", ...)。如果三个调用串行执行，慢；如果三个调用全部并发，Read 和 Grep 没问题，但 Edit 修改文件时，如果同时有另一个工具也在写同一个文件，结果就是损坏的文件。

这就是工具系统必须解决的核心矛盾：最大化并发（速度）× 保证写操作的互斥（正确）。

Claude Code 的解决方案不是一把大锁，而是通过工具接口上的两个谓词——isConcurrencySafe 和 isReadOnly——让每个工具在定义阶段就声明自己的调度属性，调度器据此在运行时把一组工具调用分批（partition）：连续的只读工具合并成一个并发批次，每个写工具单独占一个串行批次。读并发、写独占，两者不需要手动协调。

本章目标

读完这章，你会理解：Tool 接口是什么、buildTool 为什么是必要的工厂抽象、调度器怎么用 partitionToolCalls 实现"批内并发、批间串行"、MCP 外部工具怎么适配进这套接口，以及 ToolSearch 如何解决"工具太多、首屏太慢"的加载问题。

本章的分析路径：先从接口定义（3.1）入手，理解一个 Tool 必须承诺什么；再看工具注册表（3.2），理解运行时是如何把所有工具组装在一起的；然后是调度器（3.3），这是本章最关键的工程核心；最后是 MCP 接入（3.4）和 ToolSearch 懒加载（3.5）两个拓展机制。

3.1 Tool 接口——每把手的完整契约

在 Claude Code 里，一个"工具"不是一个函数，而是一个实现了 Tool<Input, Output, P> 接口的对象（Tool.ts:281）。这个接口足足有 40+ 个成员——乍看吓人，但如果按职责归类，每个字段就各就其位了：

接口解剖 Tool 接口的六个职责域 — 悬停某域查看代表字段
Tool 接口的 40+ 个成员可归入六域：身份、模式约定、执行、调度属性、权限、渲染。其中"调度属性"和"权限"是 Claude Code 架构最独特的两处设计。

对照这张图，先抓最关键的几个成员：

// Tool.ts:281 — 精简版，只保留本章必须理解的字段
type Tool<Input, Output, P> = {
  name: string                          // 唯一标识，模型靠这个名字发起调用

  inputSchema: Input                    // Zod schema，双重用途：
                                        //   1. 校验模型传来的参数
                                        //   2. 生成发给模型的 JSON Schema 描述

  call(                                 // 工具主逻辑
    args: z.infer<Input>,
    context: ToolUseContext,
    canUseTool: CanUseToolFn,
    parentMessage: AssistantMessage,
    onProgress?: ToolCallProgress<P>
  ): Promise<ToolResult<Output>>

  isConcurrencySafe(input: z.infer<Input>): boolean  // 调度关键：是否可与其他工具并发
  isReadOnly(input: z.infer<Input>): boolean          // 是否只读（影响权限预检）
  isDestructive?(input: z.infer<Input>): boolean      // 是否不可逆（删除 / 覆盖）

  validateInput?(input, ctx): Promise<ValidationResult>   // 前置校验（语义层）
  checkPermissions(input, ctx): Promise<PermissionResult>  // 权限检查

  maxResultSizeChars: number            // 结果超过此值则落盘，模型收到路径 + 预览
  mapToolResultToToolResultBlockParam(  // 把 Output 序列化成 API 要求的格式
    content: Output, toolUseID: string
  ): ToolResultBlockParam
}

两个地方值得在这里多解释：

inputSchema 的双重用途。 Zod schema 不只是运行时校验，它还是文档。Claude Code 在组装 System Prompt 时，会把每个工具的 inputSchema 自动序列化成 JSON Schema 格式，作为"这个工具怎么用、有哪些参数"的说明发给模型。这意味着：你定义 schema 的用词和结构，直接影响模型会不会正确调用这个工具。字段名取对了、描述写清楚了，就是在给模型写说明书。

maxResultSizeChars 的设计意图。 工具结果可能很大——一个 Bash("find . -name '*.ts'") 可能返回几十 KB 的路径列表。把这么大的结果直接塞进消息数组，一轮就能把 token 耗尽。maxResultSizeChars 设了一道闸：结果超限时，内容落盘存文件，消息里只剩路径 + 前几行预览（toolResultStorage.ts）。注意 FileReadTool 把这个值设为 Infinity（FileReadTool.ts:342）——原因是 FileRead 的输出本身就是文件内容，再存一份等于创造循环依赖（存进去的路径被模型再 Read，又存进去……）。

3.1.1 权限流水线：validateInput → checkPermissions → call

每次工具调用在进入 call() 之前，必须过两道关（toolExecution.ts:683）：

// toolExecution.ts（简化）
async function checkPermissionsAndCallTool(tool, input, ctx, ...) {
  // ── 第一道：语义校验（input 层）
  const isValid = await tool.validateInput?.(parsedInput, ctx)
  if (isValid?.result === false) {
    return [errorResult(isValid.message)]   // 直接拒绝，不走权限
  }

  // ── 第二道：权限决策（canUseTool / hooks / 用户确认）
  const permissionDecision = await canUseTool(tool, parsedInput, ctx)
  if (permissionDecision.behavior !== 'allow') {
    return [deniedResult()]
  }

  // ── 第三步：执行
  const result = await tool.call(parsedInput, ctx, canUseTool, msg, onProgress)
  return [resultMessage(result)]
}

validateInput 负责语义层面的前置检查——例如"文件路径不能是绝对路径"、"Bash 命令不能包含危险的 rm -rf /"这类业务规则，它与权限无关，失败直接返回错误不弹权限框。checkPermissions 则是工具级别的自定义权限逻辑（全局权限框架在第七章），例如 MCP 工具在这里统一返回 passthrough 让上层权限系统接管。

设计启示

给开发者把"参数对不对"（validateInput）和"能不能做"（checkPermissions）分成两个关卡，是个很值得学习的分层习惯。前者是无副作用的纯校验，可以在任何时机调用；后者涉及用户交互和上下文状态，调用有代价。把两者混在一起会导致"弹了权限框才发现参数本来就错了"的体验问题。

3.2 buildTool——消灭样板的工厂

你不会直接手写一个实现了 40+ 成员的 Tool 对象。Claude Code 提供了 buildTool(def) 工厂函数（Tool.ts:783），它接受一个"可省略部分成员"的 ToolDef，自动填入合理的默认值，返回一个完整的 Tool：

// Tool.ts:757 — 默认值定义
const TOOL_DEFAULTS = {
  isEnabled:          () => true,
  isConcurrencySafe:  () => false,   // ← 默认不安全（fail-closed）
  isReadOnly:         () => false,   // ← 默认写操作（fail-closed）
  isDestructive:      () => false,
  checkPermissions:   () => Promise.resolve({ behavior: 'allow', updatedInput }),
  toAutoClassifierInput: () => '',   // ← 默认不参与安全分类器
  userFacingName:     () => name,
}

// Tool.ts:783
export function buildTool<D extends AnyToolDef>(def: D): BuiltTool<D> {
  return {
    ...TOOL_DEFAULTS,
    userFacingName: () => def.name,
    ...def,                          // def 的字段覆盖默认值
  } as BuiltTool<D>
}

注意默认值的方向：isConcurrencySafe 和 isReadOnly 都默认 false——fail-closed（失效即关闭）。一个工具如果没有显式声明"我是安全的"，就被当作写操作、串行执行。这是防御式设计：宁可损失一些并发速度，也不要因为漏标导致数据损坏。

对比三个典型工具的声明：

// FileReadTool.ts:373
isConcurrencySafe() { return true  },  // 只读，可与任意工具并发
isReadOnly()        { return true  },

// FileEditTool.ts — 未声明，走默认值
// isConcurrencySafe → false（不可并发）
// isReadOnly        → false（写操作）

// BashTool.tsx — 动态判断
isConcurrencySafe(input) {
  return this.isReadOnly?.(input) ?? false
},
isReadOnly(input) {
  const hasCd = commandHasAnyCd(input.command)
  const result = checkReadOnlyConstraints(input, hasCd)
  return result.behavior === 'allow'   // 静态分析命令内容
}

BashTool 的处理最有意思：它不是静态地声明"总是可以"或"总是不行"，而是在运行时解析命令内容来动态决定。ls、cat、grep 这类命令静态分析为只读，就能享受并发；touch、rm、任何包含 cd 的命令则被保守地判定为写操作，必须串行。

概念 · Fail-Closed（失效即关闭）

Fail-Closed 是一种安全设计原则：当系统无法判断某个操作是否安全时，默认拒绝（而不是默认允许）。与之对立的是 Fail-Open，即"不确定就放行"。

在工具调度里，选择 fail-closed 意味着：一个新工具如果忘记声明 isConcurrencySafe，不会意外地与其他工具并发、造成状态损坏；代价是可能错失一些并发机会。这个代价远小于静默的数据破坏，所以是正确的权衡。

3.3 读并发与写独占——调度器的核心

现在来看本章最关键的工程核心：调度器是如何把一组工具调用编排成"读并发、写串行"的执行计划的。

整个调度逻辑在 toolOrchestration.ts 里，由三个函数组成：runTools（主入口）、partitionToolCalls（分批器）、runToolsConcurrently 和 runToolsSerially（执行器）。

3.3.1 partitionToolCalls：分批

分批算法是一个贪心分组（toolOrchestration.ts:84）：从左往右扫描工具调用列表，把相邻的"并发安全"工具合并进同一个批次，只要遇到一个"不安全"的工具就新开一个批次：

// toolOrchestration.ts:84
function partitionToolCalls(
  toolUseMessages: ToolUseBlock[],
  toolUseContext: ToolUseContext,
): Batch[] {
  return toolUseMessages.reduce((acc: Batch[], toolUse) => {
    const tool = findToolByName(toolUseContext.options.tools, toolUse.name)
    const parsedInput = tool?.inputSchema.safeParse(toolUse.input)
    const isConcurrencySafe = parsedInput?.success
      ? (() => {
          try { return Boolean(tool?.isConcurrencySafe(parsedInput.data)) }
          catch { return false }   // ← 解析失败也 fail-closed
        })()
      : false

    // 连续的"安全"工具合并进同一批次
    if (isConcurrencySafe && acc[acc.length - 1]?.isConcurrencySafe) {
      acc[acc.length - 1]!.blocks.push(toolUse)
    } else {
      acc.push({ isConcurrencySafe, blocks: [toolUse] })
    }
    return acc
  }, [])
}

这个算法最重要的特性：它保持顺序，不做全局重排。如果模型发出了 [Read, Read, Edit, Read]，分批结果是 [{Read, Read}, {Edit}, {Read}]，而不是 [{Read, Read, Read}, {Edit}]。为什么不做全局优化？因为模型发出工具调用的顺序本身可能有语义——Edit 之前的两次 Read 是为了读取它要修改的内容，Edit 之后的 Read 是验证修改结果。打乱顺序会破坏这个语义。

分批示意 partitionToolCalls 贪心分组 — 顺序保持，不做全局重排
partitionToolCalls 是保序的贪心分组：同向相邻的"并发安全"工具合并成一批，遇到写操作立刻新开批次。批次间严格串行，批次内并发。

3.3.2 runTools：批次执行

runTools 是调度器的主入口，它遍历分批结果，对每个批次选择对应的执行策略（toolOrchestration.ts:23）：

// toolOrchestration.ts:23
export async function* runTools(...): AsyncGenerator<MessageUpdate> {
  let currentContext = toolUseContext

  for (const { isConcurrencySafe, blocks } of partitionToolCalls(...)) {
    if (isConcurrencySafe) {
      // ── 并发批次：所有工具同时启动，结果按到达顺序 yield
      const queuedContextModifiers: Record<string, ...> = {}
      for await (const update of runToolsConcurrently(blocks, ...)) {
        if (update.contextModifier) {
          queuedContextModifiers[update.toolUseID].push(update.modifyContext)
        }
        yield { message: update.message, newContext: currentContext }
      }
      // 并发批次结束后，统一应用 contextModifier
      for (const block of blocks) {
        for (const modifier of queuedContextModifiers[block.id] ?? []) {
          currentContext = modifier(currentContext)
        }
      }
    } else {
      // ── 串行批次：逐个执行，实时更新 context
      for await (const update of runToolsSerially(blocks, ...)) {
        if (update.newContext) currentContext = update.newContext
        yield { message: update.message, newContext: currentContext }
      }
    }
  }
}

这里有一个微妙的细节：contextModifier。有些工具执行完之后，需要修改运行时上下文（ToolUseContext）——例如 FileEditTool 写完文件后，需要更新内部的文件状态缓存。对串行工具，这很简单：执行完立刻应用修改，下一个工具拿到的 context 已经是更新过的。

对并发工具，问题来了：多个工具同时在跑，如果每个工具都实时修改 context，就出现了经典的竞争条件（race condition）。Claude Code 的解法是延迟应用：并发批次内的所有 contextModifier 先收集起来，等整批工具都完成后再依次应用。这是并发工具"只读安全"这个假设在代码层面的体现——并发工具不得修改共享状态，修改动作统一推迟到批次结束后串行执行。

3.3.3 并发引擎：all() 函数

并发执行的核心是 utils/generators.ts 里的 all() 函数，它把多个异步 Generator 并发跑起来，限制同时活跃的数量，按到达顺序 yield 结果（generators.ts:32）：

// generators.ts:32
export async function* all<A>(
  generators: AsyncGenerator<A, void>[],
  concurrencyCap = Infinity,
): AsyncGenerator<A, void> {
  const waiting = [...generators]
  const promises = new Set<Promise<...>>()

  // 启动初始批次（受 cap 限制）
  while (promises.size < concurrencyCap && waiting.length > 0) {
    promises.add(next(waiting.shift()!))
  }

  while (promises.size > 0) {
    // 谁先返回就先 yield，不等慢的那个
    const { done, value, generator } = await Promise.race(promises)
    if (!done) {
      promises.add(next(generator))
      if (value !== undefined) yield value
    } else if (waiting.length > 0) {
      // 一个 generator 跑完，从等待队列里补充一个
      promises.add(next(waiting.shift()!))
    }
  }
}

并发上限由环境变量 CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY 控制，默认值 10（toolOrchestration.ts:11）。这个值不是随意选的：文件读操作是 I/O 密集型，10 并发能让磁盘 I/O 充分饱和，同时避免文件描述符耗尽。

时序图 · 动态 批次1并发 + 批次2串行 的时间线对比 — 小球表示正在执行的工具
并发批次的壁钟时间等于批次内最慢工具的耗时，而非各工具耗时之和。写操作串行不增加总时间，只是排队等待上一批并发完成。
设计启示

给开发者这套调度设计的精髓是"声明式并发"：工具不参与调度决策，只声明自己的属性（isConcurrencySafe），调度器根据这个声明自动生成最优执行计划。这比在工具内部手动控制锁和同步要安全得多——工具作者只需要诚实地声明，就能自动获得正确的并发行为。

给产品经理这套机制是"快而不乱"的技术基础。每轮工具调用里，Agent 能同时读多个文件、搜多个模式，但写操作始终有序进行，不会出现文件被交叉写坏的情况。这也解释了为什么 Claude Code 处理大型代码库时速度相对快——读操作是真的并发的。

3.4 MCP 工具统一接入

到目前为止我们讨论的都是"内置工具"——Bash、Read、Edit 这类随 Claude Code 发布的工具。但 Claude Code 还支持一类"外部工具"：通过 MCP（Model Context Protocol） 协议接入的第三方工具，比如数据库查询工具、Figma 设计工具、Slack 消息工具等。这些工具的代码不在 Claude Code 里，而是运行在外部进程中。

概念 · MCP（Model Context Protocol）

MCP（Model Context Protocol）是 Anthropic 提出的开放协议，定义了 LLM 应用与外部工具 / 数据源之间的标准通信接口。MCP 服务端（Server）是一个独立进程，暴露工具定义和调用接口；MCP 客户端（Client）是嵌入在 LLM 应用里的连接器，负责发现工具、传递调用请求、接收结果。

MCP 的核心价值是标准化：任何人都可以按照协议实现一个 MCP Server，让任何支持 MCP 的 AI 应用立刻获得这个工具的能力，无需修改 AI 应用本身的代码。

MCP 工具在 Claude Code 内部的接入点是 services/mcp/client.ts。当一个 MCP Server 连接后，Claude Code 遍历它暴露的所有工具定义，把每个工具"改造"成一个符合 Tool 接口的本地对象（client.ts:1768）：

// client.ts:1768 — MCP 工具适配层（精简）
for (const tool of mcpTools) {
  const fullyQualifiedName = buildMcpToolName(client.name, tool.name)
  // mcp__server_name__tool_name

  const mcpTool = {
    ...MCPTool,                      // 继承 MCP 工具的基础实现
    isMcp: true,
    name: fullyQualifiedName,

    // ① 从 MCP 协议的 annotations 映射到 isConcurrencySafe / isReadOnly
    isConcurrencySafe() {
      return tool.annotations?.readOnlyHint ?? false
    },
    isReadOnly() {
      return tool.annotations?.readOnlyHint ?? false
    },
    isDestructive() {
      return tool.annotations?.destructiveHint ?? false
    },

    // ② inputSchema 用 MCP Server 定义的 JSON Schema，而非 Zod
    inputJSONSchema: tool.inputSchema as Tool['inputJSONSchema'],

    // ③ 权限默认走"需要用户确认"流程（passthrough）
    async checkPermissions() {
      return { behavior: 'passthrough', ... }
    },

    // ④ 实际调用通过 MCP 客户端转发
    async call(args) {
      const result = await mcpClient.callTool({ name: tool.name, arguments: args })
      return { data: formatMcpResult(result) }
    },
  }
  tools.push(mcpTool)
}

关键在第①步：MCP 工具通过 annotations（工具元数据）声明自己的属性，Claude Code 把这些声明自动翻译成本地调度属性。一个数据库查询工具在 MCP 定义里标注了 readOnlyHint: true，Claude Code 就自动把它放进并发批次；一个会写文件的工具标注了 destructiveHint: true，就串行执行并在权限层弹出额外警告。外部工具无缝融入了内置工具的调度体系，无需特殊处理。

架构图 MCP 工具适配：从外部进程到 Tool 接口 — 三层映射
MCP 工具通过适配层变成标准 Tool 对象。调度器看不出内置工具和 MCP 工具的差别，两者遵循同一套并发规则。

3.5 ToolSearch——工具的懒加载

随着 MCP 生态的发展，一个重型 Claude Code 实例可能挂载几十个 MCP Server，每个 Server 暴露数十个工具。把所有工具的完整 Schema 全部写进 System Prompt，一个 Prompt 就要消耗数万 token——既慢又贵，而且大多数工具在当次任务里根本用不到。

这是一个经典的"按需加载"问题，Claude Code 的解法叫 ToolSearch。

概念 · ToolSearch（工具懒加载）

ToolSearch 是一种延迟加载（lazy loading）机制：默认只在 System Prompt 里列出工具的名字（"告知存在"），不发完整 Schema；模型通过调用 ToolSearch 工具，用关键词查询、加载某个工具的完整 Schema，然后才能调用它。

这与浏览器里的代码分割（code splitting）思路完全相同：先加载框架和首屏代码，剩余功能等到真正需要时才动态拉取。

一个工具是否"被延迟加载"由两个条件决定（toolSearch.ts:isDeferredTool）：

// toolSearch.ts
export function isDeferredTool(tool: Tool): boolean {
  if (tool.alwaysLoad === true) return false   // 明确要求首屏加载

  if (tool.isMcp === true) return true          // MCP 工具一律延迟

  if (tool.name === TOOL_SEARCH_TOOL_NAME) return false  // ToolSearch 自己不能被延迟

  return tool.shouldDefer === true              // 内置工具显式标记可延迟
}

当 ToolSearch 模式开启时，System Prompt 里看起来是这样的：

# System Prompt（精简示意）
可用工具（立即可用，完整 schema 已附）：
  Bash, Read, Edit, Write, Grep, Glob, Agent, ToolSearch ...

<system-reminder>
  deferred tools（需先调 ToolSearch 加载 schema）：
  mcp__figma__get_design_context
  mcp__figma__generate_figma_design
  mcp__linear__authenticate
  mcp__slack__send_message
  ... （仅名字，无参数描述）
</system-reminder>

当模型想用 Figma 工具时，先调用 ToolSearch：

// 模型发出的工具调用
{
  "type": "tool_use",
  "name": "ToolSearch",
  "input": {
    "query": "select:mcp__figma__get_design_context"
  }
}

// ToolSearch 返回（函数签名变得可调用）
{
  "type": "tool_result",
  "content": "<functions>
    <function>{\"name\": \"mcp__figma__get_design_context\",
               \"description\": \"...\",
               \"parameters\": {...complete schema...}}</function>
  </functions>"
}

拿到完整 Schema 后，模型就可以在下一步调用 mcp__figma__get_design_context 了。这个"ToolSearch → 实际工具调用"的两步流程，带来了约 40% 的首屏 token 节省，代价是多一轮往返（当命中缓存时几乎可以忽略，因为结果可以在模型的上下文中重用）。

设计启示

给开发者ToolSearch 的设计解决了一个普遍问题：随着功能增长，初始上下文会越来越臃肿。解决办法不是限制功能的数量，而是把"声明存在"和"提供完整描述"分成两个阶段。你设计任何大型 Prompt 系统时，都可以参考这个思路——让 LLM 先知道"有哪些能力"，再按需拉取"每个能力的完整用法"。

给产品经理ToolSearch 使得"Claude Code 能接入多少个 MCP 工具"在理论上变成了无上限——因为工具数量的增加对首屏成本的影响从线性变成了常数（每个延迟工具只贡献一行名字）。这也是 MCP 生态能规模化的基础能力之一。

3.6 工具执行的完整生命周期

综合以上四节，一次完整的工具调用经历的路径如下：

模型发出 tool_use block
  ↓
query.ts: 收集到 toolUseBlocks[]
  ↓
runTools(toolUseBlocks, ...)                    ← toolOrchestration.ts
  ↓
partitionToolCalls()                            ← 按 isConcurrencySafe 分批
  ↓
for each batch:
  isConcurrencySafe=true  → runToolsConcurrently()  → all(generators, cap=10)
  isConcurrencySafe=false → runToolsSerially()       → for...of

  for each tool in batch:
    runToolUse(toolUse, ...)                    ← toolExecution.ts
      ↓
      findToolByName()                          ← 在 tools[] 里查找
      ↓
      tool.inputSchema.safeParse(input)         ← Zod 校验参数
      ↓
      tool.validateInput?(input, ctx)           ← 语义前置检查
      ↓
      checkPermissionsAndCallTool()             ← 权限流
        canUseTool(tool, input, ctx)            ← hooks + 用户确认
        ↓
        tool.call(args, ctx, ...)               ← 工具主逻辑
        ↓
        processToolResultBlock()                ← 结果落盘（若超 maxResultSizeChars）
        ↓
        createUserMessage(tool_result)          ← 包装成 user role 消息
  ↓
yield MessageUpdate → query.ts 收集 toolResults[]
  ↓
三段拼接 → 下一轮 messages[]

这条路径里，每一个环节都是本章某个小节的主题。它们共同构成了 Agent "手"的工作原理：从模型的意图（tool_use block），经过分批调度、权限守卫、实际执行、结果打包，最终以 user role 的 tool_result 注入回消息数组，让模型能"看见"自己的行动结果。
