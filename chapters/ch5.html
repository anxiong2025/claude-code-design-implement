第五章

协调器模式——星形多 Agent

实现目标：一个 Coordinator 同时管多个 Worker，Worker 异步汇报结果，Coordinator 综合后再派新任务。

给主 Agent 一个大任务："项目里三个互不依赖的模块，各重构一遍。"三份活完全可以同时干，而"同时"的机制第四章已经给了——异步派发（一句话回顾：run_in_background: true 让派生调用立即返回 task_id，子 Agent 的结果在完成后以 task-notification 的形式注入父的后续轮次）。但机制解决不了管理问题：当三五个 Worker 同时在后台干活、结果乱序抵达时，谁来拆解任务、消化结果、决定下一步？主 Agent 的人格必须变——从亲自动手的工程师，变成带团队的 Tech Lead。这个新人格，就是本章的主角：协调器（Coordinator）。

Claude Code 对这个问题的回答出人意料地"薄"。翻开源码你会发现，协调器模式没有引入任何新的执行引擎：coordinator/coordinatorMode.ts 全文约 370 行，其中约 250 行是一个模板字符串——协调器的系统提示词。剩下的是一个环境变量开关和一段"告诉协调器 Worker 有哪些工具"的上下文注入。派生复用第四章的 AgentTool，追加指令复用 SendMessage 工具，停止复用 TaskStop 工具。协调器不是一个类，是一份提示词。这是本章最重要的一课：多 Agent 编排的复杂度大头在提示词工程，不在代码。

架构总图 星形拓扑：Coordinator 居中，一切经过中心 — Worker 之间没有任何连线
协调器模式的全部拓扑：Coordinator 通过三个工具向外辐射（派生 / 追加 / 停止），Worker 的结果通过通知队列汇回中心。整个"框架"只是一份系统提示词——执行机制全部复用前四章已有的循环、工具与调度。

5.1 多 Agent 设计模式全景

在钻进源码之前，先把地图铺开。Anthropic 的工程博客 Building Effective Agents 把多 LLM 系统归纳成一条光谱：一端是 Workflow——LLM 和工具沿着预定义的代码路径编排，可预测、易调试；另一端是 Agent——LLM 自主决定自己的流程和工具使用，灵活但要交出控制权。光谱上散布着五种可组合的工作流模式，外加自主 Agent 本身：

模式地图 六种 Agent 设计模式 — 每一格标注了它在 Claude Code 里的化身
六种设计模式与它们在 Claude Code 里的化身。注意一个事实：Claude Code 没有为任何一种模式单独建框架——模式只是"提示词 + 已有循环机制"的不同组合方式。协调器模式（④）是其中结构最重的一种，本章将看到它的实现依然轻得惊人。

这张图上值得停留的观察有三个：

其一，模式不是框架，是任务结构的分类。任务天然串行就是链式，天然可分片就是并行，无法预先拆解才轮到协调器——协调器模式的定义性特征是"子任务由中心 LLM 在运行时动态决定"，这正是它区别于第三章批内并发（调度器按静态属性分批）的地方。选模式的问题从来不是"哪个更先进"，而是"任务长什么形状"。文章的原则也直白：能用单次 LLM 调用 + 检索解决的，就别上工作流；能用工作流的，就别上自主 Agent——从最简单的方案开始，只在可测量的收益面前增加复杂度。

其二，模式可以嵌套。协调器派出的每个 Worker 自己就是一个自主体（⑥）；协调器把"实现"和"验证"分给不同 Worker，就是在星形里嵌了一个生成-评审回路（⑤）；同轮派发三个调研 Worker，用的是并行（③）。第五章不是替代前四章，是把它们组合起来。

其三，Claude Code 全部六种模式共用同一个执行基座。链式是提示词约定（先出计划、批准后执行），路由是几行模型选择逻辑，并行是调度器的自然行为（4.1 的机制图），自主体是 query() 循环本身。于是协调器模式剩下的实现问题只有一个：给循环换一个什么样的人格，才能让它管好别的循环？这就是下一节。

设计启示

给开发者市面上的多 Agent 框架大多把六种模式做成六套 API。Claude Code 的做法是反过来的：把"循环 + 工具 + 调度"这一个基座做扎实，模式退化为提示词和调用方式的组合。如果你发现自己在为每种编排方式写新的执行引擎，值得停下来问：这些引擎的差异，是不是本来可以用一段提示词表达？

5.2 Coordinator 角色设计

协调器模式由一个环境变量开启。开关本身平平无奇，值得看的是它改变了什么——答案是：只改人格与工具面板，不改循环：

// coordinator/coordinatorMode.ts — 开关（精简）
export function isCoordinatorMode(): boolean {
  return isEnvTruthy(process.env.CLAUDE_CODE_COORDINATOR_MODE)
}

// 恢复会话时，模式跟着会话走：不匹配就当场翻转环境变量
export function matchSessionMode(sessionMode) {
  if (currentIsCoordinator === sessionIsCoordinator) return undefined
  process.env.CLAUDE_CODE_COORDINATOR_MODE = sessionIsCoordinator ? '1' : undefined
  return 'Entered coordinator mode to match resumed session.'
}

开关打开后发生两件事：主循环的系统提示词被 getCoordinatorSystemPrompt() 整体替换，内置 Agent 列表被替换为 coordinator 专属的 worker 类型。协调器的全部行为规范都在那份约 250 行的提示词里。把它按主题拆开，骨架是这样的：

getCoordinatorSystemPrompt() 的内容骨架 —— coordinator/coordinatorMode.ts:111

§1 角色     你是协调者：派工调研/实现/验证，综合结果，与用户沟通。
            能直接回答的问题就直接答，不为不需要工具的事派工。
            你的每条消息都是说给用户听的；Worker 的结果是内部信号，
            不是对话伙伴——永远不要感谢或答复一个通知。
§2 工具     Agent（派生）· SendMessage（追加）· TaskStop（停止）
            纪律：不用 Worker 盯 Worker；不派琐碎小事；不设 model 参数；
            派发后简短告知用户即收口——绝不虚构或预测结果。
§3 Worker   用 subagent_type: 'worker'；工具池见 5.2.2。
§4 工作流   调研(并行) → 综合(你本人) → 实现 → 验证；并发策略见 5.5。
§5 写任务书 Worker 看不到你的对话——每份 prompt 必须自包含；
            以及本节最重要的规矩：永远先综合，再派工。
§6 完整示例 一段多轮 few-shot 对话，示范通知如何抵达、如何响应。

其中 §1 的两句话值得放大。"能直接回答的就直接答"防的是过度委托——协调器模式最常见的浪费是把一句话能答的问题包装成一次派工。"永远不要感谢通知"则暴露了一个实现细节：Worker 的结果会伪装成用户消息抵达（5.3 详述），不加这句约束，模型会礼貌地回复一个 XML 通知"谢谢你的辛勤工作"——对着传送带说客套话。提示词工程的很大一部分，是在给模型解释它所处的物理世界。

5.2.1 四阶段工作流——"综合"不可外包

工作流 四阶段循环：调研 → 综合 → 实现 → 验证 — 只有"综合"必须协调器亲自做
四阶段中三个阶段都可以派工，唯独"综合"必须协调器亲自完成——这是整份系统提示词强调最重的一条纪律，源码原话是"your most important job"。

为什么综合不可外包？提示词里有一段近乎严厉的规定：禁止写 "based on your findings"（基于你的发现）这类句子。对比一下两种派工方式：

// 反模式 —— 偷懒转发，把"理解"外包给了 Worker
Agent({ prompt: "Based on your findings, fix the auth bug" })
Agent({ prompt: "上一个 Worker 发现了 auth 模块的问题，请修复" })

// 正确 —— 协调器先消化调研结果，再写出自包含的 spec
Agent({ prompt: "修复 src/auth/validate.ts:42 的空指针。Session 上的 user 字段
  （src/auth/types.ts:15）在会话过期但 token 仍被缓存时是 undefined。
  在访问 user.id 前加空检查——为 null 时返回 401 'Session expired'。
  跑相关测试和 typecheck，提交并报告 hash。" })

两者的差别不是措辞，是信息在哪里汇聚。星形拓扑里 Worker 互不可见，唯一看得到全局的节点是中心。如果中心只做转发（"把 A 的发现交给 B"），全局视野就没人拥有了——B 拿到的是二手转述，A 的上下文已经烧掉，出了错没有任何节点能兜底。协调器的价值不是"分派"这个动作，而是它是全系统唯一被强制读懂一切的节点。这也解释了为什么提示词要求 spec 里必须出现具体路径和行号：这些细节是"我真的读懂了"的可验证证据，写不出来就说明综合没做。

5.2.2 Worker 的工具池——星形是删出来的

Worker 不是一种新机制，就是第四章的异步 SubAgent，但它的工具白名单经过了刻意裁剪（constants/tools.ts）：

// Worker 工具池 = 异步 Agent 白名单 - 内部工具
export const ASYNC_AGENT_ALLOWED_TOOLS = new Set([
  FILE_READ, GREP, GLOB, BASH, FILE_EDIT, FILE_WRITE,   // 读写与执行
  WEB_SEARCH, WEB_FETCH, SKILL, TOOL_SEARCH,            // 检索与技能
  ENTER_WORKTREE, EXIT_WORKTREE, TODO_WRITE, ...        // 隔离与规划
])
// 注意名单里没有的：Agent（不能再派生）、SendMessage（不能互发消息）
// —— INTERNAL_WORKER_TOOLS 进一步移除 TeamCreate/TeamDelete/SendMessage

删掉的两个工具正好定义了拓扑。没有 Agent 工具，Worker 无法再派生——这是 4.4 讲过的防递归约束（一句话回顾：派生深度上限为 2，防的是失控的 Agent 树和不可判定的生命周期）在协调器模式下的延续：结构是星形，不是树形。没有 SendMessage，Worker 之间无法直连——所有信息必须经过中心，保证了协调器的全局视野不被旁路。星形不是画出来的，是从完整工具池里删出来的。

与此对称，协调器一侧通过 getCoordinatorUserContext() 收到一份"Worker 能力说明书"注入上下文：Worker 有哪些工具、连了哪些 MCP Server、以及一个所有 Worker 免许可读写的 scratchpad 目录——提示词称之为"durable cross-worker knowledge"，一块跨 Worker 的持久黑板。Worker 之间不能对话，但可以往黑板上写调研笔记供后继者读，这是星形拓扑里唯一的"旁路"，而它是文件而非消息——异步、可审计、不破坏拓扑。

5.2.3 续跑还是新开——上下文重叠度决策

调研 Worker 完成后，协调器手里有两个选项：用 SendMessage 让同一个 Worker 继续干（它已经加载的上下文还在），或者用 Agent 新开一个。提示词给了一张决策表，判据只有一条：旧上下文对下一个任务是资产还是污染：

情形选择为什么
调研恰好读过要改的那些文件续跑文件已在上下文里，spec 一到就能动手
调研很广、实现很窄新开别拖着探索噪音进实现，干净上下文更聚焦
修正它自己刚犯的错续跑错误现场还在它脑子里，一句话就能定位
验证别的 Worker 写的代码新开验证者要新鲜眼光，不能带着实现者的假设
第一次实现方向整个错了新开错误路径会锚定重试，干净起步避免路径依赖
完全无关的新任务新开没有可复用的上下文

这张表其实是第四章"上下文经济学"的续篇：4.4 说 SubAgent 是一次性的可丢弃窗口，本节补上另一半——窗口也可以不丢，前提是里面装的东西对下一步有用。续跑省掉的是重新加载文件的成本，新开省掉的是被旧假设带偏的风险，两边都是上下文的账。

设计启示

给开发者协调器提示词里最有工程味的规定，都是在防"责任真空"：综合不可外包（全局视野必须有人拥有）、spec 必须带路径行号（理解必须可验证）、派发后立即收口绝不虚构结果（状态必须来自通知而非想象）。设计多 Agent 系统时，比"怎么拆任务"更先回答的问题是：每一份理解，由哪个节点负责持有？

给产品经理"协调器自己几乎不干活"不是低效，是角色净化：它的产出物是任务书和给用户的汇报，这两样恰恰是用户唯一能看到的界面。中心节点的质量决定用户对整个多 Agent 系统的感知质量。

5.3 Worker 结果聚合——task-notification 的一生

Worker 跑完了，结果怎么回到协调器？第四章末尾只说了一句"以 task-notification 注入后续轮次"，现在拆开看全程。每个 Worker 在系统里是一个 LocalAgentTask（tasks/LocalAgentTask/LocalAgentTask.tsx），从完成到被协调器读到，要走四步：

流水线 通知流水线：完成 → 构造 XML → 排队 → 回合边界注入 — 用户输入永远插在通知前面
通知的一生：Worker 终态触发 XML 构造（notified 标志原子去重），进入 later 优先级排队（用户输入是 next，永远插队），在协调器回合结束的边界以 user 角色消息注入。协议的"解析器"不是代码，是系统提示词里教给模型的标签识别规则。

构造通知的源码浓缩后只有一小段，但每个字段都有讲究（LocalAgentTask.tsx:197）：

// enqueueAgentNotification —— 精简
let shouldEnqueue = false
updateTaskState(taskId, setAppState, task => {
  if (task.notified) return task          // 已通知过：原子去重，防止重复消息
  shouldEnqueue = true
  return { ...task, notified: true }
})
if (!shouldEnqueue) return

const message = `<task-notification>
<task-id>${taskId}</task-id>                    // 续跑地址：SendMessage 的 to
<output-file>${outputPath}</output-file>        // 全程输出落盘，可追查细节
<status>${status}</status>                      // completed | failed | killed
<summary>${summary}</summary>${resultSection}${usageSection}${worktreeSection}
</task-notification>`

enqueuePendingNotification({ value: message, mode: 'task-notification' })
// ↑ 优先级 'later'：注释原话——"so user input is never starved by system messages"

三个设计决定值得点名：

投递时机选在回合边界，而不是实时打断。协调器正在生成回复时，哪怕三个 Worker 同时完成，也要等这一轮说完话。这不是技术限制——第二章讲过循环的输入队列机制（一句话回顾：所有外部输入统一进队列，回合之间统一消化），是刻意保持协调器每一轮的输入都是一个完整快照：乱序抵达的结果在队列里排好，模型一次看到当前全部新信息再规划下一步，而不是被每条通知打断成碎片决策。

用户输入的优先级永远高于通知。队列有 now / next / later 三级，用户消息是 next，通知是 later。五个 Worker 排队汇报时用户插了一句话，下一轮模型先看到人说了什么。多 Agent 系统很容易演成"系统自言自语，用户插不上嘴"，这一行优先级排序是对它的制度性防御。

result 之外还有 output-file。通知里的 <result> 只是 Worker 的最终文本（第四章的"结论二十行"），但完整过程始终落盘。协调器觉得结论可疑时，可以派下一个任务去读文件核查——聚合默认是轻的，但审计通道永远敞开。

设计启示

给开发者注意全书反复出现的同构：工具结果靠 tool_result 注入（第一章）、SubAgent 结论靠最终文本回传（第四章）、Worker 汇报靠 task-notification 注入（本章）。Claude Code 里所有的"回传"都是同一件事——把内容放进对方下一轮的输入里。没有回调、没有事件总线、没有共享内存，只有消息进队列。这让每个 Agent 的世界模型保持极简：我看到的就是我的 messages 数组。

5.4 SendMessage 通信协议

协调器手里最灵的工具是 SendMessage({ to, message })。它的输入极简，本事全在路由上——同一个工具，根据 to 指向的对象状态走出完全不同的路径（SendMessageTool.ts:741 的 call 方法）：

决策树 SendMessage 路由：同一个 to，四种命运 — 绿色是本章路径，灰色第六章展开
SendMessage 的路由按接收方状态分派。对本章重要的是左侧两条绿色路径：运行中排队注入、停止后磁盘复活——它们共同实现了"Worker 可以被随时追加指令"的体验。

两条绿色路径的源码骨架：

// SendMessageTool.call() —— 本进程 Worker 分支（精简）
const task = appState.tasks[agentId]
if (task.status === 'running') {
  // ① 运行中：进 pendingMessages，绝不打断
  queuePendingMessage(agentId, input.message, setAppState)
  return { message: `已排队，将在 ${to} 的下一个工具轮送达` }
}
// ② 已停止（completed / failed / killed）甚至已被驱逐出内存：
//    从磁盘 transcript 复活，带着全部旧上下文继续
const result = await resumeAgentBackground({ agentId, prompt: input.message, ... })

路径①的关键词是"工具轮边界"。Worker 正在跑一个 Bash 命令时收到新指令，指令不会打断命令，而是进入该任务的 pendingMessages 数组；Worker 每轮工具执行结束、准备下一次模型调用时，drainPendingMessages 把攒下的消息全部取走，作为附件拼进下一轮输入（attachments.ts:1085）。这和 5.3 协调器收通知的时机选择是同一个哲学的两个方向：消息永远在回合边界交接，任何 Agent 的任何一轮都不被打断。打断意味着半途而废的工具调用和被撕裂的思路，排队意味着每轮开始时世界是完整的。

路径②回收了第四章埋的伏笔。4.3 说过子 Agent 的 transcript 会落盘（当时的用途是"可查"）；现在它多了一个用途——可复活。resumeAgentBackground 从磁盘读回消息数组，做三重清洗（丢弃未解决的 tool_use、孤儿 thinking 块、空白 assistant 消息——都是中断可能留下的残渣），重建工具池和系统提示词，然后像什么都没发生过一样继续对话。对协调器而言，"Worker 停了"不是句号，是逗号：一个 completed 的调研 Worker，三轮之后仍能被一句 SendMessage 唤回来干实现，它读过的三十个文件还在脑子里。

最值得回味的是图底部那行字：接收方永远以为自己收到了普通 user 消息。Worker 的循环里没有"协调器"这个概念，没有任何针对多 Agent 的特殊代码——它就是第一章那个循环，收消息、干活、回话。整个协调器模式是单方面的：只有中心知道自己在协调，成员只是在过普通的 Agent 生活。这种不对称让系统的每个组件都保持最简单的世界模型，复杂度全部收敛在中心一点。

设计启示

给开发者很多多 Agent 框架为"Agent 间通信"设计了 RPC、话题订阅、共享状态。Claude Code 的答案是：通信 = 往对方的下一轮输入里放一条消息，状态 = 对方自己的 messages 数组。协议瘦到只剩投递语义，因为接收方是一个 LLM，它最擅长的就是理解自然语言消息——传输层不需要表达业务语义，业务语义在消息文本里。

5.5 并发管理与任务终止

5.5.1 两层并发控制——硬调度与软策略

协调器模式下，"并发"这个词同时发生在两层，很容易混淆，值得摆在一起看清楚：

层级         谁做决定                   管什么                     什么性质
──────────────────────────────────────────────────────────────────────────
硬调度       partitionToolCalls        一轮内的多个工具调用        代码强制
（第三章）    （按 isConcurrencySafe    读并发、写独占              绝不越界
              静态属性分批）
──────────────────────────────────────────────────────────────────────────
软策略       协调器（提示词约定）        跨任务的 Worker 编排        模型自觉
（本章）      "调研自由并行；             写任务按文件集串行；        可能违反
              实现一次一个文件集"        验证可与实现交叠

为什么软策略不能也做成硬调度？因为判断"两个实现任务会不会改到同一批文件"需要理解任务语义，而调度器只看得见工具调用的静态属性。"修 auth 模块的空指针"和"给 payment 模块加日志"能不能并行，取决于两个模块有没有共享文件——这是语义问题，只有读得懂任务书的模型答得了。于是 Claude Code 把这层判断放进提示词（"Write-heavy tasks — one at a time per set of files"），由协调器在派发时自行掂量。守不住怎么办？第四章的 Worktree 隔离（一句话回顾：每个 Worker 可在独立的 git 工作树里干活，改动以分支交付）就是软策略失守时的硬兜底——策略防冲突于未然，隔离保证冲突真发生时不损坏任何东西。

5.5.2 停止、复活与生命周期状态机

把 5.3、5.4 讲过的所有转换拼起来，Worker 的一生是一张完整的状态机：

生命周期 Worker 状态机：终态不是终点 — 只要 transcript 在盘上，就能复活
Worker 状态机的核心特征：所有"结束"都是软的。completed 可以续跑、failed 带着错误现场等修正、killed 也回传部分结果并保留 transcript——真正不可逆的只有磁盘上 transcript 被清理那一刻。

终止路径上有两个容易错过的细节，都体现了"信息不浪费"的原则：

被杀的 Worker 也交作业。stopTask（tasks/stopTask.ts）里有一段耐人寻味的区别对待：后台 Bash 被杀时，"exit code 137" 通知被抑制——那是噪音；但 Agent 任务被杀时通知照发，因为它携带 extractPartialResult 从已产生的消息里抢救出的部分结果——那是 payload。你半路叫停一个调研 Worker，它读过的东西不会白读。

批量终止合并汇报。用户按 ESC 时 killAllRunningAgentTasks 停掉所有 Worker，同时用 markAgentsNotified 把每个任务的 notified 标志直接置位——五个 Worker 不会发五条讣告刷屏，系统发一条聚合消息。去重标志在这里被反向利用：先置位，通知就永远不会发出。

失败处理的策略同样围绕上下文经济学展开：Worker 报告测试失败时，提示词的第一选择是 SendMessage 续跑同一个 Worker——它拥有完整的错误现场（改了什么、报了什么错、试过什么），一句"58 行和 72 行的断言还在失败，改成新的报错文案"就够了；只有当续跑再次失败，才换思路新开或上报用户。对照 5.2.3 的决策表，这正是"修正它自己刚犯的错 → 续跑"那一行。

设计启示

给开发者这个状态机最反直觉的设计是"没有真正的终态"。传统任务系统里 killed 就是死了；这里 kill 只是 abort 掉进程内的循环，身份（agentId）和记忆（transcript）都在盘上，SendMessage 一到就原地复活。把"执行"和"身份+记忆"分离之后，停止变得廉价而无损——这是敢于"随手停、随手续"的底气，也间接鼓励协调器大胆派工：派错了就停，成本近乎为零。

给产品经理ESC 一键全停 + 聚合汇报 + 部分结果保留，这三件事共同回答用户对多 Agent 最深的恐惧："失控了怎么办"。刹车必须比油门显眼，这是自治系统赢得信任的前提。

本章的示例在 code/examples/coordinator_demo.py。它在第四章 SubAgent 骨架之上实现了一个迷你协调器：Coordinator 循环用 spawn_worker 异步派生 Worker（立即拿到 task_id），Worker 完成后把 task-notification 投进优先级队列，在 Coordinator 的回合边界注入；send_message 对运行中的 Worker 排队注入、对已完成的 Worker 从 transcript 复活——纯本地模拟，无需 API Key。
