第六章

Agent 间通信——Mailbox

实现目标：任意 Agent 可以给另一个 Agent 发消息，接收方可以阻塞等待，也可以非阻塞轮询。

想象这样一个场景：你开了三个终端窗口，每个窗口里跑着一个独立的 Claude Code——一个当领队，负责分工和验收；两个当队友，各自埋头改一个模块。领队想提醒队友 ada："utils 目录 bob 正在改，你先别碰。"这句话怎么送过去？要知道这是三个互不相识的操作系统进程：不共享内存，不知道彼此的端口和 PID，甚至不知道对方此刻还活着没有。这正是 Claude Code 团队模式（Agent Teams / Swarm）每时每刻要面对的通信问题，也是本章要解决的问题。

第五章的协调器从来不用操心这件事，因为那是"一个进程里的多 Agent"：Coordinator 和 Worker 全都活在同一个 Node 进程中，所谓"发消息"（SendMessage 工具），本质是往同一份内存里的 pendingMessages 数组 push 一项。没读过第五章也不要紧，只需记住这个对比：单进程内传话，就是改一块共享内存。可团队模式里每个成员都是独立进程，跑在 tmux（终端分屏工具）的不同窗格里——共享内存这条路，彻底断了。

两个互不相识的进程要通信，工程上的标准答案是 socket、HTTP、消息队列。Claude Code 选了一个"土"得多的方案：每人一只 JSON 文件当信箱，写信加锁追加，收信定时轮询。土，但仔细掂量会发现它精确匹配了这个场景的每一项需求——本章就从这只文件信箱出发，一路讲到跑在它上面的高层协议：权限跨进程转授、计划审批、优雅关机。你会再次看到第五章出现过的规律：传输层越简单，语义越容易做对；复杂度应该住在协议层，而协议大半是提示词和 JSON 约定。

架构总图 文件即消息总线：两个进程，唯一的共享介质是磁盘 — 没有 socket，没有端口，没有 broker
Mailbox 的全部架构：每个成员一只 JSON 信箱文件，写方加锁追加，收方轮询未读。进程之间零直连——一方崩溃、重启、换机器（只要挂同一块盘），通信语义不受任何影响。
本章目标

读完这章，你会理解：

  • 6.1为什么用文件当消息总线——一只加锁的 JSON 信箱如何胜过 socket 与 broker

6.1 Mailbox 架构——文件即消息总线

先看数据结构，全部机制围绕它展开（utils/teammateMailbox.ts）：

// 信箱地址：~/.claude/teams/{team}/inboxes/{agent_name}.json
export function getInboxPath(agentName: string, teamName?: string): string {
  return join(getTeamsDir(), sanitize(team), 'inboxes', `${sanitize(agentName)}.json`)
}

// 一封信：发件人 + 正文 + 已读标记，没有了
export type TeammateMessage = {
  from: string          // 发件人名字（自愿声明，见 6.4 的信任模型）
  text: string          // 正文：自然语言，或 JSON 编码的协议消息（6.3）
  timestamp: string
  read: boolean         // 收件方消费后置 true——递送语义的关键
  color?: string        // 发件人的 UI 颜色
  summary?: string      // 5-10 词摘要，给 UI 预览用
}

写入是唯一需要小心的操作：一个团队里多个进程可能同时给同一个人写信，两个 read-modify-write 交错就会丢信。解法是经典的锁文件 + 重试退避：

// writeToMailbox —— 精简（teammateMailbox.ts:134）
await writeFile(inboxPath, '[]', { flag: 'wx' })   // 不存在则创建（原子）
const release = await lockfile.lock(inboxPath, {
  retries: { retries: 10, minTimeout: 5, maxTimeout: 100 },  // 排队等锁，不失败
})
try {
  const messages = await readMailbox(recipientName)   // 拿到锁后重读最新状态
  messages.push({ ...message, read: false })
  await writeFile(inboxPath, jsonStringify(messages, null, 2))
} finally {
  await release()
}

三行注释值得展开成三个设计判断：

为什么是文件，不是 socket / HTTP / 消息队列？把场景的约束列出来，答案几乎是推出来的。第一，通信双方是不同后端上的进程——可能同进程（in-process 队友）、可能是 tmux 窗格、iTerm 窗格，未来还可能是远程会话；socket 需要端口协商和服务发现，文件路径本身就是全局约定好的"地址"。第二，Agent 的回合以秒到分钟计，1 秒的轮询延迟在这个时间尺度上是免费的——为省这 1 秒去引入一个 broker，是拿复杂度换一个没人感知的指标。第三，消息天然要求持久与可审计：进程崩了信还在，重启接着读；用户随时可以 cat 一眼信箱排查问题。第四，零基础设施——不需要用户装 Redis，不需要起 daemon。文件系统是唯一保证存在的中间件。

为什么锁要重试退避，而不是失败报错？注释原话："concurrent callers (multiple Claudes in a swarm) wait for the lock instead of failing immediately"。发信方是模型的工具调用，一次失败对模型而言是一次需要理解和重试的异常；把重试下沉到传输层（最多 10 次、5–100ms 退避），模型看到的世界就永远是"发信必成功"。底层多做一点，上层的心智模型就少一个分支。

为什么拿到锁后要重读？锁保护的是 read-modify-write 全程，而不是 write 一下。等锁期间别人可能已经追加了消息，用等锁前读的旧数组覆盖回去就会丢信。这是所有"文件当数据库"场景的第一课，源码用注释把它钉死在那里。

信箱之外还有一份名册：TeamCreate 工具创建 ~/.claude/teams/{team}/config.json，记录全体成员（名字、agentId、颜色、活跃状态），同时创建一个同名任务列表目录——Team 与 TaskList 一一对应，任务的指派（TaskUpdate 的 owner 字段）和消息的投递共享同一套成员名单。广播（to: '*'）就是读名册、逐个写信箱的语法糖。

设计启示

给开发者选传输层时先问时间尺度。Agent 回合是秒级的，消息延迟 1 秒无感知——此时持久性、可审计性、零部署的价值远大于吞吐和延迟。反过来，如果你的 Agent 需要毫秒级互动（比如协作编辑同一个文件的光标），文件轮询就是错的。没有"最好的消息机制"，只有和交互节奏匹配的消息机制。

6.2 寻址与收信——轮询、阻塞与递送语义

6.2.1 我是谁——三层身份解析

信箱按名字寻址，那"我的名字"从哪来？这个看似平凡的问题因为队友有两种形态而变得讲究（utils/teammate.ts）：

身份解析优先级（getAgentName / getTeamName）

① AsyncLocalStorage        进程内队友。同一个进程里可能同时跑着领队
   （teammateContext）      和 N 个进程内队友，全局变量会互相覆盖——
                            AsyncLocalStorage 给每条异步调用链一份
                            隔离的身份，谁在跑就解析出谁。
② dynamicTeamContext       tmux 队友。独立进程，身份来自启动时的
   （CLI 参数注入）          CLI 参数（--agent-id --team-name）。
③ AppState.teamContext     领队。它不是被派生的，身份记在应用状态里。

值得停一秒的是第①层。进程内队友和领队共享同一份内存、同一个 React 状态树，"当前代码在为谁跑"这个问题没有全局答案，只有调用链上下文能回答。AsyncLocalStorage 在这里扮演的角色，和第四章 SubAgent 的隔离 messages 数组同构：并发的身份必须随执行流走，不能存在共享的地方。

6.2.2 收信的三条通道

本章副标题说"接收方可以阻塞等待，也可以非阻塞轮询"，源码里实际有三条通道，按接收方的状态各司其职：

机制图 三条收信通道：谁在什么状态下用哪条 — 已读标记永远在安全递送之后
三条收信通道覆盖接收方的三种状态：闲着（轮询成轮）、常驻等活（阻塞循环）、干活中（轮边界注入）。无论哪条，已读标记都在消息安全落地之后才写回——这是整个递送语义的支点。

递送语义值得单独讲透，因为它是"文件当消息队列"最容易做错的地方。useInboxPoller 的注释把顺序写得很明白：先递送或可靠入队，再标记已读。如果反过来——先标已读再递送——进程在两步之间崩溃，消息就永远消失了；按现在的顺序，崩溃的后果只是下次轮询重读一遍，即经典的 at-least-once（至少一次）语义。重复递送的代价由消费端的幂等去重抵消：比如权限请求入队前按 tool_use_id 查重（"markMessagesAsRead failed on a prior poll — skip if already queued"）。丢失不可恢复、重复可以过滤，两害相权，分布式系统五十年的老结论，在这里用几十行代码复刻了一遍。

另一个容易错过的细节在阻塞循环里：出队前先全量扫描一遍未读消息找 shutdown 请求，找到就插队处理。注释给了原因：点对点消息可能洪泛（队友之间聊得太热闹），如果严格 FIFO，关机指令会被淹没在队列尾部，想停都停不下来。控制类消息优先于数据类消息——这个取舍在 5.3 的通知队列里是"用户输入优先于系统通知"，方向相反，原则相同：队列必须为"改变队列命运的消息"留插队通道。

设计启示

给开发者注意全书出现第三次的同构：工具结果注入（第一章）、协调器通知注入（第五章）、信箱消息注入（本章），全部是"把内容放进对方下一轮输入"。Mailbox 没有发明新的通信范式，它只是把"对方的输入队列"从进程内存搬到了文件系统——于是通信自动获得了跨进程、持久化、可审计三个属性，而模型侧的世界观完全不用变：还是收到一条消息，还是下一轮看到。

6.3 消息分类——数据面与控制面

信箱里流动的消息其实是两个物种。一种给模型看：自然语言，"我把 auth 模块查完了，结论如下"；一种给代码看：JSON 编码的协议消息，"type: permission_request, tool: Bash, input: …"。收信端的第一件事就是把它们分开（useInboxPoller.ts 的分拣循环）：

分拣器 数据面给模型，控制面给代码 — 分错任何一个方向都是事故
同一只信箱跑两个平面：自然语言走数据面进模型上下文，JSON 协议消息走控制面进代码处理器。分拣的依据只有一个——text 字段能否解析出已知的 type。

为什么这个分离值得用一个守卫函数（isStructuredProtocolMessage）严防死守？源码注释给了正向理由："If getTeammateMailboxAttachments consumes them first, they get bundled as raw text in attachments and never reach their intended handlers"——数据面抢跑，控制面的处理器就永远收不到消息，权限请求石沉大海，Worker 卡死在等待里。反过来想还有一层：让模型读到一坨 {"type":"permission_response",...} 不仅无用，而且有害——模型可能把它当成要回应的内容，生成一段针对 JSON 的"回复"，白烧 token 还污染对话。LLM 系统里的每一字节上下文都该是模型能行动的信息；机器对机器的信令，模型一个字都不该看见。

反方向的分错更隐蔽：把该给模型看的消息拦在代码层。分拣表里有一行专门处理这个微妙情况——shutdown_request 被识别为协议消息，但处理方式是 regularMessages.push(m)：双面投递。因为关机请求的最终决定权在模型（6.4 详述），代码只负责识别和优先级，模型必须读到原文才能判断"我手头的活能不能停"。计划审批请求同理：领队侧自动批准之后，请求原文仍然进入领队模型的上下文——"the approval is already sent, but the model has context about what the teammate is doing"。控制面处理动作，数据面保留知情权，两面不冲突。

数据面的包装格式也顺带看一眼：<teammate-message teammate_id="ada" color="red" summary="...">。和第五章的 <task-notification> 一样，这是"伪装成 user 消息的非用户输入"，靠 XML 标签让模型识别说话的是谁。color 甚至会传进 UI，让领队屏幕上每个队友的发言带自己的颜色——协议为人和模型各服务一半。

设计启示

给开发者"数据面/控制面分离"是网络设备的经典架构，在 LLM 系统里有了新含义：模型的上下文窗口是数据面，代码路径是控制面。设计多 Agent 协议时对每种消息问一句：模型读到它能做什么？能行动的进上下文，不能行动的走代码，需要知情又需要代码动作的双面投递。这个三分法比"全部塞给模型"（浪费+误导）和"全部代码处理"（模型失去决策权）都更精确。

6.4 协议实战——权限瞬移与优雅关机

传输层（6.1）和分拣器（6.3）就位后，真正有产品价值的东西是跑在上面的协议。挑两个最完整的看：一个解决"多进程的权限弹窗该弹在哪"，一个解决"怎么体面地解散团队"。

6.4.1 权限瞬移——跨进程 RPC，传输层是两封信

场景：tmux 窗格里的队友要跑 rm -rf build/，权限系统要求用户确认。弹窗弹在队友自己的窗格里？用户正盯着领队的窗格，根本看不见，队友一卡就是十分钟。Claude Code 的答案是把弹窗瞬移到领队的屏幕上：

时序图 权限请求的跨进程往返 — 一次 RPC，传输层是两封 JSON 信
权限瞬移的完整往返：队友挂起工具调用并注册回调，请求信 1 秒内到达领队并弹出与本地完全一致的权限 UI，用户的决定原路写回，回调 resolve，工具从挂起处继续。整套跨进程 RPC 没有一个 socket。

两个细节把这个协议从"能用"提升到"好用"。其一，领队侧不是弹一个简陋的"允许 ada 执行 Bash？"文本框，而是把请求灌进本地权限系统的标准队列（ToolUseConfirmQueue），复用全部原生 UI——Bash 命令高亮、文件编辑 diff 视图，只多一个"来自 ada"的徽章。跨进程的请求和本地的请求在用户眼里没有任何区别，这是第七章权限系统统一入口的直接红利。其二，响应消息携带 updated_input 和 permission_updates——用户在领队屏幕上不仅能批准，还能修改命令再批准、能勾选"以后都允许"，这些决定会通过 team_permission_update 广播给全队生效。

顺着这条线还能看到信任模型的边界，源码处理得很诚实：信箱文件谁都能写，from 字段是自愿声明，所以关键协议都做了来源校验——计划审批只认 from === 'team-lead'（"prevent teammates from forging approvals"），模式下发同样只认领队。但这层校验防的是混乱（队友互相误发、模型幻觉出一条审批），不是恶意——同一台机器、同一个用户账户下的进程本来就在同一信任域里，真正的安全边界在第七章的权限系统，不在信箱。

6.4.2 优雅关机——对一个 LLM，你不能直接 kill

解散团队时，领队对每个队友发的不是 SIGKILL，是一封请求信，然后是一次三步握手：

优雅关机握手（leader ⇄ teammate，全程走信箱）

① leader → shutdown_request { requestId, reason }
     队友的等待循环里它有插队权（6.2），不会被聊天消息淹没；
     但它走"双面投递"——最终决定权在队友的模型手里。

② 队友模型读到请求，检查手头工作，二选一：
     approveShutdown → shutdown_approved { paneId, backendType }
     rejectShutdown  → shutdown_rejected { reason: "还有未提交的改动" }

③ leader 收到 approved 后才动手（useInboxPoller.ts:677）：
     backend.killPane(paneId)      按队友自报的窗格号杀进程
     removeTeammateFromTeamFile    从名册除名
     unassignTeammateTasks         回收它名下的任务，重新可指派
     聚合一条 teammate_terminated  通知领队模型与 UI

为什么要这么绕？因为被关的是一个正在思考的 LLM，不是一个无状态进程。它可能正拿着改到一半没提交的代码、写到一半的调研结论；直接 kill，这些就全丢了（第五章的 Worker 被杀有部分结果抢救，但那是单进程内存里的便利，跨进程没有）。把"能不能停"设计成一个问句，让队友自己收尾再交回控制权，代价只是一封信的往返，换来的是永远不会杀掉一个手里有活的 Agent。拒绝也不是终点——拒绝理由会出现在领队的上下文里，领队模型可以催、可以等、可以改主意。

③ 里还有一个不起眼但关键的动作：回收任务。Team 与 TaskList 一一对应（6.1），队友退场时它名下的任务自动解除指派，回到可认领状态——否则任务会被一个已经不存在的人永远占着。名册、任务、进程三份状态在关机这一刻同步清理，这是"Team = TaskList" 这个 1:1 设计最实际的回报。

设计启示

给开发者协调器（第五章）与团队（本章）是多 Agent 的两种拓扑，差异全在"成员是什么"：Worker 是一次性的执行器——跑完即死、无身份、被动接活；队友是常驻的对等体——有名字、能互聊、闲时等活、退场要谈判。前者适合可拆解的单一大任务，后者适合边界清晰的长期分工。注意两者共用同一个 SendMessage、同一套"消息进对方下一轮输入"的语义——拓扑变了，通信原语一个没换。

给产品经理"权限弹窗瞬移到领队屏幕"是这一章最值得抄的产品决策：多 Agent 系统的审批必须收敛到用户正在看的那一块屏幕，否则并行度越高，用户越像在玩打地鼠。
