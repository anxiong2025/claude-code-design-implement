第七章

权限与安全

实现目标：不同场景下工具的使用需要不同级别的授权，SubAgent 不能越权操作。

Agent 干活的每一步都是一次工具调用：读文件、跑测试、改代码、删目录。现在问一个绕不开的问题——模型发出了 rm -rf build/，执行，还是不执行？每一步都弹窗问用户，三五个回合之后用户必然烦到点下"全部允许"，安全归零；从来不问，一条幻觉出来的 rm -rf ~ 就是不可逆的灾难。权限系统要做的，是在"烦"和"险"之间造出一个精确的调节器——这是所有 Agent 产品都绕不开的设计题。

Claude Code 的答案是一条判定流水线：每个工具调用在执行前穿过一串顺序固定的关卡——PreToolUse 钩子、deny 规则、工具自查、安全检查、权限模式、allow 规则——在任何一关得出结论就立即离开流水线；走完全程仍无结论的，一律去问用户。无论多复杂的场景，系统最终只输出三个词之一：allow、deny、ask。这条流水线的主干在 utils/permissions/permissions.ts（约 1500 行），但触角伸得很远：Bash 一个工具的命令匹配就有 2600 行（src/tools/BashTool/bashPermissions.ts），文件路径判定又是约 1800 行（utils/permissions/filesystem.ts）——权限是 Claude Code 里代码密度最高的子系统之一，因为它是唯一一个"判断错一次就可能不可逆"的子系统。

架构总图 权限判定流水线：一次工具调用的旅程 — 否决类关卡永远排在豁免类关卡之前
一次工具调用的完整判定旅程。流水线在任何一关得出 allow/deny 即短路退出；④ 里的三类"豁免免疫"判断即使在 bypassPermissions 模式下也必须弹窗。走完全程仍无结论的调用，兜底一律是 ask——fail-closed。dontAsk 不在这条链内：它在整条流水线跑完、得出最终 ask 之后，由最外层把 ask 改判 deny（permissions.ts:503，注释"so it can't be bypassed by early returns"），所以任何提前 return 都逃不过它。

7.1 判定流水线——顺序即安全语义

先看主干代码的骨架。hasPermissionsToUseToolInner 的主干是三组——deny 组（1）、allow 组（2）、兜底（3），细分为下面这些判断，编号沿用源码注释里的原编号：

// hasPermissionsToUseToolInner —— 精简（permissions.ts:1158）
if (context.abortController.signal.aborted) throw new AbortError()  // 入口先查中止（7.5）

// 1a. 整个工具被 deny 规则禁用             → deny，游戏结束
// 1b. 整个工具有 ask 规则                 → ask（可被沙箱自动放行的 Bash 例外）
// 1c. 问工具自己：tool.checkPermissions(parsedInput, context)
//     Bash 拆子命令逐条判，文件工具判路径（7.2）
// 1d. 工具自查说 deny                     → deny
// 1e. 工具要求人机交互（AskUserQuestion）   → ask，bypass 模式也拦
// 1f. 内容级 ask 规则（Bash(npm publish:*)）→ ask，bypass 模式也拦
// 1g. 安全检查（.git/ .claude/ shell 配置） → ask，bypass 模式也拦

// 2a. bypassPermissions 模式              → allow
// 2b. 整个工具被 allow 规则放行            → allow

// 3. 走到这里还没有结论（passthrough）      → 一律转成 ask
return { ...toolPermissionResult, behavior: 'ask' }

这段代码最重要的信息不在任何一行里，而在行与行的顺序里。三个设计判断值得逐个展开：

为什么 deny（1a）排在 bypass（2a）之前？因为 deny 规则是用户亲手写下的否决票——"这个工具永远不许碰"。bypassPermissions 豁免的是"默认要问"的麻烦，不能豁免"用户明说不行"的禁令。把否决判断放在豁免判断之前，"切到 bypass 模式赶进度"就永远不会误伤你 deny 掉的生产数据库。顺序颠倒一下，整个安全模型就塌了——这就是"顺序即语义"。

为什么 1e/1f/1g 三类 ask 也排在 bypass 之前？它们是三种"豁免免疫"：用户显式配置的内容级 ask 规则（源码注释："This must be respected even in bypass mode, just as deny rules are respected"）；天生需要人回答的交互工具；以及安全检查——对 .git/、.claude/、shell 配置文件这类路径的写入。最后一类保护的其实是权限系统自己：settings.json 里存着规则，钩子脚本开机即运行，改一行，下次启动时整个权限系统就被架空了。看门人可以放行任何人，唯独不能让人改门锁。

为什么兜底（3）是 ask 不是 allow？任何一关都没表态的调用，说明它不在任何已知的安全清单里——未知即危险，fail-closed。第三章 buildTool 见过同款哲学（一句话回顾：工具不声明 isConcurrencySafe 就按"不可并发"处理，宁可慢不可错）。权限这里是：宁可多问一次，不可放过一次。

设计启示

给开发者权限判定的正确形状是一条有序短路链：否决类判断在最前，豁免类判断在中间，兜底分支永远保守。审计一个权限系统不必读完实现，先把它的判定顺序表画出来，找"有没有一条路径能绕过否决"——找到即漏洞。Claude Code 把三类"bypass 也拦"的判断显式排在 bypass 之前，这个顺序本身就是可以被单元测试锁死的安全属性。

给产品经理"烦"与"险"之间的那个调节器，是 Agent 产品的信任基石。用户能接受偶尔多问一次（fail-closed 的代价），但绝不能接受"切了个模式就把我 deny 掉的生产库删了"。把否决排在豁免之前，等于向用户承诺一条永不背叛的底线——这条承诺越清晰、越可解释，用户越敢把更高的自动化权限交给产品。

7.2 规则语言——Tool(specifier) 与复合命令

流水线里反复出现的"规则"长什么样？它是一门刻意保持简单的小语言，格式只有两种：Tool（管整个工具）和 Tool(content)（管这个工具的某类输入）：

// 规则字符串 → 结构化规则（permissionRuleParser.ts:93）
'Bash'                         → { toolName: 'Bash' }               // 整个工具
'Bash(npm install)'            → { …, ruleContent: 'npm install' }  // 精确一条命令
'Bash(npm:*)'                  → 前缀匹配：npm 开头的一切
'Bash(git diff *)'             → 通配符匹配（* 转成正则 .*）
'Read(~/.ssh/**)'              → 文件工具：路径 glob
'Bash(python -c "print\(1\)")' → 括号转义后精确匹配

// 通配符里的一个讲究（shellRuleMatching.ts:143）——
// 'git *' 的尾部通配是可选的：既匹配 'git add'，也匹配裸 'git'，
// 与前缀规则 'git:*' 的语义对齐；多通配模式（'* run *'）不享受此待遇
if (regexPattern.endsWith(' .*') && unescapedStarCount === 1) {
  regexPattern = regexPattern.slice(0, -3) + '( .*)?'
}

每条规则挂一个行为（allow / ask / deny）和一个来源。来源共八层：五层设置文件（userSettings 用户全局、projectSettings 项目共享、localSettings 项目本地不入库、flagSettings 命令行 --settings、policySettings 管理员托管），加三个运行时源（cliArg、command、session 本次会话）。关键的设计选择是：来源不决定优先级，行为才决定优先级——所有来源的规则摊平后统一按 deny → ask → allow 的顺序查（7.1 的 1a/1b/2b），一条 session 级的 deny 照样压过 userSettings 里的 allow。来源真正决定的是两件事：规则展示时标注"来自哪里"，以及用户点"总是允许"时新规则写回哪个文件。

最难的部分在 Bash。一条 cd src && npm test > out.log 怎么判？答案是拆开来，逐个判，一票否决：

机制图 复合命令的判定：拆分 → 逐个匹配 → 最严的说了算 — 全 allow 才 allow，一个 deny 即 deny
复合命令按连接符拆成子命令逐个判定，聚合时取最严结果。弹窗上用户看到的是"哪几段没过关"，而不是笼统的"这条命令需要审批"——精确到段，用户才只为真正的风险付注意力。

拆分-聚合之外，Bash 匹配还有一批对抗性的细节：判定前剥掉注释行和 timeout 这类"安全包装"（stripSafeWrappers），防止危险命令靠伪装混过前缀匹配；LD_*/DYLD_*/PATH 开头的环境变量赋值被单独盯防（BINARY_HIJACK_VARS），因为它们能把后面任何"无害命令"劫持成任意代码。还有一层防呆：Bash(python:*) 这样的 allow 规则看似具体，实则等于放行任意代码执行——dangerousPatterns.ts 维护了一张解释器/包运行器/远程执行的前缀清单（python、node、npx、bash、ssh……），这类过宽规则在进入自动化程度更高的模式时会被主动剥除。

设计启示

给开发者权限规则是一门小语言，它最重要的设计约束不是表达力，而是必须能从一次对话里长出来：用户点"总是允许"的瞬间，系统要能自动生成一条宽窄合适的规则（精确命令 → 精确规则，带参命令 → prefix:* 规则）。规则语言越花哨，自动生成越容易出错。

给产品经理"看似具体、实则放行一切"的规则（python:*、ssh:*）是权限系统的暗门，值得像 Claude Code 一样维护一张危险前缀清单主动拦截——用户自己很难意识到 npx:* 和 * 是等价的。

7.3 权限模式——改默认值，不改规则

规则管的是"这一条输入"，模式管的是整个会话的姿态。五个档位（PermissionMode.ts）：

default            规则说了算，规则没意见就问用户
acceptEdits        工作目录内的文件写操作免弹窗（filesystem.ts:1366 的快速通道），
                   Bash、目录外写入照常要问
plan               只读调研，动手前先交计划——约束主要在提示词层（见下）
bypassPermissions  规则之外全放行；deny/ask 规则、安全检查、交互工具依然拦（7.1）
dontAsk            "问 = 拒"：无人值守场景，宁可失败也不挂起等一个不存在的人
                   （不在 Shift+Tab 循环里，只能经 settings 的 defaultMode / 无人值守启动进入）

用户侧切换：Shift+Tab 循环 default → acceptEdits → plan →（bypass 如可用）→ default
                                     （getNextPermissionMode.ts:34）

五个档位里有两个值得拆开看。plan 模式是权限系统里最"薄"的一档：它几乎不改判定流水线，主要靠一条每轮注入的 plan_mode 附件提醒模型"现在是调研阶段，别动手"，加上一个专门的出口——ExitPlanMode 工具把"开始动手"变成一次显式的用户审批。第五章的规律再次出现（一句话回顾：协调器不是一个类，是一份提示词）：约束能放在提示词层就不放在代码层，因为"什么算调研、什么算动手"本身是语义判断。代码层只补两个硬细节：进 plan 前如果在 bypass 模式，isBypassPermissionsModeAvailable 会记住它，批准计划后无缝恢复（permissions.ts:1270）；plan 文件本身永远可写（checkEditableInternalPath）——不然模型连计划都写不出来。

bypassPermissions 则是被看管最严的一档：管理员可以通过 policySettings 让它整体不可用；服务端还有一个远程熔断开关（bypassPermissionsKillswitch.ts），在第一次查询前检查一次，必要时把整个会话降回受控模式。至于 dontAsk 的"问变拒"，实现位置很讲究——它在外层包装函数的最末尾做转换（permissions.ts:503），注释解释了原因："This is done at the end so it can't be bypassed by early returns"。转换放在入口，任何一条提前 return 的路径都可能漏掉它；放在出口，无论走哪条路径都逃不过。

（延伸，默认不开）源码里其实还藏着第六档 auto 模式，紧挨着 dontAsk 的转换之后（permissions.ts:519 起）：ask 请求不弹窗，改由一个 AI 分类器裁决，并按连续拒绝次数熔断（denialTracking）。它受特性门控、且被 isExternalPermissionMode 显式排除在外部可用模式之外，所以正文的"五个档位"不含它——但对做无人值守 Agent 的读者，"把'要不要问人'本身也交给一个小模型判断"是个可迁移的思路，值得知道它存在。

设计启示

给开发者模式和规则是正交的两层：模式改"没规则时的默认答案"，规则改"这一条的例外"，且规则永远赢。这个分层让"临时切 bypass 赶进度"是安全的——你的 deny 规则还在。设计自己的系统时警惕把两层揉在一起的诱惑（"bypass 就是全放行嘛"），那会让每次切模式都变成一次全量安全审计。

给产品经理档位是给用户的信任度旋钮，但旋钮拧到头也要保留几条不可豁免的红线（安全检查、显式 ask 规则）。用户会手滑、会忘记自己在哪个模式——红线是产品替用户兜的底。

7.4 弹窗之后——批准、写回与钩子的边界

流水线输出 ask 之后发生什么？useCanUseTool（useCanUseTool.tsx）接手，按运行形态把这个"问题"路由到三条去路：

机制图 ask 的三条去路与"总是允许"的写回 — 弹窗是入口，规则文件是出口
ask 按运行形态路由：协调器场景先让自动检查代审；swarm 队友把请求转发到领队屏幕（第六章的信箱协议）；本地则进弹窗队列。用户点"总是允许"时，附带的规则建议被持久化——下次同类调用直接在 7.1 的 2b 步放行。

弹窗上的四个选项里藏着两个产品级细节。"修改后允许"：用户可以直接编辑命令再批准，修改后的输入通过 updatedInput 一路替换到工具执行——不必"拒绝、口头纠正、等模型重试"绕一大圈。"总是允许"：ask 决策附带 suggestions（一组 PermissionUpdate），用户勾选即写回对应的设置文件。deny 同样有讲究：拒绝时的 message（规则级 deny 是固定串"Permission to use X has been denied."）会被当成一条 tool_result（is_error）回给模型（toolExecution.ts:1023、1031），模型知道"这条路被拒了"，才能换一条合规的路走，而不是原地重试。（createPermissionRequestMessage 负责的是 ask 弹窗/问询的文案，不是 deny 回执，别把两者混为一谈。）

最后是钩子。PreToolUse 钩子在权限判定之前运行（toolExecution.ts:800），它可以表态 allow/deny/ask，也可以只修改输入。它的权限边界由 resolveHookPermissionDecision（toolHooks.ts:332）一锤定音：

// resolveHookPermissionDecision —— 精简（toolHooks.ts:332）
// 源码注释：hook 'allow' does NOT bypass settings.json deny/ask rules

if (hook.behavior === 'allow') {
  const ruleCheck = await checkRuleBasedPermissions(tool, input, ctx)
  if (ruleCheck === null)            return hook       // 跳过弹窗，直接执行
  if (ruleCheck.behavior === 'deny') return ruleCheck  // deny 规则压过钩子
  return await canUseTool(...)                         // ask 规则：照样弹窗
}
if (hook.behavior === 'deny') return hook              // 钩子的否决立即生效
// 钩子说 ask 或沉默 → 走正常流水线（带上钩子给的理由进弹窗）

注意这个不对称：钩子的 deny 立即生效，allow 却要再过一遍规则。钩子是用户写的自动化脚本，它有资格替用户拒绝（拒绝永远安全），但无权替用户推翻用户自己写的 deny/ask 规则——两个"用户的意志"冲突时，显式配置的规则赢过运行时的脚本。

设计启示

给开发者钩子是自动化的用户，不是超级用户：任何自动批准通道（钩子、AI 代审、CI 机器人）的权限上限必须等于它代替的那个人，且永远低于显式否决。校验方法同 7.1——找"这条通道能不能绕过 deny"。

给产品经理弹窗的价值密度取决于"总是允许"的设计：一次弹窗应该顺手消灭一类未来弹窗，否则你是在训练用户无脑点确认——权限疲劳的终点是权限失效。

7.5 中止信号——Esc 如何传遍整棵 Agent 树

权限管"能不能开始"，中止管"能不能停下"。用户按下 Esc，此刻可能有一个模型请求在流式返回、三个工具在并发执行、一个 SubAgent 在深处调研、还有一个权限弹窗挂着等回答——全都要停，而且停完之后对话还得是合法状态、还能继续聊。Claude Code 用一件标准武器解决：AbortController，但用法有讲究。

机制图 中止信号的树形传播 — 单向下行：父停子必停，子停不连累父
中止信号从回合级 AbortController 单向下行：断流、杀进程、收弹窗、停同步子树；异步 SubAgent 故意不在传播树上。最关键的一步是善后——为悬空的 tool_use 合成配对的 tool_result，保住 API 协议不变量。

整套机制里最值得学的是善后这一步。第一章讲过 API 的硬性协议（一句话回顾：每个 tool_use 块必须有一个配对的 tool_result 块，以 user 角色注入下一轮）。用户在工具跑到一半时按 Esc，消息数组里就留下了没有结果的 tool_use——这个状态是不合法的，直接拿去调 API 会报错。所以 query 循环中止后做的第一件事，是给每个悬空的 tool_use 合成一条 "Interrupted by user" 的 tool_result（query.ts:1015 起的中止处理），再补一条 [Request interrupted by user] 的用户消息。代价是几行合成代码，换来的是：中止不是会话的死刑，只是一个回合的句号——用户下一句话可以直接接着聊。

另一个容易被低估的决定是作用域：AbortController 是回合级的（REPL 每个回合新建一个），不是会话级的。中止是"取消这一次"，不是"关掉这一切"——作用域选错，Esc 就从"后悔药"变成了"自毁按钮"。而同步与异步 SubAgent 的区别对待（共享 vs 独立 controller）延续了第四章的语义（一句话回顾：run_in_background: true 的派发即刻返回 task_id，后台独立运行）：同步派生是父任务的一部分，父停子必停；异步派生是独立的后台任务，它的生死应该由显式的 TaskStop 管理，而不是被父回合的 Esc 连坐。

设计启示

给开发者"取消"要当成协议特性设计，不是异常处理，三条铁律：①信号单向下行（子的失败不该炸掉父）；②每个挂起点都注册响应（模型流、工具进程、权限弹窗——漏一个就有"停不下来"的角落）；③中止后的状态必须仍然合法（合成 tool_result）。市面上"一按停止就只能重开会话"的 Agent，九成是丢了第③条。

给产品经理Esc 键是用户对一个自治系统最基本的安全感来源——"我随时能喊停"。但真正决定体验的是喊停之后：能不能立刻接着聊，而不是被迫重开会话丢掉全部上下文。"中止只是一个回合的句号，不是会话的死刑"，这句话应该成为任何 Agent 产品的中止体验标准；停得利落、停完还能接着干，用户才敢放手让它跑长任务。
