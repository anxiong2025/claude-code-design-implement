第九章

生成式交互模式——让 Agent 主动生成"提问"与"计划"

实现目标：把"向人提问""先规划再动手""让人看着选"变成模型可调用的工具，让人成为循环里一种"可被调用的能力"。

你让 Agent"给项目加个缓存"。它读了几个文件，然后停下来——屏幕上弹出一张卡片：缓存后端用哪种？ 左边三个选项 Redis / 内存 / 文件，右边跟着一栏真实配置预览：Redis 那栏写着"多进程共享、需多起一个依赖"，内存那栏写着"零依赖最快、重启即失"。你用方向键上下扫一遍对比，回车选了 Redis。Agent 拿到你的选择，接着往下写。它没有猜，也没有一条道走到黑写完再让你推翻重来——它在真正动手前，把一个只有你能拍板的决定，交还给了你。

这件事的奇妙之处在于："提问"本身是模型主动发起的一次工具调用。一句话回顾第三章的工具契约：工具是一个带 inputSchema、call() 的对象，模型填好参数发起调用，运行时执行后把结果以 tool_result（user 角色）回流进下一轮。前八章里，工具的返回值都是机器数据——文件内容、命令输出、子 Agent 的报告。本章的三件工具不一样：它们的"执行"就是把界面交给人，返回值是一个活人的选择。于是人被塞进了 ReAct 循环，成为和 Read、Bash 平级的一种"可被调用的能力"。

本章拆三件事，都是这个思路的变体：AskUserQuestion（问清楚——把模糊需求变成选择题）、并排预览（看着选——把选项旁边挂上可对比的实物）、Plan 模式（先规划再动手——把"要不要开工"变成一次显式审批）。三者共享同一个机制内核：一次 behavior:'ask' 的工具调用，把控制权和终端让给人，再把人的回应包成 tool_result 送回模型。

架构总图 人被接进 ReAct 循环：提问也是一次工具调用 — 返回值不是机器数据，是人的选择
模型在循环里遇到只有人能拍板的岔路，主动发起一次 behavior:'ask' 的工具调用；运行时把终端让给人渲染选择题/预览/计划；人的回应被包成 tool_result 回流进下一轮。人因此成为 ReAct 循环里一种可被调用的能力。

9.1 从流式到生成式交互——为什么"提问"是一个工具

先厘清一个容易混的概念。第一章讲过流式输出：模型逐块吐字，用户能实时看、随时打断——那是把模型的输出流式地展示给人。本章讲的是反方向的事：模型主动把一个决定权交还给人，并等人回应。前者是"看它干活"，后者是"它请你拍板"。

那"请你拍板"为什么要做成工具，而不是在产品代码里写一个 if 需要确认 then 弹窗？因为什么时候需要问、问什么、给哪几个选项，是语义判断，只有正在干活的模型知道。产品经理没法预先枚举"用户可能在哪一步遇到歧义"——那等于要提前想清楚所有任务的所有岔路。把提问做成工具，决策就落到了唯一有上下文的角色身上：模型读完代码，发现"缓存后端"这个选择它替用户做不了主，于是当场生成一道贴合此刻情境的选择题。这就是"生成式交互"四个字的重量——交互界面是模型按需生成的，不是产品预先写死的。

Claude Code 把这类工具的描述写得很直白（AskUserQuestionTool/prompt.ts:7）：

// src/tools/AskUserQuestionTool/prompt.ts:7
export const DESCRIPTION =
  'Asks the user multiple choice questions to gather information, ' +
  'clarify ambiguity, understand preferences, make decisions or offer them choices.'

// 工具提示词的四条用途（prompt.ts:32 起）
// 1. Gather user preferences or requirements
// 2. Clarify ambiguous instructions
// 3. Get decisions on implementation choices as you work
// 4. Offer choices to the user about what direction to take.

注意这四条没有一条是"到某个固定步骤就弹窗"，全是"当你（模型）发现自己需要"。工具只是给了模型一个开口，何时开口由模型的判断决定——这和把它写成产品流程里的固定关卡，是两种完全不同的产品哲学。

设计启示

给开发者当"要不要征询用户"取决于运行时的语义上下文时，别把它做成宿主代码里的固定分支——做成一个模型可调用的工具，让判断权落到唯一掌握上下文的角色手里。你付出的是一个 schema 和一段回流逻辑，换来的是"任意任务、任意岔路都能就地问"的通用能力。

给产品经理传统软件的交互是设计师预先画死的有限状态机；生成式交互里，界面（问哪些、给哪几个选项）由模型按当前任务临时生成。这解锁了"无法穷举的长尾场景"——你不必预想用户会在第几步卡住，模型会在它卡住的地方替你把选择题递到用户面前。评估这类产品，要看"该问的时候问了没、不该问的时候闭嘴没"，而不是界面画得多全。

9.2 Schema 约束与两条自动兜底

既然选择题由模型现场生成，就必须有一套 schema 把它约束成人能一眼看懂、手能快速选的形状。AskUserQuestion 的输入 schema 全是硬约束（AskUserQuestionTool.tsx:62）：

// src/tools/AskUserQuestionTool/AskUserQuestionTool.tsx —— 精简
const questionOptionSchema = z.object({
  label:       z.string(),          // 选项文字，1–5 词
  description: z.string(),          // 这个选项意味着什么 / 有何取舍
  preview:     z.string().optional() // 可选：并排预览内容（见 9.4）
})
const questionSchema = z.object({
  question:    z.string(),          // 完整问句，以问号结尾
  header:      z.string(),          // 极短标签，UI 里当 chip（≤12 字符）
  options:     z.array(questionOptionSchema).min(2).max(4),  // 2–4 个选项
  multiSelect: z.boolean().default(false)                    // 默认单选
})
const inputSchema = z.strictObject({
  questions:   z.array(questionSchema).min(1).max(4),        // 1–4 个问题
  ...commonFields                                            // answers/annotations/metadata
}).refine(UNIQUENESS_REFINE.check, { message: UNIQUENESS_REFINE.message })

每个数字都在替用户的注意力把关：一次最多 4 问（问题墙会把人吓退）、每问 2–4 个选项（少于 2 个不成其为选择，多于 4 个就该换个问法）、header ≤ 12 字符（它要塞进一个 chip 标签）。还有一条 .refine 唯一性校验（UNIQUENESS_REFINE，AskUserQuestionTool.tsx:32）：问题文本不能重复、同一问题内的选项 label 不能重复——否则用户根本分不清两个一模一样的选项差在哪。这些校验和第三章任何工具的入参校验走同一条路：结构不对，第一道关就打回，模型重发，压根轮不到打扰用户。

schema 之外，还有两条不写在 schema 里、但由代码或提示词强制的自动兜底：

机制图 两条自动兜底：永远有退路，推荐项永远在手边 — 模型只管出题，兜底由系统补
模型只负责出题；系统自动追加 "Other" 自由输入项（label "__other__"，QuestionView.tsx:206），并要求模型把推荐项放第一、加 "(Recommended)" 标注。前者保证用户永远有退路，后者让模型的主张明示且可否决。

第一条：永远有 "Other"。渲染选项时系统会在末尾自动追加一个自由输入项（QuestionView.tsx:206，label 为 "__other__"），提示词也明写"Users will always be able to select Other"。模型给的 2–4 个选项不可能穷尽现实，"Other" 是给用户留的逃生门——它不必在模型设想的框里选。第二条：推荐项置顶。提示词要求"若你有倾向，把它放第一并在 label 末尾加 (Recommended)"（prompt.ts:41）。这是一个微妙的产品平衡：模型该有主张（否则选择题变成甩锅），但主张必须明示且可被一眼否决，而不是藏在默认值里替用户做了主。

设计启示

给开发者让模型生成 UI，schema 就是安全带：数量上限（≤4 问、≤4 选项）防信息过载，唯一性 refine 防"两个看起来一样的选项"，字符上限护住布局。把这些约束放在 schema 层用校验强制，而不是写在提示词里靠模型自觉——校验失败能自动打回重试，提示词只能祈祷。

给产品经理"Other 永远在"和"推荐项标注"是两条可迁移的交互伦理：给 AI 生成的选项一定留自由输入的逃生门（否则用户被 AI 的想象力困住），让 AI 的倾向看得见、能一键推翻（否则"智能默认"就成了越俎代庖）。好的 AI 交互不是替用户决定，是把决定整理成好拍板的样子递过去。

9.3 答案回流——从 behavior:'ask' 到 tool_result

题出好了，接下来是最能体现"人是一种工具"的一环：人的选择怎么变回模型能读的数据。关键是一个权限判定的返回值 behavior:'ask'（AskUserQuestionTool.tsx:182）：

// src/tools/AskUserQuestionTool/AskUserQuestionTool.tsx —— 精简
async checkPermissions(input) {
  return { behavior: 'ask' as const, message: 'Answer questions?', updatedInput: input }
}
// 'ask' 不是"问要不要用这个工具"，而是"这次工具的执行就是问人"——
// 运行时据此把终端让给 AskUserQuestion 的界面，阻塞等人选完

// 人选完之后，答案被包成 tool_result（tsx:224）
mapToolResultToToolResultBlockParam({ answers, annotations }, toolUseID) {
  const answersText = Object.entries(answers)
    .map(([q, a]) => `"${q}"="${a}"`).join(', ')
  return {
    type: 'tool_result',
    content: `User has answered your questions: ${answersText}. ` +
             `You can now continue with the user's answers in mind.`,
    tool_use_id: toolUseID,          // 第一章的铁律：靠 id 配对
  }
}

回顾第七章的权限流水线（一句话回顾：每次工具调用前先过 checkPermissions，返回 allow / deny / ask 三种表态之一）。behavior:'ask' 在别的工具那里意思是"这步有风险，弹窗问用户批不批"；在 AskUserQuestion 这里，"问用户"本身就是工具的全部功能——运行时看到 ask，就把终端交给这个工具的界面，阻塞循环，等人选完。人选完后，选择经由 mapToolResultToToolResultBlockParam 拼成一句自然语言塞进 tool_result，以 user 角色、按 tool_use_id 配对回流——和第一章讲的任何工具结果注入，一模一样。模型下一轮读到的不是"用户点了按钮"这种 UI 事件，而是"用户回答了你的问题：'缓存后端用哪种？'='Redis'"这种它天然就懂的话。

细节里藏着好设计：多选的多个 label 用 ", " 连接成一个字符串（AskUserQuestionPermissionRequest.tsx:426）；单问题单选且立即提交时直接回流、免掉"下一题"的翻页；选了 "Other" 则用用户的自由输入文本替代 label。无论哪条路径，最终都收敛成 answers 这个 问题→答案 的字典，再拼成那一句话。UI 的复杂性（方向键、多选、翻页、自由输入）全部在回流这一步被抹平成模型只需要读一句话。

设计启示

给开发者人机交互的返回值要"翻译成模型的母语"。别把原始 UI 事件（点击了第几个按钮、勾选了哪些框）丢给模型，那需要它反推语义；直接拼成 "问题"="答案" 的自然语言，模型零成本理解。复用既有的 tool_result 通路（而非发明一种新消息类型）意味着人的输入和文件读取、命令输出在循环里完全同构——不需要为"人"在循环里开一条特殊分支。

给产品经理无论交互界面多花哨（并排预览、多选、翻页），回到模型那里都收敛成一句"用户选了 X"。这条"UI 复杂、回流简单"的分界线是可复制的：交互层可以尽情为人优化体验，但喂回智能层的必须是被提炼过的语义。这也解释了为什么加一种新的问法（比如滑块、日期选择器）不需要动模型——回流契约不变，模型无感。

9.4 并排预览——让人"看着实物选"

开篇那张卡片右边的配置预览，是这套机制里体验最重的一块。当选项之间的差别"说不清、要看才懂"时（三种缓存后端的配置、两种布局的样子、几段实现的代码），光有一行文字描述不够——得把实物摆在选项旁边让人对比。这就是 preview 字段的用途：给某个选项挂一段预览内容，UI 自动切换成左右并排的布局。

// src/components/permissions/AskUserQuestionPermissionRequest/QuestionView.tsx:235
const hasAnyPreview = !question.multiSelect && question.options.some(o => o.preview)
if (hasAnyPreview) { /* 渲染 PreviewQuestionView：左选项、右预览 */ }

// PreviewQuestionView.tsx:233 —— 左右分栏的宽度就俩常量
const LEFT_PANEL_WIDTH = 30          // 左栏：选项列表固定 30 列
const GAP = 4                        // 中间留 4 列空隙
const previewMaxWidth = columns - LEFT_PANEL_WIDTH - GAP   // 右栏吃掉剩余终端宽度

两个细节值得停一下。其一，hasAnyPreview 的第一个条件是 !multiSelect——预览只对单选开放。因为并排预览的心智模型是"高亮某一项 → 右边显示它的实物"，一次只能聚焦一项；多选要同时勾好几个，"右边该显示谁"没有答案。源码在多选路径里干脆连 "Other" 都不追加（PreviewQuestionView.tsx:69："Only real options — no 'Other' for preview questions"），保持并排视图的纯粹。其二，预览内容要么是 markdown、要么是 HTML 片段，由 getQuestionPreviewFormat()（定义在 bootstrap/state.ts:1120，被 AskUserQuestionTool.tsx import 后调用）决定；HTML 模式下还有一道 validateHtmlPreview（定义在 AskUserQuestionTool.tsx:250）把关：禁止 <html>/<body>/<!doctype> 整页、禁止 <script>/<style>——预览是嵌进宿主界面的片段，不能让模型生成的内容跑脚本或改样式污染整个页面。

机制图 并排预览：左栏定宽选项，右栏动态实物 — 只对单选开放
左栏是固定 30 列的选项列表，右栏吃掉终端剩余宽度显示当前高亮项的预览实物。上下移动高亮，右栏实时切换。预览只对单选开放，因为多选无法确定"右边显示谁"；HTML 预览经 validateHtmlPreview 限制为安全片段。

把这块拆开看，它回答的是一个很产品的问题：用户凭什么做这个选择？纯文字选项要求用户在脑子里想象每个选项的后果；并排预览把后果直接摆出来——选 Redis 是这份配置、选内存是那份配置，差异肉眼可见。决策质量因此提升，而这一切依然由模型生成：预览内容是模型现场写的配置/代码/mockup，不是产品预先备好的素材。

设计启示

给开发者让 AI 生成的可视内容嵌进宿主 UI，一定要有一道"片段而非整页"的校验：禁 <html>/<body>、禁 <script>/<style>。源码注释说得实在——HTML5 解析器按规范会容错吞下几乎任何东西，所以这道校验不是解析，是在检查模型意图并挡掉明确不该出现的东西。生成式 UI 的安全边界要自己划，别指望渲染器兜底。

给产品经理"看着选"比"读着选"决策质量高一个量级，尤其当选项差异是"要看才懂"的那种。但预览有代价——它只对单选开放，因为并排视图一次只能聚焦一项。这里的取舍很典型：为了让对比足够清晰，宁可牺牲多选的灵活。做 AI 选择类交互时，先问"选项差异用一行字说得清吗"，说不清才上预览，且默认单选。

9.5 Plan 模式——把"先规划再动手"做成显式审批

AskUserQuestion 解决"某个具体决定该问谁"；还有一类更大的信任问题：整件事该不该现在就开工。你说"给应用加用户认证"，这不是一道选择题能收口的——它牵扯架构选择、要动好几个文件、做错了返工成本高。这时候合适的不是问一道题，而是让 Agent 先探索、先出一份完整计划，等你点头再动手。这就是 Plan 模式，由一对工具支撑：EnterPlanMode（进）和 ExitPlanMode（出）。

// src/tools/EnterPlanModeTool/EnterPlanModeTool.ts —— 精简
const inputSchema = z.strictObject({})   // 空 schema：进 plan 模式不需要任何参数
async call(_input, context) {
  if (context.agentId) {
    throw new Error('EnterPlanMode tool cannot be used in agent contexts')  // 子 Agent 不能进
  }
  handlePlanModeTransition(appState.toolPermissionContext.mode, 'plan')
  // 切成 plan 模式：只读探索，除计划文件外不许写任何文件
}

// src/tools/ExitPlanModeTool/ExitPlanModeV2Tool.ts —— 精简
// 注意：ExitPlanMode 不把计划当参数传——计划早已写进磁盘上的计划文件，
// 这个工具只是"我写完了，请审批"的信号
async checkPermissions() {
  return { behavior: 'ask' as const }    // 又是 'ask'：把计划推给用户批准
}

两个细节点出 Plan 模式的定位。其一，EnterPlanMode 的输入 schema 是空的——进入规划态不需要模型提供任何东西，它就是个纯粹的模式开关；而且 context.agentId 存在（即子 Agent 上下文）时直接抛错，只有主 Agent 能进 plan 模式（一句话回顾第四章：子 Agent 是隔离的执行单元，没有终端也没人给它批计划）。其二，ExitPlanMode 不接收计划内容作参数——计划早已被模型写进磁盘上的计划文件，这个工具只发一个"写完了，请审批"的信号，同样走 behavior:'ask' 把计划推到用户面前。审批通过后，权限模式从 plan 恢复成进入前的模式（prePlanMode），Agent 才获得写权限开始动手。

最见功力的是 AskUserQuestion 和 ExitPlanMode 的分工，提示词把边界划得很死（ExitPlanModeTool/prompt.ts:22）：

// src/tools/ExitPlanModeTool/prompt.ts:22
Do NOT use AskUserQuestion to ask "Is this plan okay?" or "Should I proceed?"
- that's exactly what THIS tool does. ExitPlanMode inherently requests approval.

// src/tools/EnterPlanModeTool/prompt.ts:54
If you would use AskUserQuestion to clarify the approach, use EnterPlanMode instead

翻译成一句话：AskUserQuestion 管"澄清"，ExitPlanMode 管"批准"，两者不能互相冒充。规划阶段有具体拿不准的点（用哪种认证？OAuth 还是 JWT？）——用 AskUserQuestion 问,这是澄清；计划整体写好了要用户拍板——用 ExitPlanMode，这是批准。反过来都是错的：拿 AskUserQuestion 去问"我的计划行不行"是把审批降格成了一道选择题（而且用户在点 ExitPlanMode 之前根本看不到计划全文）；在还没搞清方向时就 ExitPlanMode，是拿一份自己都没想清的计划去要签字。同一个 behavior:'ask' 内核，两种工具按"澄清 vs 批准"的语义分工——这正是生成式交互的成熟之处：不只有"问"这一个动作，而是把人在决策链上的不同角色（顾问、审批人）都建模成了工具。

设计启示

给开发者把"人的介入"再往下切分：澄清（提问、可多次、局部）和批准（一次、全局、有法律感的"我同意开工"）是两种不同的交互，别用一个工具混着做。EnterPlanMode 空 schema + 子 Agent 禁用 + ExitPlanMode 不传计划正文（从文件读），这些约束共同保证了"规划态"是一个干净的、只读的、可被审批的阶段，而不是"边想边改还顺手写了几个文件"的糊状态。

给产品经理高风险、高返工成本的任务，正确的交互不是"边做边问一堆小问题"，而是"先给一份完整方案、一次性审批"。这对应两种产品节奏：澄清是低摩擦、随时可发生的；批准是高仪式感、集中一次的。把它们分开，用户既不会被碎问题打断，也不会在没看到全局计划时被要求签字。评估 Agent 产品的"可控性"，要看它是否在真正动手前给了用户一个看得见、能否决的全局计划关卡。

本章代码

本章的示例在 code/examples/askuserquestion_demo.py。它在本地完整复刻 AskUserQuestion 的一次调用：schema 校验（1–4 问、2–4 选项、唯一性）、自动追加 "Other"（预览题除外）、推荐项置顶、并排预览渲染、多选/单选/Other 三种答案收集，以及最后把选择包成 tool_result 回流——纯本地运行，无需 API Key。

# 纯本地运行，无需 API Key
cd code
uv run python examples/askuserquestion_demo.py
