第一章

Agent Loop——内核

如简介中的架构图（Claude Code · Architecture）所示，Agent Loop（ReAct 推理循环）只有两个构成要素：一个 while 循环，一个只增不减的消息数组。Harness 运行时、工具系统、多 Agent 编排均以此循环为核心依赖——它们的存在是为了让这个循环运行得更稳定、更持久、更具扩展性。本章先将 Agent Loop 的结构与设计原因讲清楚，后续各章的分析才有基准。

要理解为什么需要这样一个循环，需要先明确一个前提：LLM 本身是无状态的——每次 API 调用相互独立，模型不保留上次调用的任何信息。对于"找出项目里所有 TODO 并逐一修复"这类需要十几步工具调用的任务，单次调用显然不够，需要一套机制驱动模型持续执行直到任务完成。

这套机制的最小实现只需 25 行 Python：

# 这 25 行就是 Claude Code src/query.ts 的最小骨架
# 后面五节会逐行解释每处设计背后的 "为什么"

import anthropic, json

client = anthropic.Anthropic()
TOOLS = [{"name": "bash", "description": "执行 shell 命令", "input_schema": {
    "type": "object", "properties": {"cmd": {"type": "string"}}, "required": ["cmd"]
}}]

def run_tool(name, input):        # ④ 工具执行（1.4 节）
    if name == "bash":
        import subprocess
        return subprocess.run(input["cmd"], shell=True, capture_output=True, text=True).stdout

def agent_loop(user_msg, max_turns=10):
    messages = [{"role": "user", "content": user_msg}]   # ① 状态载体（1.1 节）
    turn = 0
    while True:                                            # ② while(true)（1.2 节）
        resp = client.messages.create(                     # ③ 调模型（1.3 节）
            model="claude-opus-4-8", max_tokens=4096,
            tools=TOOLS, messages=messages, stream=True
        )
        blocks, tool_uses = [], []
        for event in resp:                                 # ③ 流式逐块接收
            if hasattr(event, "content_block"):
                blocks.append(event.content_block)
            elif hasattr(event, "delta") and event.delta.type == "input_json_delta":
                blocks[-1].input = getattr(blocks[-1], "input", "") + event.delta.partial_json
        for b in blocks:
            if b.type == "tool_use":
                tool_uses.append(b)                        # ③ 亲眼见到 tool_use 才算数
        if not tool_uses:                                  # ② 唯一正常出口
            return next(b.text for b in blocks if b.type == "text")
        results = [{"type": "tool_result", "tool_use_id": b.id,  # ④ 结果配对
                    "content": run_tool(b.name, json.loads(b.input))} for b in tool_uses]
        messages += [{"role": "assistant", "content": blocks},    # ① 三段拼接
                     {"role": "user",      "content": results}]
        turn += 1
        if turn >= max_turns:                              # ⑤ 熔断（1.5 节）
            return f"[已达最大轮次 {max_turns}，任务未完成]"

print(agent_loop("列出当前目录的所有 .py 文件，并统计总行数"))

1.1 ① 消息数组——Agent 的唯一记忆

骨架第一行：

messages = [{"role": "user", "content": user_msg}]   # ① 状态载体

为什么要这么一个数组？因为 LLM 是无状态服务——每次 API 调用都是独立的，模型不记得上次你们说了什么。你发什么，它就只看到什么，上一轮的内容除非你重新带过去否则它完全不知道。

所以 Claude Code 的做法朴素到近乎粗暴：把所有说过的话全存进一个数组，每轮追加新内容，下一轮把整个数组重新发给模型。记忆不在模型里，在调用方手里。对应源码：

// query.ts:205
type State = {
  messages: Message[]   // ← 本次请求的完整对话历史（跨请求的持久存储见本节末尾）
  turnCount: number     // 轮次计数（1.5 节讲）
  // ...还有 8 个字段，全是"恢复机制"的账本，下面解释
}

初学者读源码时最容易被 State 里那一堆字段吓退。这里给一个减负的读法：这 10 个字段里，只有 messages 是"业务状态"，其余 9 个几乎全是"恢复用的临时账本"——记录"上一轮为什么要重试""compact 压缩到哪了""max_tokens 恢复过几次"。它们每轮基本都会重置，属于第二章 Harness 的话题。第一章你只需盯住 messages 这一个字段。

循环每转完一圈，末尾把三类新消息拼进数组，构成下一轮的输入（query.ts:1716）：

// query.ts:1715-1716
const next: State = {
  messages: [
    ...messagesForQuery,   // ① 本轮送入模型的历史
    ...assistantMessages,  // ② 本轮模型返回的内容（含 tool_use）
    ...toolResults,        // ③ 本轮工具执行结果（以 user 角色注入）
  ],
  turnCount: nextTurnCount,
  // ...其余字段重置
}
state = next   // 下一轮迭代从这里读
数据流 · 悬停高亮 下一轮 messages = 三段拼接 — 对照上面 query.ts:1716
每轮循环末尾，三段消息按 [...历史, ...模型输出, ...工具结果] 拼成新数组——这就是 Agent「记忆」增长的全部机制。

一个容易被忽视、但很能说明设计意图的细节：拼进去的是 messagesForQuery，不是 state.messages 原样。messagesForQuery 是原始历史经过预处理（压缩、截断、裁剪超大工具结果）之后的版本。这意味着——"完整历史"和"真正喂给模型的历史"是两回事。第一章我们默认它们相等，把它当"全量追加"来理解；第二章会看到 Harness 如何在这一步动手脚来控制 token。

设计启示

给开发者把"状态"和"计算"彻底分离：模型负责无状态的计算，状态收敛到调用方的一个数组里。这带来三个白捡的好处——可持久化（存下数组就能续上对话，见第八章 Session）、可回放（重放数组就能复现任何一轮）、可裁剪（要省 token，动数组就行）。凡是你自己要设计的 Agent，第一件事就是想清楚"状态数组长什么样"。

给产品经理"AI 记不住上文"从来不是模型缺陷，而是产品有没有把历史完整带上的工程选择。当你评估一个 Agent 产品的上下文能力时，真正该问的是："它的历史数组保留策略是什么？超长了怎么裁？"——这决定了体验的连贯性上限。

概念 · QueryEngine 与 queryLoop 的层级关系

上面"messages 是 Agent 记忆的载体"这句话需要补一个精确边界：query.ts 里的 State.messages，只是一次用户请求期间的工作副本，不是跨越整个会话、真正持久存在的那份记忆。持续跨越"用户发第 1 条消息、第 2 条……"存在的容器，是更外层的 QueryEngine（QueryEngine.ts:184）——它的实例字段 mutableMessages（QueryEngine.ts:202）才是会话级、贯穿整个 QueryEngine 生命周期的历史。

两者的分工：用户每发一条消息，QueryEngine 先把它 push 进 mutableMessages（QueryEngine.ts:431），再复制一份快照传给 query()（QueryEngine.ts:434, 675）——这才是本章讲的 while(true) ReAct 循环，只为这一条用户消息运行，内部可能转很多轮工具调用，但终归以 completed 退出。循环每 yield 一条消息，QueryEngine 就把它追加回 mutableMessages（QueryEngine.ts:768 等处）；下一条用户消息到来时，再重复这个过程。

所以严格地说：本章后面 1.5 节讲的 turnCount，数的是"一次用户请求内部的工具调用轮数"，不是"聊了几条消息"——这是两个不同量级的计数，很容易被默认成同一件事。第一章聚焦的，是 QueryEngine 每次调用里的这一段 queryLoop；QueryEngine 本身如何管理跨请求的会话生命周期，属于第八章 Session 持久化的话题。

1.2 ② while(true)——为什么是无界循环

骨架里：

while True:   # ② 循环本体，不设上界是刻意的

为什么不是 for i in range(max_turns)？因为不知道要转几圈才能完成任务——这由模型决定，不由框架预设。这个"边想边做边根据结果调整"的范式叫 ReAct，来自 2022 年 Yao 等人的论文，它解决了纯 CoT 推理的致命问题：模型闭眼脑补容易产生幻觉，而 ReAct 每一步都拿真实结果"校准"。

要真正讲清楚 ReAct，得先认识它的"前身"——思维链 CoT。

概念 · 思维链 CoT

思维链（Chain-of-Thought，CoT）是一种提示技术：让模型在给出答案之前，先用自然语言把中间推理一步步"说出来"（经典触发语是"让我们一步步思考"）。由 Google 的 Wei 等人在 2022 年提出，它显著提升了大模型在数学、逻辑等复杂任务上的表现。

但 CoT 有个致命短板：它只会"想"，不会"查"。模型闭着眼睛推理，中间步骤全靠记忆里的知识"脑补"，一旦某个事实记错，整条推理链就跟着错到底——这正是大模型"幻觉"的高发区。

ReAct 就是为补上这个短板而生的。

概念 · ReAct 是什么

ReAct（Reasoning + Acting，推理 + 行动）是一种构建 AI 智能体的框架，核心是把 CoT 推理 和 调用外部工具的行动 交织起来：模型每"想"一步，就可以"做"一件事（查资料、调 API、执行命令），再把做的真实结果拿回来接着想。这样推理就不再是闭门造车，而是每一步都被真实世界的反馈"校准"。

它由 Shunyu Yao 等人（普林斯顿大学与 Google Research）在论文《ReAct: Synergizing Reasoning and Acting in Language Models》中提出（arXiv 2022 年 10 月首发，2023 年 ICLR 正式发表）。

它从哪来：一个打包行李的类比。ReAct 的灵感来自人类怎么用"内心独白"处理复杂任务。想象你在为短途旅行打包：先冒出一个念头（想："目的地天气怎么样？"），于是去查天气预报（做），看到"会很冷"（观察）；这个观察引出下一个念头（想："我有哪些保暖衣服？"），你去翻衣柜（做），却发现"厚衣服都收在储藏室了"（观察，一个意外），于是再调整下一步。你不是照着一份僵死的清单执行，而是边想边做、根据新信息动态调整——这就是 ReAct 的精髓。

落到形式上，ReAct 把智能体的活动组织成三个交替进行的动作：

  • Thought（思考 / 推理）——就是 CoT 那一步：模型用自然语言把大任务拆成小步，想清楚下一步该干什么
  • Action（行动）——模型发起一次工具调用（查搜索引擎、调 API、读知识库……；在 Claude API 里就是一个 tool_use block）
  • Observation（观察）——把行动的真实结果喂回模型（在 Claude API 里就是 tool_result），它据此重新评估进展，决定是给出答案，还是进入下一轮 Thought

为什么非得这样交替，不能只想或只做？因为"只想"和"只做"各有致命缺陷，而 ReAct 恰好让一方补上另一方的短板：

做法问题所在
只有推理（纯 CoT）脱离真实信息，容易一本正经地胡编（幻觉）
只有行动（无推理地堆工具调用）不会规划、不会根据结果调整，遇到意外就卡死
ReAct（推理 ⇄ 行动）推理有真实信息兜底 → 更少幻觉、能适应意外、且推理过程可读可调试

正是这几点，让 ReAct 成了当今几乎所有主流 Agent（包括 Claude Code）的理论底座——它第一次让大模型从"会聊天"迈向"能自主解决问题"。

重要澄清 · 别被"ReAct 提示"绕晕

你去看 ReAct 原论文或 LangChain 的 ZERO_SHOT_REACT_DESCRIPTION，会发现智能体是靠一段文本格式的提示词工作的——模型被要求逐字输出 Thought: … / Action: … / Observation: …，框架再用字符串解析把 Action 抠出来执行。这叫 ReAct 提示（ReAct prompting）。

但 Claude Code 并不用这种文本格式。它走的是 Anthropic 的原生工具调用（function calling / tool use）：模型直接输出结构化的 tool_use block（带机器可读的 JSON 参数），无需靠正则去解析文本，更稳、更省 token。

所以要分清两层：ReAct是"循环范式"（想→做→观察的反馈回路），function calling是"行动的实现机制"。Claude Code = ReAct 的循环骨架 + function calling 的落地方式。这也解释了为什么你在 Claude Code 的消息里看不到 Thought: 这种文本——它把"想"和"做"编码进了结构化的 message，而不是纯文本。

三步一圈，循环往复，直到模型在某一轮的 Thought 里判断"任务完成了"，不再发出 Action 为止。Claude Code 的实现就是一个 while(true)（query.ts:307）——正是章首那张可以点的循环图。每次迭代严格对应一个完整的 Thought → Action → Observation 周期，落到源码是这六步：

  1. 预处理 messages（压缩 / 裁剪，第二章展开）
  2. [Thought] 调用模型、流式接收：text block 是推理文字，tool_use block 是行动决定
  3. 本轮出现 tool_use 了吗？—— 否 就 return 'completed'（循环唯一的正常出口）
  4. [Action] 执行工具
  5. [Observation] 把 tool_result 注入 messages
  6. turnCount++，检查是否超 maxTurns，然后 continue 回到第 2 步

到这里，"何时停"这个问题浮出水面：循环怎么知道该转下一圈，还是该收工？直觉上你会想：看模型返回的 stop_reason 是不是 'tool_use' 不就行了？这正是所有人自己实现时都会踩的坑。Claude Code 源码在这里留了一句醒目的注释（query.ts:554）：

// query.ts:554-556
// Note: stop_reason === 'tool_use' is unreliable -- it's not always set correctly.
// Set during streaming whenever a tool_use block arrives — the sole
// loop-exit signal. If false after streaming, we're done ...

翻译过来：stop_reason 这个字段不可靠，不总是被正确设置，不能拿它当循环判据。Claude Code 的做法是自己维护一个布尔标志 needsFollowUp——在流式接收过程中，只要亲眼解析到一个 tool_use block，就把它置为 true（query.ts:832-834）：

// query.ts:826-835
if (message.type === 'assistant') {
  assistantMessages.push(message)
  const msgToolUseBlocks = message.message.content.filter(
    content => content.type === 'tool_use',
  )
  if (msgToolUseBlocks.length > 0) {
    toolUseBlocks.push(...msgToolUseBlocks)
    needsFollowUp = true   // ← 循环是否继续，只认这个信号
  }
}

"我实际收到了工具调用"是自己观察到的事实，"服务端告诉我它停下是因为工具调用"是外部的声明——Claude Code 选择相信前者。

设计启示

给开发者这是一条能迁移到任何系统的防御式设计原则：控制流的判据，优先用"你亲自观测到的事实"，而不是"上游承诺的元数据"。元数据可能缺失、可能语义漂移、可能因协议升级而失真；而"我这轮到底解析出没有工具调用"是确定的。你自己写 Agent 时，别把循环命脉押在 stop_reason 上。

1.3 ③ 流式逐块接收——为什么不能等全量

骨架里调模型时传了 stream=True，然后逐事件遍历：

for event in resp:                                 # ③ 流式逐块接收
    if hasattr(event, "content_block"):
        blocks.append(event.content_block)
    elif hasattr(event, "delta") and event.delta.type == "input_json_delta":
        blocks[-1].input = getattr(blocks[-1], "input", "") + event.delta.partial_json

为什么不直接用非流式、等模型输出全了再处理？两个原因：体验——用户能看到文字逐字出现；可中断——用户随时 Ctrl+C，在下一个 block 边界干净停下来。这个机制背后是一个绕不开的协议：

概念 · SSE 流式响应

SSE（Server-Sent Events，服务端推送事件）是一种基于 HTTP 的单向流式协议：服务端不必等整个响应算完，而是一边生成一边把内容分成许多小事件持续推给客户端。大模型逐 token 生成文字，天然适合用 SSE 边生成边下发。

为什么要流式，而不是算完一次性返回？一次完整回答可能要生成几十秒；若一次性返回，用户只能对着空白等待，也无法中途干预。流式则让"生成"和"消费"同时进行——这是下面两个产品级好处的技术前提。

Claude Code 收到的就是这样一条 SSE 流，内容一块一块（block by block）陆续到达。Claude Code 的策略是：每拼装完一个 content block，就立刻把它 yield 出去，绝不等整条 message 收完。为什么要这么做？两个产品级的理由：

  • 体验——用户能看到文字一个字一个字冒出来，而不是盯着转圈等十几秒
  • 可中断——正因为是逐块处理，用户随时按 Ctrl+C，循环能在下一个 block 的边界干净地停下（这正是 1.5 节 aborted_streaming 退出路径的由来）

流式处理的核心是一个状态机，按到达的事件类型逐步拼装每个 block：

for await (const part of stream) {
  switch (part.type) {
    case 'content_block_start':
      // 新 block 开始。tool_use 的 input 先初始化为空字符串，
      // 因为它的参数是流式拼接的 JSON，还没收全
      blocks[part.index] = { ...part.content_block, input: '' }
      break

    case 'content_block_delta':
      // 增量到达，追加到对应 block：
      //   text_delta       → block.text  += delta.text
      //   input_json_delta → block.input += delta.partial_json
      break

    case 'content_block_stop':
      // 一个 block 收全了 → 立刻组装成 AssistantMessage 并 yield
      // query.ts 收到后马上检查它是不是 tool_use，决定 needsFollowUp
      yield assembledMessage
      break
  }
}

这里有两个值得记住的工程细节：

第一，tool_use 的参数是"流式拼接的 JSON 字符串"。模型不会一次性给你一个完整的参数对象，而是把 {"city": "北京"} 这段 JSON 一个片段一个片段地发过来，拼在 input 字符串上。只有等 content_block_stop、整段收全了，才能 JSON.parse 成真正的参数对象。这解释了一个常见困惑："为什么不能边收工具参数边执行？"——因为在收全之前，你手里的参数是不完整、无法解析的。

第二，逐块 yield 意味着一条 assistant message 可能包含多个 block：先来一段 text（模型的推理），再来一个或多个 tool_use（模型的决定）。这也是为什么模型能在一轮里同时发起多个工具调用——下一节就用到这一点。

设计启示

给产品经理"流式输出"不是锦上添花的动效，而是 Agent 可交互性的地基。有了逐块流，才谈得上"实时展示进度""随时打断""看着它一步步干活"这些让 Agent 从黑盒变成协作者的体验。评估 Agent 产品时，"能不能中途喊停、喊停后状态是否干净"是一条硬指标。

1.4 ④ 工具结果注入——为什么是 user 角色

骨架里执行完工具后这样构造结果：

results = [{"type": "tool_result", "tool_use_id": b.id,   # ④ 靠 id 配对
            "content": run_tool(b.name, json.loads(b.input))} for b in tool_uses]
messages += [{"role": "assistant", "content": blocks},    # ① 拼回数组
             {"role": "user",      "content": results}]   # tool_result 是 user 角色！

初学者最困惑的地方：工具结果为什么是 user 角色而不是 system 或 function？模型发出了 tool_use，现在轮到最后一个环节：怎么把结果还给模型？

概念 · 工具调用协议

Anthropic 的工具调用协议规定了一次工具往返的固定形态：模型在 assistant 消息里发出一个 tool_use block（含唯一的 id、工具名 name、参数 input）；调用方执行后，必须回一个 tool_result block，并用 tool_use_id 指回刚才那个 id。

为什么要这么严格？因为模型是无状态的（回顾 1.1）——它不"记得"自己发过哪个调用，只能靠消息里的 id 把"我问的"和"你答的"对上号。这个协议就是模型和调用方之间的一纸契约。

而这纸契约里，藏着一个初学者常觉得"反直觉"的约定：工具结果不是以某种"系统消息"或"函数返回"的身份传回，而是被包装成 tool_result block、以 user（用户）角色追加进消息列表。这是 Anthropic API 的强制协议：每个 tool_use 必须有一个 tool_use_id 完全相同的 tool_result 与之配对，且必须紧跟在那条 assistant 消息之后。

下面是一次完整 ReAct 循环之后，messages 数组的真实形态（顺便演示 1.3 说的"一轮里发起多个工具调用"）：

# 用户要"查北京和上海的天气再比较"，模型一轮里发起两个工具调用

  ]},

  {"role": "user", "content": [

把结果放回数组之前，还要过一道 normalizeMessagesForAPI（query.ts:1396）——它负责把 Claude Code 内部的消息格式转成 API 认的"线上格式"：过滤内部专用的虚拟消息、合并相邻的 user 消息（某些后端如 Bedrock 的兼容要求）、清理超大媒体、剥掉内部字段。这层"内外格式转换"你现在不必细究，只需知道：内部消息模型 ≠ API wire 格式，中间隔着一层规整化。

最后点一个上面例子已经埋下的伏笔：同一轮的多个工具调用是并发执行的（read-only 工具走并发，并发上限由环境变量 CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY 控制，默认 10）；而会改动文件/状态的 write 类工具必须串行。为什么这样区分、调度器怎么实现，是第三章"工具系统"的主题。

设计启示

给开发者注意这个协议设计的精妙：用 tool_use_id 做配对，而不依赖顺序。正因如此，多个工具才能乱序并发、谁先返回都不影响正确性——只要 id 对得上就能归位。你设计任何"发起—回执"式的异步接口时，都值得学这一招：用显式 ID 配对，而非隐式顺序。

1.5 ⑤ 熔断与全部退出路径

骨架末尾：

turn += 1
if turn >= max_turns:                              # ⑤ 熔断
    return f"[已达最大轮次 {max_turns}，任务未完成]"

循环的正常出口只有一个：模型不再发 tool_use（completed）。但万一模型陷入死循环，一遍遍调工具却始终给不出结论呢？任何 while(true) 都必须配一个熔断闸，否则就是定时炸弹。这就是 maxTurns（Python 骨架里的 max_turns，源码里的 maxTurns 参数）。

概念 · 熔断 circuit breaker

熔断（circuit breaker）这个词借自电路里的保险丝：当电流超过安全阈值，保险丝主动熔断，牺牲局部来保住整体。软件里沿用了这个思想——给一个可能失控的过程设一道硬性上限，一旦触及就强制停止，用一个可控的失败，换掉一个不可控的灾难（无限循环、账单爆炸、资源耗尽）。

在 ReAct 智能体里，"该在什么时候、用什么条件结束循环"本就是一个核心设计问题。设一个最大迭代次数（maxTurns）是最简单直接的一种熔断手段。

先说清 turnCount 的语义，它容易被误解：它从 1 起步，含义是"即将向模型发起第几次调用"——初始用户请求是 turn 1，第一次工具 follow-up 是 turn 2，以此类推。每当有工具结果、循环准备再转一圈时，在末尾 +1 并检查上限（query.ts:1705）：

// query.ts:1679, 1705-1711
const nextTurnCount = turnCount + 1

if (maxTurns && nextTurnCount > maxTurns) {
  yield createAttachmentMessage({ type: 'max_turns_reached', maxTurns })
  return { reason: 'max_turns', turnCount: nextTurnCount }
}

completed 和 max_turns 只是最常见的两个出口。一个成熟的循环，退出路径远不止"正常结束"和"跑太久"——它要应对中断、报错、上下文超限、Hook 拦截等各种现实。下面是 Claude Code 全部的 Terminal 退出路径：

设计启示

给开发者把一堆 if 散落各处，不如像 Claude Code 这样：让循环体的每个出口都返回一个带 reason 的结构化结果。退出路径一旦被显式枚举成一张表，就变得可测试、可监控、可穷尽——你能对着这张表逐条问"这个分支我处理了吗"。

给产品经理Agent 会失败，这是必然，不是意外。产品的成熟度，恰恰体现在"失败得体不体面"：是无声地卡死，还是明确告诉用户"我达到最大步数了/上下文太长了/被你打断了"？上面这张退出路径表，本质上就是一份"Agent 会以哪些方式收场"的产品清单——每一种都对应一句该给用户的话。

2.1 queryLoop 的职责边界

源码里有两个长得很像的函数——query() 和 queryLoop()。初看容易当成同一件事。它们的真实分工是：

  • query()：薄壳。唯一的业务逻辑是把 consumedCommandUuids 的生命周期通知分成"正常完成"和"中途异常"两个信号
  • queryLoop()：主引擎。整个 Harness 的实现，async generator，驱动 while(true) 循环
// query.ts:219–239 — query() 是个薄壳
export async function* query(params: QueryParams) {
  const consumedCommandUuids: string[] = []
  const terminal = yield* queryLoop(params, consumedCommandUuids)
  // yield* 之后的代码只在"正常 return"时执行
  // throw / .return() 关闭时，以下代码被跳过 → 不发 completed 信号
  for (const uuid of consumedCommandUuids) {
    notifyCommandLifecycle(uuid, 'completed')
  }
  return terminal
}

为什么要这样分？因为 consumedCommandUuids 的生命周期信号是不对称的：正常完成发 completed，抛异常或用户中断时不发（而是发 started-without-completed）。yield* 的语义天然实现了这个不对称——throw 穿透、.return() 关闭时，yield* 之后的代码都不会执行。

概念 · async generator 与 yield*

async generator 是一种同时支持 await（异步）和 yield（惰性产出）的函数，调用后返回一个异步迭代器，每次 next() 才推进到下一个 yield。yield* 是"委托"——把当前 generator 的所有产出都转交给另一个 generator，直到对方结束才继续。它天然保留了 throw 穿透和提前关闭语义，这是 Claude Code 用它来分离正常结束和异常两条路径的技术基础。

进入 queryLoop 后，第一件事是快照所有"不可变配置"，然后构建"可变状态" State：

// query.ts:268–279 — 可变状态，整轮原子替换
let state: State = {
  messages: params.messages,          // 消息历史（核心，见第一章）

2.2 Prompt 组装流水线

第一章说"每轮把整个 messages 数组发给模型"，但 messages 里的第一条——system prompt——在进入 queryLoop 之前已经被精心加工过。它不是一个字符串，而是一个字符串数组，每个元素代表一个"节"（section）。这个设计背后隐藏着一套精心维护的 prompt cache 策略。

概念 · Prompt Cache

Anthropic API 支持 prompt caching：把 system prompt 里不会变化的前缀打上 cache_control: {type: "ephemeral"} 标记，服务端首次处理后缓存这段内容的 KV（5 分钟 TTL），后续命中的请求跳过这部分的 prefill 计算，直接从缓存恢复——速度提升 60–85%，命中的 token 费用折扣到原价约 10%。代价：缓存只对"完全相同的前缀"有效，任何一个字符改变都会使缓存失效。所以 Claude Code 把"绝对不变"和"每轮可能变"的内容严格分到两个不同的 cache 块里。

组装流程分五步，可以分为两大阶段："进 loop 之前的 prompt 构建"（步骤 1–3）和"进 loop 之后每轮的实时附加"（步骤 4）：

流程图 Prompt 组装流水线
步骤 1–3 进入 loop 前完成；步骤 4–5 每轮执行。DYNAMIC_BOUNDARY 标记把静态节（global cache）和动态节（不缓存）切分成两个 system 块，让静态指令跨用户共享缓存。工具定义走独立的 tools[] 参数，不混入 system prompt，避免工具变更破坏 cache。

步骤 1：静态前缀与 prompt cache

getSystemPrompt() 按固定顺序拼接 7 个节：intro → system → doing_tasks → actions → using_tools → tone_and_style → output_efficiency。这 7 个节的内容几乎不随用户输入变化——它们是让 Claude 知道"自己是谁、该怎么做事"的基础指令，是 prompt cache 能长期命中的"压舱石"。

这批节在 API 请求里以 cacheScope: 'global' 发送，意味着跨组织、跨用户共享同一份缓存——同一份静态指令，全球所有 Claude Code 用户共享同一个服务端缓存 KV。

步骤 2：动态节注册表

静态前缀之后，插入一个特殊标记：

// constants/prompts.ts:114–115
// WARNING: Do not remove or reorder this marker without updating cache logic in:
//   - src/utils/api.ts (splitSysPromptPrefix)
//   - src/services/api/claude.ts (buildSystemPromptBlocks)
export const SYSTEM_PROMPT_DYNAMIC_BOUNDARY = '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'

注释里的警告不是客套话——splitSysPromptPrefix 在发送 API 请求前扫描这个字符串，把 prompt 数组劈成两半：boundary 之前的走 cacheScope: 'global'，之后的走 cacheScope: null（不缓存）。如果有人不小心删掉或移动了这个标记，全局缓存会静默失效，每次请求多花十几倍的 token 计算费用。

boundary 之后是动态节注册表。每个动态节通过 systemPromptSection(name, factory) 声明，首次求值后缓存到进程级 Map，直到 /clear 或 /compact 才清除——这是第二层缓存，防止同一 session 内每轮重新求值：

// constants/systemPromptSections.ts:20–30
export function systemPromptSection(name: string, compute: ComputeFn): SystemPromptSection {
  return { name, compute, cacheBreak: false }  // cacheBreak=false → 进程级缓存
}

export function DANGEROUS_uncachedSystemPromptSection(
  name: string, compute: ComputeFn, _reason: string
): SystemPromptSection {
  return { name, compute, cacheBreak: true }  // cacheBreak=true → 每轮重算
}

// 求值时检查缓存
if (!s.cacheBreak && cache.has(s.name)) {
  return cache.get(s.name)  // 直接返回缓存值
}
const value = await s.compute()
setSystemPromptSectionCacheEntry(s.name, value)  // 写入缓存

动态节包括：memory（MEMORY.md 截断后注入，最多 200 行 / 25,000 字节）、env_info（工作目录/平台）、language（响应语言）、token_budget（Token 预算提示）等。

唯一例外是 MCP 指令——用 DANGEROUS_uncachedSystemPromptSection 豁免缓存：

// Note: MCP instructions are NOT cached because connected MCP clients

  'mcp_instructions',

MCP 服务器可以随时连接或断开，如果缓存了"某个 MCP 工具可用"的信息，而实际上它已断开，模型会尝试调用一个不存在的工具。所以 MCP 节必须每轮重算，代价是每次都会破坏 prompt cache——这是两害相权取其轻的工程决策，并且函数名里的 DANGEROUS_ 前缀就是给未来维护者的警告：你要加一个新的 uncached 节，先想清楚为什么。

步骤 3：QueryEngine 最终合并

  ...(customPrompt !== undefined ? [customPrompt] : defaultSystemPrompt),
  ...(memoryMechanicsPrompt ? [memoryMechanicsPrompt] : []),  // SDK 自定义记忆机制

customPrompt 是 API/SDK 调用方的完全自定义 prompt（完全替代默认 system prompt）；appendSystemPrompt 是追加内容（不替代，叠加）。企业部署时，管理员下发的合规策略就走 appendSystemPrompt 注入。

步骤 4：尾部实时上下文（每轮执行）

appendSystemContext() 在 queryLoop 入口处调用，把实时变化的数据追加到 system prompt 末尾：当前 git status、当前日期时间等。这些内容每轮可能不同，不能进 prompt cache，所以放到 boundary 之后的末尾，让前面已经缓存的静态块不受影响。

步骤 5：工具定义——不在 prompt 里

工具以 API-native tools[] 数组形式传入 callModel，完全独立于 system prompt。这不是偶然的设计，而是为了保护 prompt cache：如果工具定义混进 system prompt，每次新增一个工具、修改一行工具描述，都会使整个 system prompt 的 cache 失效——而 system prompt 通常有几千到一万个 token，失效一次等于白白多花一次 prefill 费用。

把工具定义单独走 tools 参数，工具列表的变化不影响 system prompt cache，两者互不干扰。

设计启示

给开发者Prompt cache 的命中率，本质上是"你有多少内容在所有请求间保持完全不变"。Claude Code 为了最大化命中率，做了三层隔离：① 静态内容 vs 动态内容分到不同 cache 块；② 动态内容按"进程级缓存 vs 每轮重算"分成普通节和 DANGEROUS 节；③ 工具定义完全移出 system prompt。这三层设计让即使是对话进行到第 50 轮，静态前缀依然能命中跨组织的 global cache。自己构建 Agent 时，第一步要问：哪些 prompt 内容是真正不变的？把它们物理分离到独立的 cache 块。

2.3 Token 预算与 max_tokens 控制

每次调用模型时必须传 max_tokens——这不是可选项，而是 Anthropic API 的强制参数。传多少？答案分三层：默认值、升级值、续写值。

默认值：8k cap 的工程逻辑

Claude Code 的 max_tokens 默认值并不等于模型上限。当 tengu_otk_slot_v1 feature flag 开启时，默认值被钳制在 CAPPED_DEFAULT_MAX_TOKENS = 8_000：

// utils/context.ts
export const CAPPED_DEFAULT_MAX_TOKENS = 8_000   // feature flag 开启时的默认
export const ESCALATED_MAX_TOKENS     = 64_000   // 升级后的上限

为什么只给 8k？AI 服务后端为每个请求预留等于 max_tokens 的内存 slot。实际统计中 p99 的 Claude Code 回答不超过 4,911 tokens，却用 32k/64k 的默认值预留了 8–16 倍资源，白白挤占并发容量。8k 覆盖了超过 99% 的实际场景，对那不到 1% 的长输出请求，通过下面的升级机制透明处理。

升级机制（escalate）

当模型以 max_output_tokens 错误返回时，Harness 自动把 max_tokens 升到 64k 并重试同一请求——对用户完全透明：

// query.ts:1196–1221 — 一次性升级，不是递归
const capEnabled = getFeatureValue_CACHED_MAY_BE_STALE('tengu_otk_slot_v1', false)
if (capEnabled && maxOutputTokensOverride === undefined) {
  logEvent('tengu_max_tokens_escalate', { escalatedTo: ESCALATED_MAX_TOKENS })
  state = {
    ...state,
    maxOutputTokensOverride: ESCALATED_MAX_TOKENS,
    transition: { reason: 'max_output_tokens_escalate' },
  }
  continue  // 用 64k 重跑同一轮
}

续写恢复（recovery）

升级到 64k 后仍被截断？进入续写模式，注入一条 meta 消息推动模型继续输出：

// query.ts:1224–1252
// 注入到 messages 里，以 user 角色发给模型
"Output token limit hit. Resume directly — no apology, no recap,
 no commentary, no markdown code fences — just continue writing
 the code / content from where you left off:"

续写最多尝试 MAX_OUTPUT_TOKENS_RECOVERY_LIMIT = 3 次（maxOutputTokensRecoveryCount 字段计数），之后放弃。3 次上限与 thinking block 有关：thinking block 必须随其所在的 assistant trajectory 完整保留，无限续写会导致 thinking 链断裂。

Token 预算（TOKEN_BUDGET feature）

query/tokenBudget.ts 实现了一个"推动模型把 token 用够"的机制——与 max_tokens 方向相反，它解决的是模型"能写 5000 字但只写了 500 就收工"的问题：

// query/tokenBudget.ts — 预算检查核心
export function checkTokenBudget(
  tracker: BudgetTracker,
  budget: number | null,
  globalTurnTokens: number,
): TokenBudgetDecision {
  if (budget === null || budget <= 0) return { action: 'stop', completionEvent: null }

  const pct = Math.round((globalTurnTokens / budget) * 100)
  // 连续 3 次增量 < 500 token → 认为模型已进入 diminishing returns
  const isDiminishing =
    tracker.continuationCount >= 3 &&
    delta < DIMINISHING_THRESHOLD   // 500 tokens

  if (!isDiminishing && globalTurnTokens < budget * COMPLETION_THRESHOLD) { // 0.9
    return { action: 'continue' }  // 注入 nudge 消息，驱动模型继续输出
  }
  return { action: 'stop', completionEvent: pct >= 95 ? 'budget_reached' : null }
}

2.4 记忆预取时机

Claude Code 的记忆系统有两条完全不同的路径，分别解决两个不同问题：

路径数据源载体时机特点
路径 AMEMORY.md 索引system prompt 动态节进入 loop 之前同步、固定、覆盖所有 turns
路径 B记忆目录 ~/.claude/memory/tool_result 附件注入loop 启动时异步预取异步、尽力而为、按相关性筛选

路径 A：MEMORY.md 静态索引

MEMORY.md 是一个"指针文件"——每行一个条目，是各记忆文件的摘要指针，不包含内容本身。通过 loadMemoryPrompt() 同步读取，截断到 200 行 / 25,000 字节，拼入 system prompt 动态节。这在 queryLoop 启动之前就已完成，对所有 turns 可见。

路径 B：relevant_memories 动态预取

queryLoop 进入 while(true) 之前，先启动一次异步预取：

预取流程分两步：

概念 · Explicit Resource Management（using 关键字）

TypeScript 5.2 / ECMAScript Stage 3 提案引入的资源管理语法。实现了 Symbol.dispose 方法的对象，用 using 声明后，当前作用域（函数、块）退出时会自动调用 dispose()——包括正常返回、抛出异常、generator 的 .return() 关闭。相当于 C# 的 using、Python 的 with。Claude Code 用它来保证记忆预取的 AbortController 无论如何都能被清理，无需手写 try/finally。

消费逻辑里还有一个顺序依赖值得单独提：filterDuplicateMemoryAttachments 先过滤、再把新文件写入 readFileState。如果顺序反过来——先写入再过滤——那么预取产生的记忆会被自己的 filter 全部拦截掉（自引用死结）。

设计启示

给开发者记忆预取是"尽力而为"而非"阻塞等待"的典型案例。用 settledAt 轮询替代 await，保证了记忆加载慢或失败时，主循环不卡、turn 不延迟。这种"我有机会就消费，没机会就下轮再试"的非阻塞附件注入模式，在任何需要"顺手追加额外上下文"的 Agent 场景下都可以直接复用。

还要注意session 级有 MAX_SESSION_BYTES = 60KB 的累计注入上限——超过后停止预取，compact 后重置计数，自然允许重新注入。这防止了记忆文件无限增长导致的 token 膨胀。

2.5 上下文压缩触发

消息数组只增不减，总有一天会把上下文窗口撑爆。Claude Code 有两套压缩手段，分别处理"快满了"和"已经满了"：

Proactive compact（主动压缩）

每个 iteration 进入 API 调用之前，检查当前消息量是否超过阈值：

// services/compact/autoCompact.ts:72–91
export function getAutoCompactThreshold(model: string): number {
  const effectiveContextWindow = getEffectiveContextWindowSize(model)
  // 预留 13,000 tokens 的缓冲，留给压缩本身产生的输出
  return effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS   // 13_000
}
// 触发条件：tokenCount(messages) - snipTokensFreed >= autoCompactThreshold
// snipTokensFreed：history snip 已释放的 tokens，防止刚压缩就误判

阈值 = 上下文窗口 - 13k 缓冲。触发后，调用一次 Sonnet 生成对话摘要，替换旧消息，同轮继续（不 continue 循环，直接用压缩后的 messages 调 API）。有 circuit breaker：连续失败 3 次后停止重试：

// autoCompact.ts:257–265
if (tracking?.consecutiveFailures >= MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES) {  // 3
  return { wasCompacted: false }  // 停止，防止无限调压缩 API
}

Reactive compact（被动压缩）

Proactive compact 没来得及触发，模型调用直接返回了 prompt_too_long 错误。这条路径更复杂，因为存在一个协议冲突：SDK 调用方约定收到 error 消息就终止 session，但 Harness 还想在 error 之后继续运行 reactive compact。解决方案是"暂扣（withheld）"：

// query.ts:811–813 — 在流式循环中暂扣 PTL 错误消息
if (reactiveCompact?.isWithheldPromptTooLong(message)) {
  withheld = true   // 先不 yield 给 SDK，内部记录
  assistantMessages.push(message)
}

// 流式结束后，按优先级尝试恢复（query.ts:1086–1183）：
// 优先级 1：context-collapse drain（便宜，重应用已计算的折叠）
if (hasPendingCollapse && !isCollapseRetry) {
  state = { ...state, transition: { reason: 'collapse_drain_retry' } }
  continue
}
// 优先级 2：reactive compact（贵，需要调一次 API 生成摘要）
if (!hasAttemptedReactiveCompact) {
  const compacted = await reactiveCompact.tryReactiveCompact(...)
  if (compacted) {
    state = { ...state, hasAttemptedReactiveCompact: true,
              transition: { reason: 'reactive_compact_retry' } }
    continue   // 用压缩后 messages 重试整个 iteration
  }
}
// 两条路都走完还是 PTL：surfacing 错误，return { reason: 'prompt_too_long' }
yield lastMessage  // 现在才把暂扣的错误消息放出去
return { reason: 'prompt_too_long' }

hasAttemptedReactiveCompact 是防死循环守卫：只允许 reactive compact 尝试一次。没有这个守卫，"reactive compact → 压缩后消息还是太长 → 再次 PTL → 再次 reactive compact → ..."会无限循环烧光 API 调用额度。

决策树 上下文压缩触发路径
两级防线：proactive 在调用前主动压缩，reactive 在错误后被动恢复。drain 优于 reactive compact，因为 drain 只需重应用计算结果，不需要新的 API 调用。
设计启示

给开发者"暂扣"模式（withheld）是处理"内部恢复 vs 外部协议"冲突的通用手法。当你的系统有内部恢复逻辑，但下游消费方约定"收到 error 就终止"时，先把 error 扣在手里，等内部路径跑完，再决定是把成功结果发出去还是释放 error。这避免了修改下游协议或双写两套事件序列。

给产品经理autocompact 对用户完全透明——他们看到的是"Agent 继续工作"，而不是"上下文超限报错"。这种"引擎层自动兜底"的设计，是 Claude Code 能处理长时间、大项目任务的基础能力之一。

第三章

工具系统——Agent 的手

实现目标：Agent 能调 Bash、读写文件等多种工具，且读操作并发、写操作串行，互不干扰。

模型在同一轮里发出三个工具调用——Read("src/query.ts")、Grep("isConcurrencySafe")、Edit("src/Tool.ts", ...)。全部串行执行，慢；全部并发执行，两个读操作没问题，但 Edit 与任何同时进行的写操作相撞，结果就是损坏的文件。这就是工具系统的核心矛盾：最大化并发（速度）× 保证写操作互斥（正确）。

Claude Code 的解法不是一把大锁，也不是让工具各自管理锁，而是把决策拆成两半：工具只声明属性，调度器只执行策略。每个工具通过 isConcurrencySafe 谓词在定义阶段声明"我能不能与别人并发"，调度器据此在运行时把一组调用分批（partition）——连续的只读工具合并成并发批次，每个写工具独占一个串行批次。两者互不知晓对方的存在，却共同构成正确的执行计划。

3.1 Tool 接口——一个契约，四个受众

在 Claude Code 里，一个"工具"不是一个函数，而是一个实现了 Tool<Input, Output, P> 接口的对象（Tool.ts:281）。这个接口有 40+ 个成员，乍看臃肿——但它并不是设计失控，而是因为一个工具要同时对四个完全不同的消费者作出承诺：

  • 对模型：inputSchema、description()、prompt()——"我是谁、怎么调用我"；
  • 对调度器：isConcurrencySafe()、isReadOnly()——"我能不能与别人并发"；
  • 对权限系统：validateInput()、checkPermissions()、isDestructive()——"执行我之前要过哪些关"；
  • 对 UI：renderToolUseMessage() 等——"我在终端里长什么样"。

四个受众各取所需，互不越界：调度器从不读 schema，模型从不见 render 函数。按这个视角看接口分域：

接口解剖 Tool 接口的六个职责域 — 悬停某域查看代表字段
Tool 接口的 40+ 个成员可归入六域：身份、模式约定、执行、调度属性、权限、渲染。其中"调度属性"和"权限"是 Claude Code 架构最独特的两处设计。

抓住本章要用到的核心成员：

// Tool.ts:281 — 精简版，只保留本章必须理解的字段
type Tool<Input, Output, P> = {
  name: string                          // 唯一标识，模型靠这个名字发起调用

  inputSchema: Input                    // Zod schema，双重用途：
                                        //   1. 校验模型传来的参数
                                        //   2. 生成发给模型的 JSON Schema 描述

  call(                                 // 工具主逻辑
    args: z.infer<Input>,
    context: ToolUseContext,
    canUseTool: CanUseToolFn,
    parentMessage: AssistantMessage,
    onProgress?: ToolCallProgress<P>
  ): Promise<ToolResult<Output>>

  isConcurrencySafe(input: z.infer<Input>): boolean  // 调度关键：是否可与其他工具并发
  isReadOnly(input: z.infer<Input>): boolean          // 是否只读（影响权限预检）
  isDestructive?(input: z.infer<Input>): boolean      // 是否不可逆（删除 / 覆盖）

  validateInput?(input, ctx): Promise<ValidationResult>   // 前置校验（语义层）
  checkPermissions(input, ctx): Promise<PermissionResult>  // 权限检查

  maxResultSizeChars: number            // 结果超过此值则落盘，模型收到路径 + 预览
  mapToolResultToToolResultBlockParam(  // 把 Output 序列化成 API 要求的格式
    content: Output, toolUseID: string
  ): ToolResultBlockParam
}

两个成员值得多说一句：

inputSchema 是校验器，更是说明书。 Zod schema 会被自动序列化成 JSON Schema 发给模型，作为"这个工具有哪些参数、怎么用"的文档。这意味着 schema 的字段名和描述文案直接决定模型会不会正确调用——写 schema 就是在给模型写 API 文档。这也解释了为什么 Claude Code 的工具描述都长得像教程而不像类型定义。

maxResultSizeChars 是 token 的闸门。 一个 Bash("find . -name '*.ts'") 可能返回几十 KB。结果超限时内容落盘，消息里只留路径 + 预览（toolResultStorage.ts）。有一个例外暴露了设计者的思考深度：FileReadTool 把它设为 Infinity（FileReadTool.ts:342）——FileRead 的输出本身就是文件内容，再落盘等于制造循环（存下的路径被模型再 Read，又落盘……）。

3.1.1 权限流水线：validateInput → checkPermissions → call

每次工具调用在进入 call() 之前，必须过两道关（toolExecution.ts:683）：

// toolExecution.ts（简化）
async function checkPermissionsAndCallTool(tool, input, ctx, ...) {
  // ── 第一道：语义校验（input 层）
  const isValid = await tool.validateInput?.(parsedInput, ctx)
  if (isValid?.result === false) {
    return [errorResult(isValid.message)]   // 直接拒绝，不走权限
  }

  // ── 第二道：权限决策（canUseTool / hooks / 用户确认）
  const permissionDecision = await canUseTool(tool, parsedInput, ctx)
  if (permissionDecision.behavior !== 'allow') {
    return [deniedResult()]
  }

  // ── 第三步：执行
  const result = await tool.call(parsedInput, ctx, canUseTool, msg, onProgress)
  return [resultMessage(result)]
}

validateInput 负责语义层面的前置检查——"路径必须是绝对路径"这类业务规则，与权限无关，失败直接返回错误、不弹权限框。checkPermissions 则是工具级的自定义权限逻辑（全局权限框架在第七章），例如 MCP 工具在这里统一返回 passthrough，让上层权限系统接管。

还有一条贯穿整条流水线的铁律：错误不逃逸。参数校验失败、权限被拒、call() 执行中抛异常——所有失败路径都不会向上抛，而是被包装成 is_error: true 的 tool_result 送回消息数组。对循环来说，错误只是另一种观察结果：模型看到"Edit 失败：old_string 不唯一"，下一轮自己修正参数重试。工具系统靠这条规则获得了自愈能力——任何单次工具失败都不足以杀死 Agent Loop。

设计启示

给开发者把"参数对不对"（validateInput）和"能不能做"（checkPermissions）拆成两关：前者是无副作用的纯校验，随时可调；后者涉及用户交互，调用有代价。混在一起就会出现"弹了权限框才发现参数本来就错了"的体验问题。而"错误不逃逸"是所有 Agent 系统的必选项——异常必须转换成模型可读的数据，否则模型永远没有机会自我修正。

3.2 buildTool——消灭样板的工厂

你不会直接手写一个实现了 40+ 成员的 Tool 对象。Claude Code 提供了 buildTool(def) 工厂函数（Tool.ts:783），它接受一个"可省略部分成员"的 ToolDef，自动填入合理的默认值，返回一个完整的 Tool：

// Tool.ts:757 — 默认值定义
const TOOL_DEFAULTS = {
  isEnabled:          () => true,
  isConcurrencySafe:  () => false,   // ← 默认不安全（fail-closed）
  isReadOnly:         () => false,   // ← 默认写操作（fail-closed）
  isDestructive:      () => false,
  checkPermissions:   () => Promise.resolve({ behavior: 'allow', updatedInput }),
  toAutoClassifierInput: () => '',   // ← 默认不参与安全分类器
  userFacingName:     () => name,
}

// Tool.ts:783
export function buildTool<D extends AnyToolDef>(def: D): BuiltTool<D> {
  return {
    ...TOOL_DEFAULTS,
    userFacingName: () => def.name,
    ...def,                          // def 的字段覆盖默认值
  } as BuiltTool<D>
}

注意默认值的方向：isConcurrencySafe 和 isReadOnly 都默认 false——fail-closed（失效即关闭）。一个工具如果没有显式声明"我是安全的"，就被当作写操作、串行执行。这是防御式设计：宁可损失一些并发速度，也不要因为漏标导致数据损坏。

对比三个典型工具的声明：

// FileReadTool.ts:373
isConcurrencySafe() { return true  },  // 只读，可与任意工具并发
isReadOnly()        { return true  },

// FileEditTool.ts — 未声明，走默认值
// isConcurrencySafe → false（不可并发）
// isReadOnly        → false（写操作）

// BashTool.tsx — 动态判断
isConcurrencySafe(input) {
  return this.isReadOnly?.(input) ?? false
},
isReadOnly(input) {
  const hasCd = commandHasAnyCd(input.command)
  const result = checkReadOnlyConstraints(input, hasCd)
  return result.behavior === 'allow'   // 静态分析命令内容
}

BashTool 的处理最有意思：它不是静态地声明"总是可以"或"总是不行"，而是在运行时解析命令内容来动态决定。ls、cat、grep 这类命令静态分析为只读，就能享受并发；touch、rm、任何包含 cd 的命令则被保守地判定为写操作，必须串行。

概念 · Fail-Closed（失效即关闭）

Fail-Closed 是一种安全设计原则：当系统无法判断某个操作是否安全时，默认拒绝（而不是默认允许）。与之对立的是 Fail-Open，即"不确定就放行"。

在工具调度里，选择 fail-closed 意味着：一个新工具如果忘记声明 isConcurrencySafe，不会意外地与其他工具并发、造成状态损坏；代价是可能错失一些并发机会。这个代价远小于静默的数据破坏，所以是正确的权衡。

3.3 读并发与写独占——调度器的核心

现在来看本章最关键的工程核心：调度器是如何把一组工具调用编排成"读并发、写串行"的执行计划的。

整个调度逻辑在 toolOrchestration.ts 里，由三个函数组成：runTools（主入口）、partitionToolCalls（分批器）、runToolsConcurrently 和 runToolsSerially（执行器）。

3.3.1 partitionToolCalls：分批

分批算法是一个贪心分组（toolOrchestration.ts:84）：从左往右扫描工具调用列表，把相邻的"并发安全"工具合并进同一个批次，只要遇到一个"不安全"的工具就新开一个批次：

// toolOrchestration.ts:84
function partitionToolCalls(
  toolUseMessages: ToolUseBlock[],
  toolUseContext: ToolUseContext,
): Batch[] {
  return toolUseMessages.reduce((acc: Batch[], toolUse) => {
    const tool = findToolByName(toolUseContext.options.tools, toolUse.name)
    const parsedInput = tool?.inputSchema.safeParse(toolUse.input)
    const isConcurrencySafe = parsedInput?.success
      ? (() => {
          try { return Boolean(tool?.isConcurrencySafe(parsedInput.data)) }
          catch { return false }   // ← 解析失败也 fail-closed
        })()
      : false

    // 连续的"安全"工具合并进同一批次
    if (isConcurrencySafe && acc[acc.length - 1]?.isConcurrencySafe) {
      acc[acc.length - 1]!.blocks.push(toolUse)
    } else {
      acc.push({ isConcurrencySafe, blocks: [toolUse] })
    }
    return acc
  }, [])
}

这个算法最重要的特性：它保持顺序，不做全局重排。如果模型发出 [Read, Read, Edit, Read]，分批结果是 [{Read, Read}, {Edit}, {Read}]，而不是"优化"成 [{Read, Read, Read}, {Edit}]。一个更聪明的调度器完全可以做依赖分析、构建 DAG、全局重排——但 Claude Code 拒绝了这条路，因为它建立在一个错误的前提上：调度器比模型更懂执行顺序。事实相反，模型才是规划者。Edit 之前的 Read 是在读取将要修改的内容，Edit 之后的 Read 是在验证修改结果——这个顺序本身就是模型规划的一部分，调度器读不懂它，也不应该猜。保序贪心是"调度器只做自己确定的优化"这一克制原则的直接产物：相邻只读操作交换顺序一定无害（这是 isReadOnly 从语义上保证的），除此之外一律不动。

分批示意 partitionToolCalls 贪心分组 — 顺序保持，不做全局重排
partitionToolCalls 是保序的贪心分组：同向相邻的"并发安全"工具合并成一批，遇到写操作立刻新开批次。批次间严格串行，批次内并发。

3.3.2 runTools：批次执行

runTools 是调度器的主入口，它遍历分批结果，对每个批次选择对应的执行策略（toolOrchestration.ts:23）：

// toolOrchestration.ts:23
export async function* runTools(...): AsyncGenerator<MessageUpdate> {
  let currentContext = toolUseContext

  for (const { isConcurrencySafe, blocks } of partitionToolCalls(...)) {
    if (isConcurrencySafe) {
      // ── 并发批次：所有工具同时启动，结果按到达顺序 yield
      const queuedContextModifiers: Record<string, ...> = {}
      for await (const update of runToolsConcurrently(blocks, ...)) {
        if (update.contextModifier) {
          queuedContextModifiers[update.toolUseID].push(update.modifyContext)
        }
        yield { message: update.message, newContext: currentContext }
      }
      // 并发批次结束后，统一应用 contextModifier
      for (const block of blocks) {
        for (const modifier of queuedContextModifiers[block.id] ?? []) {
          currentContext = modifier(currentContext)
        }
      }
    } else {
      // ── 串行批次：逐个执行，实时更新 context
      for await (const update of runToolsSerially(blocks, ...)) {
        if (update.newContext) currentContext = update.newContext
        yield { message: update.message, newContext: currentContext }
      }
    }
  }
}

这里有一个微妙的细节：contextModifier。有些工具执行完之后，需要修改运行时上下文（ToolUseContext）——例如 FileEditTool 写完文件后，需要更新内部的文件状态缓存。对串行工具，这很简单：执行完立刻应用修改，下一个工具拿到的 context 已经是更新过的。

对并发工具，问题来了：多个工具同时在跑，如果每个工具都实时修改 context，就出现了经典的竞争条件（race condition）。Claude Code 的解法是延迟应用：并发批次内的所有 contextModifier 先收集起来，等整批工具都完成后再依次应用。这是并发工具"只读安全"这个假设在代码层面的体现——并发工具不得修改共享状态，修改动作统一推迟到批次结束后串行执行。

3.3.3 并发引擎：all() 函数

并发执行的核心是 utils/generators.ts 里的 all() 函数，它把多个异步 Generator 并发跑起来，限制同时活跃的数量，按到达顺序 yield 结果（generators.ts:32）：

// generators.ts:32
export async function* all<A>(
  generators: AsyncGenerator<A, void>[],
  concurrencyCap = Infinity,
): AsyncGenerator<A, void> {
  const waiting = [...generators]
  const promises = new Set<Promise<...>>()

  // 启动初始批次（受 cap 限制）
  while (promises.size < concurrencyCap && waiting.length > 0) {
    promises.add(next(waiting.shift()!))
  }

  while (promises.size > 0) {
    // 谁先返回就先 yield，不等慢的那个
    const { done, value, generator } = await Promise.race(promises)
    if (!done) {
      promises.add(next(generator))
      if (value !== undefined) yield value
    } else if (waiting.length > 0) {
      // 一个 generator 跑完，从等待队列里补充一个
      promises.add(next(waiting.shift()!))
    }
  }
}

并发上限由环境变量 CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY 控制，默认值 10（toolOrchestration.ts:11）。这个值不是随意选的：文件读操作是 I/O 密集型，10 并发能让磁盘 I/O 充分饱和，同时避免文件描述符耗尽。

时序图 · 动态 批次1并发 + 批次2串行 的时间线对比 — 小球表示正在执行的工具
并发批次的壁钟时间等于批次内最慢工具的耗时，而非各工具耗时之和。写操作串行不增加总时间，只是排队等待上一批并发完成。
设计启示

给开发者这套调度的精髓是"声明式并发"：工具不参与调度决策，只声明属性；调度器不理解工具语义，只执行策略。工具作者只要诚实声明，就自动获得正确的并发行为——比在每个工具内部手写锁安全得多。值得注意的是，下一章的主角 Agent 工具也声明了 isConcurrencySafe() → true：正因如此，模型一轮发出的多个 SubAgent 才能落进同一个并发批次、真正并行派生。

给产品经理这套机制是"快而不乱"的技术基础：Agent 能同时读多个文件、搜多个模式，写操作却始终有序。Claude Code 处理大型代码库快，读操作真并发是主要原因之一。

3.4 MCP 工具统一接入

前三节讨论的都是内置工具。Claude Code 还支持通过 MCP（Model Context Protocol）接入的第三方工具——数据库查询、Figma、Slack 等——它们的代码运行在外部进程里。工具系统面临的问题是：外部工具如何享受与内置工具完全一致的调度、权限和结果处理？

概念 · MCP（Model Context Protocol）

MCP 是 Anthropic 提出的开放协议，定义 LLM 应用与外部工具/数据源之间的标准接口：Server 是独立进程，暴露工具定义与调用端点；Client 嵌在 LLM 应用里，负责发现工具、转发调用。核心价值是标准化——任何人实现一个 MCP Server，所有支持 MCP 的 AI 应用立刻获得该能力。

MCP 工具在 Claude Code 内部的接入点是 services/mcp/client.ts。当一个 MCP Server 连接后，Claude Code 遍历它暴露的所有工具定义，把每个工具"改造"成一个符合 Tool 接口的本地对象（client.ts:1768）：

// client.ts:1768 — MCP 工具适配层（精简）
for (const tool of mcpTools) {
  const fullyQualifiedName = buildMcpToolName(client.name, tool.name)
  // mcp__server_name__tool_name

  const mcpTool = {
    ...MCPTool,                      // 继承 MCP 工具的基础实现
    isMcp: true,
    name: fullyQualifiedName,

    // ① 从 MCP 协议的 annotations 映射到 isConcurrencySafe / isReadOnly
    isConcurrencySafe() {
      return tool.annotations?.readOnlyHint ?? false
    },
    isReadOnly() {
      return tool.annotations?.readOnlyHint ?? false
    },
    isDestructive() {
      return tool.annotations?.destructiveHint ?? false
    },

    // ② inputSchema 用 MCP Server 定义的 JSON Schema，而非 Zod
    inputJSONSchema: tool.inputSchema as Tool['inputJSONSchema'],

    // ③ 权限默认走"需要用户确认"流程（passthrough）
    async checkPermissions() {
      return { behavior: 'passthrough', ... }
    },

    // ④ 实际调用通过 MCP 客户端转发
    async call(args) {
      const result = await mcpClient.callTool({ name: tool.name, arguments: args })
      return { data: formatMcpResult(result) }
    },
  }
  tools.push(mcpTool)
}

关键在第①步：MCP 协议的 annotations 元数据被自动翻译成本地调度属性。数据库查询工具标注 readOnlyHint: true，就自动进入并发批次；标注 destructiveHint: true 的工具串行执行并在权限层触发额外警告。注意 ?? false 的写法——3.2 节的 fail-closed 原则延伸到了信任边界之外：annotations 是外部 Server 自报的，没声明的一律按最坏情况（写操作）处理。适配完成后，调度器完全看不出一个工具是内置的还是外部的。

架构图 MCP 工具适配：从外部进程到 Tool 接口 — 三层映射
MCP 工具通过适配层变成标准 Tool 对象。调度器看不出内置工具和 MCP 工具的差别，两者遵循同一套并发规则。

3.5 ToolSearch——工具的懒加载

MCP 解决了"接入"，却制造了新问题：一个重型实例可能挂载几十个 Server、数百个工具，把所有 Schema 全部写进 System Prompt 要消耗数万 token——既慢又贵，而且大多数工具当次任务根本用不到。Claude Code 的解法叫 ToolSearch。

概念 · ToolSearch（工具懒加载）

ToolSearch 把"声明存在"和"提供完整用法"拆成两个阶段：System Prompt 里默认只列延迟工具的名字；模型需要时先调 ToolSearch 拉取完整 Schema，再发起真正的调用。与浏览器的代码分割（code splitting）思路相同——先加载首屏，其余按需拉取。

一个工具是否"被延迟加载"由两个条件决定（toolSearch.ts:isDeferredTool）：

// toolSearch.ts
export function isDeferredTool(tool: Tool): boolean {
  if (tool.alwaysLoad === true) return false   // 明确要求首屏加载

  if (tool.isMcp === true) return true          // MCP 工具一律延迟

  if (tool.name === TOOL_SEARCH_TOOL_NAME) return false  // ToolSearch 自己不能被延迟

  return tool.shouldDefer === true              // 内置工具显式标记可延迟
}

  mcp__figma__get_design_context, mcp__slack__send_message, ...
</system-reminder>

             "parameters": {...完整 schema...}}</functions>

代价是多一轮往返，收益是首屏 token 节省约 40%——而且 Schema 一旦进入上下文就可以在整个会话中复用，往返成本只付一次。

设计启示

给开发者ToolSearch 的设计解决了一个普遍问题：随着功能增长，初始上下文会越来越臃肿。解决办法不是限制功能的数量，而是把"声明存在"和"提供完整描述"分成两个阶段。你设计任何大型 Prompt 系统时，都可以参考这个思路——让 LLM 先知道"有哪些能力"，再按需拉取"每个能力的完整用法"。

给产品经理ToolSearch 使得"Claude Code 能接入多少个 MCP 工具"在理论上变成了无上限——因为工具数量的增加对首屏成本的影响从线性变成了常数（每个延迟工具只贡献一行名字）。这也是 MCP 生态能规模化的基础能力之一。

第四章

SubAgent——并行隔离执行

实现目标：主 Agent 能派出 SubAgent 并行干活，SubAgent 看不到父 Agent 的历史，结果干净返回。

让 Agent 在大型代码库里查一个问题，它可能要读几十个文件、产生几百 KB 的中间结果，而最终有用的结论往往只有二十行。麻烦在于，主 Agent 的上下文窗口是整个系统里最稀缺的资源——如果这些搜索过程全发生在它的窗口里，窗口很快被"过程噪音"塞满，触发压缩、丢失早期信息。Claude Code 的解法是派一个"临时工"替它去干脏活，这就是本章的主角 SubAgent。SubAgent 的本质是一个一次性的、可丢弃的上下文窗口——把高体积低价值的"过程"隔离在子窗口里烧掉，只把低体积高价值的"结论"带回父窗口。这笔账，本书称之为上下文经济学。

实现这个"临时工"几乎不需要任何新机器，只需要一个递归转折。第一章建立过 Agent 的最小结构（一句话回顾：一个消息数组 + 一个 while 循环 + 一组工具），而循环本身也可以被包装成一个工具。Claude Code 里的 Agent 工具（源码里叫 AgentTool）就是这么一个东西——它的 call() 方法内部跑的不是一段函数逻辑，而是一个完整的、全新的 query() 循环。父 Agent 调用它，就像调用 Read 或 Bash 一样自然；区别只是这个"工具"的执行过程本身又是一个会思考、会调工具的 Agent。

SubAgent 的全部原理：AgentTool 在 call() 里孵化一个全新的 query() 循环。穿过隔离边界的只有两条通道——prompt 进、最终文本出；父的历史不下去，子的过程噪音不上来。

4.1 派生原理——AgentTool 只是一个普通工具

先看接口。AgentTool 对调度器、对模型而言与 Read、Bash 没有任何区别——它实现同一个 Tool 接口，声明同样的调度属性（AgentTool.tsx）：

4.1.1 AgentDefinition——每种子 Agent 是一份"人格配置"

subagent_type 参数选择一份 AgentDefinition：它决定子 Agent 的系统提示词（人格）、工具白名单、模型、权限模式。定义来自两个来源：内置定义（builtInAgents.ts：general-purpose、Explore、Plan 等）和用户自定义——一个带 frontmatter 的 Markdown 文件（.claude/agents/*.md，由 loadAgentsDir.ts 加载）：

内置的 Explore 是理解这套机制的最佳样本：它的系统提示词开头就是 "You are a file search specialist… READ-ONLY MODE"，工具白名单只有读操作，且被明确要求"尽可能并行发出多个 grep / read"。也就是说，一种 agent 类型 = 一份提示词人格 + 一份工具白名单 + 一组默认参数，定义与执行机制完全解耦——新增一种子 Agent 不需要写一行 TypeScript。

4.1.2 runAgent——七步装配

选定定义后，runAgent（runAgent.ts:248）把一个子循环从零装配出来：

// runAgent.ts — 装配骨架（大幅精简）
export async function* runAgent({ agentDefinition, promptMessages, ... }) {
  // ① 解析模型：调用参数 > 定义的 model > 继承父模型
  const model = getAgentModel(agentDefinition.model, parentModel, modelParam)
  const agentId = createAgentId()                       // ② 独立身份

  // ③ 初始消息：只有任务书，没有父的历史（隔离的起点）
  const initialMessages = [...promptMessages]

  // ④ 工具池：按定义过滤，Agent 工具本身被移除（4.4 拆分约束）
  const tools = resolveAgentTools(agentDefinition, availableTools, isAsync)

  // ⑤ 系统提示词：用定义正文重建，不是父的 System Prompt
  const systemPrompt = await getAgentSystemPrompt(agentDefinition, ...)

  // ⑥ 隔离上下文：新 readFileState、新/共享 AbortController（4.3）
  const ctx = createSubagentContext(toolUseContext, { agentId, ... })

  // ⑦ 跑一个完整的 Agent Loop——与第一章的循环是同一个函数
  for await (const message of query({
    messages: initialMessages, systemPrompt, toolUseContext: ctx,
    maxTurns: agentDefinition.maxTurns,
  })) {
    await recordSidechainTranscript([message], agentId)  // 过程落盘可审计
    yield message
  }
}

第⑦步值得停一秒：子 Agent 调用的 query() 与父 Agent 的主循环是同一个函数。没有"子 Agent 运行时"这种东西——整套系统只有一个 Agent Loop 实现，SubAgent 只是用不同的参数再跑一遍它。递归结构的美感在于此：第一章的循环、本章的子循环、第五章的 Worker，全是同一段代码。

设计启示

给开发者"把 Agent 包装成工具"意味着多 Agent 能力不需要独立的编排框架：派生走工具调用、并行走工具调度（第三章）、结果回传走 tool_result 注入（第一章）。已有的三套机制拼起来就是多 Agent 系统。如果你在设计自己的 Agent 架构，先问一句：我真的需要编排层吗，还是递归复用已有的循环就够了？

4.2 上下文隔离机制

隔离不是"顺便"的实现细节，而是 SubAgent 的价值所在。上一节第③步已经出现了隔离的核心：initialMessages = [...promptMessages]——子 Agent 的消息数组从任务书开始，父 Agent 几十轮的历史一个字节都不带。配套地，createSubagentContext（forkedAgent.ts:345）把运行时状态逐项拆开：该克隆的克隆、该新建的新建、该共享的显式声明共享：

// forkedAgent.ts:345 — 隔离清单（精简）
function createSubagentContext(parent, overrides) {
  return {
    readFileState: cloneFileStateCache(...),   // 文件缓存：克隆，互不污染
    abortController: overrides.shareAbortController
      ? parent.abortController                  // sync：父按 ESC 一起停
      : createChildAbortController(...),        // async：独立生死（4.3）
    setAppState: overrides.shareSetAppState
      ? parent.setAppState                      // sync：可以更新全局状态
      : () => {},                               // async：全局状态只读（no-op）
    // 任务注册必须始终能到达根 store——否则子 Agent 的后台
    // bash 进程永远无法被清理，变成 PPID=1 的僵尸进程
    setAppStateForTasks: parent.setAppStateForTasks ?? parent.setAppState,
  }
}
隔离清单 隔离边界上：什么能穿过，什么被拦下 — 两条绿色通道，四类红色拦截
隔离边界的默认规则：穿过去的只有 prompt 和最终文本。父的历史、权限批准不下去；子的过程噪音、中间推理不上来。

4.2.1 隔离粒度细到"按类型裁剪注入"

隔离不只是"不给"，还包括"给多少"。源码里有两段注释暴露了这套系统运行的真实规模（runAgent.ts:386）：

// Read-only agents (Explore, Plan) don't act on commit/PR/lint rules from
// CLAUDE.md — the main agent has full context and interprets their output.
// Dropping claudeMd here saves ~5-15 Gtok/week across 34M+ Explore spawns.
const shouldOmitClaudeMd = agentDefinition.omitClaudeMd && ...

// Explore/Plan are read-only search agents — the parent-session-start
// gitStatus (up to 40KB, explicitly labeled stale) is dead weight.
// Saves ~1-3 Gtok/week fleet-wide.

Explore 每周被派生 3400 万次。CLAUDE.md 里的提交规范、gitStatus 快照对一个只读搜索 Agent 毫无用处，但每次派生都要为它们付 token。按类型裁剪注入内容，一周省下 5~15 G token——当派生规模到了千万级，上下文注入的每一段都是要算经济账的。这也反过来解释了为什么 AgentDefinition 要有 omitClaudeMd 这种看起来琐碎的字段：隔离机制的设计粒度是跟着真实成本走的。

4.2.2 例外：fork 模式反向而行

有一类任务恰恰需要父的完整上下文——"顺着我们刚才的讨论，把这个修完"。为此 AgentTool 提供 fork 路径（省略 subagent_type 时触发，forkSubagent.ts）：把父的整个 messages[] 复制给子（forkContextMessages），先用 filterIncompleteToolCalls 清掉未完成的工具调用避免 API 报错，并通过 useExactTools 让子沿用父的工具数组、thinking 配置——目标是让子的 API 请求前缀与父逐字节一致，从而命中同一份 prompt cache。

fork 和 fresh 是上下文经济学的两个极端解：fresh 派生付出"重新交代背景"的成本，换来干净的窗口；fork 派生付出"复制整个窗口"的成本，但由于缓存命中，这份复制几乎免费。判据也随之不同——fresh 适合独立子问题，fork 适合"接着当前语境干，但过程噪音别回来"的活。

设计启示

给开发者隔离与缓存是一对天然矛盾：隔离要求上下文不同，缓存要求上下文相同。Claude Code 没有折中，而是把两个极端都做成显式选项（fresh / fork），让调用方按任务性质选择。当你发现两个设计目标相互拉扯时，"提供两个都纯粹的模式"往往优于"提供一个都不纯粹的折中"。

给产品经理"Explore 一周 3400 万次派生"这个数字值得记住：SubAgent 不是高级功能，而是 Claude Code 每个会话都在大量使用的基础操作。子 Agent 的启动延迟、上下文成本的任何优化，都会被这个乘数放大。

4.3 同步与异步派发

同一个 runAgent，一个 isAsync 标志切出两种完全不同的生命周期。差异集中在四件事上：

// runAgent.ts:524 — 差异①：生死绑定
const agentAbortController = isAsync
  ? new AbortController()          // async：独立控制器。用户按 ESC 中断
                                   // 父 Agent 时，后台子 Agent 继续活着
  : toolUseContext.abortController // sync：共享父控制器，一起生一起死

// runAgent.ts:440 — 差异②：权限降级（又是 fail-closed）
// async 无法弹出交互式权限框 → 拿不到批准的操作自动拒绝
if (shouldAvoidPrompts) {
  toolPermissionContext = { ...ctx, shouldAvoidPermissionPrompts: true }
}

差异③是结果的送达方式。sync 派生阻塞父的这一轮：父循环等子循环跑完，把最终文本作为 tool_result 直接接上。async 派生立即返回一个"已启动"的 tool_result，其中的措辞是设计的一部分（AgentTool.tsx:1328）——它明确告诉父模型接下来该怎么表现：

Async agent launched successfully.
agentId: agent-xxx（内部 ID，可用 SendMessage 继续对话）
The agent is working in the background. You will be notified
automatically when it completes.
output_file: .../agent-xxx.output   ← 可以 tail 看进度，但——
Do not duplicate this agent's work.  ← 别抢它的活
（子 Agent 完成后，结果以 task-notification 的形式
  出现在父 Agent 后续某一轮的消息里）

差异④是工具池：async 子 Agent 的工具被 ASYNC_AGENT_ALLOWED_TOOLS 白名单再过滤一遍——不能 AskUserQuestion（没有终端可问）、不能进 Plan 模式（没人批准计划）。凡是需要"人在场"的能力，后台一律没有。

时序对比 sync 阻塞派发 vs async 后台派发 — 注意 ESC 中断的作用范围
sync 共享父的生死与终端；async 独立存活、权限降级、结果以通知形式异步送达。sync 与 async 之间不是硬边界——阻塞中的 sync Agent 可以随时被转入后台。
设计启示

给开发者async 派生的权限处理再次体现 fail-closed：后台 Agent 弹不出权限框，系统的选择不是"降低门槛放行"，而是"拿不到批准就拒绝"。注意这要求任务本身适合后台化——需要频繁授权的任务放后台会寸步难行，这是拆分决策（4.4）要考虑的约束之一。

4.4 SubAgent 拆分约束

机制讲完了，剩下的问题是"怎么用对"。约束分三层：结构上不能做什么、prompt 上应该写什么、什么任务值得拆。

4.4.1 结构约束：递归只有一层

装配工具池时（4.1 第④步），filterToolsForAgent 默认把 Agent 工具从子 Agent 的池里移除（constants/tools.ts:36）——子 Agent 不能再派生子 Agent。三个理由：成本失控（每层派生都是乘数，两层 = 平方级 token）；可观测性（用户在终端只能看到一层进度，孙子 Agent 的行为无处展示）；权限链断裂（父批准的是"派一个子"，孙子的操作离授权语境已经太远）。需要更深的层级时，答案不是嵌套，而是第五章的协调器模式——星形拓扑，所有 Worker 都直连同一个 Coordinator。

4.4.2 Prompt 约束：像给新同事交代任务

子 Agent 看不到父的历史（4.2），这个机制事实直接决定了 prompt 的写法。AgentTool 的工具描述里，官方给模型的指令是这么写的（prompt.ts，意译）：

  • 零上下文假设："像给一个刚走进房间的聪明同事交代任务——他没看过这场对话，不知道你试过什么。"要交代目标、已排除的方向、足够的背景，让对方能自己做判断。
  • 不许外包理解：禁止写"根据你的调研结果修掉这个 bug"这种 prompt——这是把"综合理解"推给了子 Agent。正确的写法要证明你自己已经理解了：给出文件路径、行号、具体改什么。
  • 查找给命令，调查给问题：确定性查找直接给出精确命令；开放性调查只给问题本身——预设的步骤在前提错误时全是死重。
  • 别偷看、别抢跑（async）：不要中途去读子 Agent 的 output_file（会把它的过程噪音拉回自己的窗口，隔离白做了）；在通知到达前不许编造或预测子 Agent 的结论。

4.5 Worktree 文件系统隔离

第三章的"写独占"调度有一个隐含边界：它只管一个 Agent 的一轮之内的工具调用。多个并行 SubAgent 是各自独立的循环，各有各的调度器——两个后台 Agent 同时改同一个文件，没有任何机制拦截。上下文隔离解决了"记忆打架"，文件系统这层需要自己的隔离方案。

概念 · git worktree

git worktree 是 Git 的原生功能：为同一个仓库检出多个工作目录，它们共享同一个对象库（.git），只是各自有独立的工作区和分支。因此创建成本是毫秒级、磁盘增量只有工作区文件本身——远轻于 clone。这让"每个 Agent 一个独立副本"从奢侈品变成日常操作。

派生时带上 isolation: "worktree"，AgentTool 在装配前多做一步（worktree.ts:902）：

// AgentTool.tsx:590 + worktree.ts:902（精简）
if (effectiveIsolation === 'worktree') {
  const slug = `agent-${agentId.slice(0, 8)}`
  // ① 在主仓库的 .claude/worktrees/ 下建工作树 + 临时分支，
  //    并记下当前 HEAD（之后判断"有没有改动"的基准）
  worktreeInfo = await createAgentWorktree(slug)   // { worktreePath, headCommit, ... }
}
// ② 子 Agent 的整个执行包进 cwd 覆盖——它眼中的"项目根目录"就是工作树
runWithCwdOverride(worktreeInfo.worktreePath, () => runAgent(...))

// ③ 子 Agent 结束后：对比 headCommit 判断去留
const changed = await hasWorktreeChanges(worktreePath, headCommit)
if (!changed) await removeAgentWorktree(...)   // 没改动 → 无痕清理
else keep()   // 有改动 → 保留，worktreePath/Branch 写进 tool_result，父来决定合并
生命周期 Agent Worktree：创建 → 隔离执行 → 按变更决定去留
Worktree 隔离的收尾逻辑与"结论回传"同构：过程（工作树）默认丢弃，只有产生了变更（结论）才保留并回传路径，合并权留给父。

注意收尾策略的分寸感：系统自动创建、自动清理，但从不自动合并。子 Agent 的改动躺在临时分支上，tool_result 只回传路径和分支名——要不要合、怎么合，是父 Agent（最终是用户）的决策。机器负责隔离，人负责合流，这与权限系统"工具声明、用户批准"的边界划分是同一种哲学。

给开发者本章出现了三层互补的隔离，各管一个资源维度：上下文隔离（4.2）管记忆、AbortController 分树（4.3）管生死、worktree（4.5）管文件系统。做多 Agent 系统时值得照着清点一遍：还有哪个共享资源没有隔离方案？（提示：第六章的 Mailbox 管的是通信这一维。）

协调器模式——星形多 Agent

实现目标：一个 Coordinator 同时管多个 Worker，Worker 异步汇报结果，Coordinator 综合后再派新任务。

在钻进源码之前，先把地图铺开。Anthropic 的工程博客 Building Effective Agents 把多 LLM 系统归纳成一条光谱：一端是 Workflow——LLM 和工具沿着预定义的代码路径编排，可预测、易调试；另一端是 Agent——LLM 自主决定自己的流程和工具使用，灵活但要交出控制权。光谱上散布着五种可组合的工作流模式，外加自主 Agent 本身：

这张图上值得停留的观察有三个：

其一，模式不是框架，是任务结构的分类。任务天然串行就是链式，天然可分片就是并行，无法预先拆解才轮到协调器——协调器模式的定义性特征是"子任务由中心 LLM 在运行时动态决定"，这正是它区别于第三章批内并发（调度器按静态属性分批）的地方。选模式的问题从来不是"哪个更先进"，而是"任务长什么形状"。文章的原则也直白：能用单次 LLM 调用 + 检索解决的，就别上工作流；能用工作流的，就别上自主 Agent——从最简单的方案开始，只在可测量的收益面前增加复杂度。

其二，模式可以嵌套。协调器派出的每个 Worker 自己就是一个自主体（⑥）；协调器把"实现"和"验证"分给不同 Worker，就是在星形里嵌了一个生成-评审回路（⑤）；同轮派发三个调研 Worker，用的是并行（③）。第五章不是替代前四章，是把它们组合起来。

其三，Claude Code 全部六种模式共用同一个执行基座。链式是提示词约定（先出计划、批准后执行），路由是几行模型选择逻辑，并行是调度器的自然行为（4.1 的机制图），自主体是 query() 循环本身。于是协调器模式剩下的实现问题只有一个：给循环换一个什么样的人格，才能让它管好别的循环？这就是下一节。

设计启示

给开发者市面上的多 Agent 框架大多把六种模式做成六套 API。Claude Code 的做法是反过来的：把"循环 + 工具 + 调度"这一个基座做扎实，模式退化为提示词和调用方式的组合。如果你发现自己在为每种编排方式写新的执行引擎，值得停下来问：这些引擎的差异，是不是本来可以用一段提示词表达？

5.2.1 四阶段工作流——"综合"不可外包

工作流 四阶段循环：调研 → 综合 → 实现 → 验证 — 只有"综合"必须协调器亲自做
四阶段中三个阶段都可以派工，唯独"综合"必须协调器亲自完成——这是整份系统提示词强调最重的一条纪律，源码原话是"your most important job"。

为什么综合不可外包？提示词里有一段近乎严厉的规定：禁止写 "based on your findings"（基于你的发现）这类句子。对比一下两种派工方式：

// 反模式 —— 偷懒转发，把"理解"外包给了 Worker
Agent({ prompt: "Based on your findings, fix the auth bug" })
Agent({ prompt: "上一个 Worker 发现了 auth 模块的问题，请修复" })

// 正确 —— 协调器先消化调研结果，再写出自包含的 spec
Agent({ prompt: "修复 src/auth/validate.ts:42 的空指针。Session 上的 user 字段
  （src/auth/types.ts:15）在会话过期但 token 仍被缓存时是 undefined。
  在访问 user.id 前加空检查——为 null 时返回 401 'Session expired'。
  跑相关测试和 typecheck，提交并报告 hash。" })

两者的差别不是措辞，是信息在哪里汇聚。星形拓扑里 Worker 互不可见，唯一看得到全局的节点是中心。如果中心只做转发（"把 A 的发现交给 B"），全局视野就没人拥有了——B 拿到的是二手转述，A 的上下文已经烧掉，出了错没有任何节点能兜底。协调器的价值不是"分派"这个动作，而是它是全系统唯一被强制读懂一切的节点。这也解释了为什么提示词要求 spec 里必须出现具体路径和行号：这些细节是"我真的读懂了"的可验证证据，写不出来就说明综合没做。

删掉的两个工具正好定义了拓扑。没有 Agent 工具，Worker 无法再派生——这是 4.4 讲过的防递归约束（一句话回顾：派生深度上限为 2，防的是失控的 Agent 树和不可判定的生命周期）在协调器模式下的延续：结构是星形，不是树形。没有 SendMessage，Worker 之间无法直连——所有信息必须经过中心，保证了协调器的全局视野不被旁路。星形不是画出来的，是从完整工具池里删出来的。

与此对称，协调器一侧通过 getCoordinatorUserContext() 收到一份"Worker 能力说明书"注入上下文：Worker 有哪些工具、连了哪些 MCP Server、以及一个所有 Worker 免许可读写的 scratchpad 目录——提示词称之为"durable cross-worker knowledge"，一块跨 Worker 的持久黑板。Worker 之间不能对话，但可以往黑板上写调研笔记供后继者读，这是星形拓扑里唯一的"旁路"，而它是文件而非消息——异步、可审计、不破坏拓扑。

5.2.3 续跑还是新开——上下文重叠度决策

调研 Worker 完成后，协调器手里有两个选项：用 SendMessage 让同一个 Worker 继续干（它已经加载的上下文还在），或者用 Agent 新开一个。提示词给了一张决策表，判据只有一条：旧上下文对下一个任务是资产还是污染：

情形选择为什么
调研恰好读过要改的那些文件续跑文件已在上下文里，spec 一到就能动手
调研很广、实现很窄新开别拖着探索噪音进实现，干净上下文更聚焦
修正它自己刚犯的错续跑错误现场还在它脑子里，一句话就能定位
验证别的 Worker 写的代码新开验证者要新鲜眼光，不能带着实现者的假设
第一次实现方向整个错了新开错误路径会锚定重试，干净起步避免路径依赖
完全无关的新任务新开没有可复用的上下文

这张表其实是第四章"上下文经济学"的续篇：4.4 说 SubAgent 是一次性的可丢弃窗口，本节补上另一半——窗口也可以不丢，前提是里面装的东西对下一步有用。续跑省掉的是重新加载文件的成本，新开省掉的是被旧假设带偏的风险，两边都是上下文的账。

设计启示

给开发者协调器提示词里最有工程味的规定，都是在防"责任真空"：综合不可外包（全局视野必须有人拥有）、spec 必须带路径行号（理解必须可验证）、派发后立即收口绝不虚构结果（状态必须来自通知而非想象）。设计多 Agent 系统时，比"怎么拆任务"更先回答的问题是：每一份理解，由哪个节点负责持有？

给产品经理"协调器自己几乎不干活"不是低效，是角色净化：它的产出物是任务书和给用户的汇报，这两样恰恰是用户唯一能看到的界面。中心节点的质量决定用户对整个多 Agent 系统的感知质量。

5.3 Worker 结果聚合——task-notification 的一生

Worker 跑完了，结果怎么回到协调器？第四章末尾只说了一句"以 task-notification 注入后续轮次"，现在拆开看全程。每个 Worker 在系统里是一个 LocalAgentTask（tasks/LocalAgentTask/LocalAgentTask.tsx），从完成到被协调器读到，要走四步：

流水线 通知流水线：完成 → 构造 XML → 排队 → 回合边界注入 — 用户输入永远插在通知前面
通知的一生：Worker 终态触发 XML 构造（notified 标志原子去重），进入 later 优先级排队（用户输入是 next，永远插队），在协调器回合结束的边界以 user 角色消息注入。协议的"解析器"不是代码，是系统提示词里教给模型的标签识别规则。

构造通知的源码浓缩后只有一小段，但每个字段都有讲究（LocalAgentTask.tsx:197）：

// enqueueAgentNotification —— 精简
let shouldEnqueue = false
updateTaskState(taskId, setAppState, task => {
  if (task.notified) return task          // 已通知过：原子去重，防止重复消息
  shouldEnqueue = true
  return { ...task, notified: true }
})
if (!shouldEnqueue) return

const message = `<task-notification>
<task-id>${taskId}</task-id>                    // 续跑地址：SendMessage 的 to
<output-file>${outputPath}</output-file>        // 全程输出落盘，可追查细节
<status>${status}</status>                      // completed | failed | killed
<summary>${summary}</summary>${resultSection}${usageSection}${worktreeSection}
</task-notification>`

enqueuePendingNotification({ value: message, mode: 'task-notification' })
// ↑ 优先级 'later'：注释原话——"so user input is never starved by system messages"

三个设计决定值得点名：

投递时机选在回合边界，而不是实时打断。协调器正在生成回复时，哪怕三个 Worker 同时完成，也要等这一轮说完话。这不是技术限制——第二章讲过循环的输入队列机制（一句话回顾：所有外部输入统一进队列，回合之间统一消化），是刻意保持协调器每一轮的输入都是一个完整快照：乱序抵达的结果在队列里排好，模型一次看到当前全部新信息再规划下一步，而不是被每条通知打断成碎片决策。

用户输入的优先级永远高于通知。队列有 now / next / later 三级，用户消息是 next，通知是 later。五个 Worker 排队汇报时用户插了一句话，下一轮模型先看到人说了什么。多 Agent 系统很容易演成"系统自言自语，用户插不上嘴"，这一行优先级排序是对它的制度性防御。

result 之外还有 output-file。通知里的 <result> 只是 Worker 的最终文本（第四章的"结论二十行"），但完整过程始终落盘。协调器觉得结论可疑时，可以派下一个任务去读文件核查——聚合默认是轻的，但审计通道永远敞开。

设计启示

给开发者注意全书反复出现的同构：工具结果靠 tool_result 注入（第一章）、SubAgent 结论靠最终文本回传（第四章）、Worker 汇报靠 task-notification 注入（本章）。Claude Code 里所有的"回传"都是同一件事——把内容放进对方下一轮的输入里。没有回调、没有事件总线、没有共享内存，只有消息进队列。这让每个 Agent 的世界模型保持极简：我看到的就是我的 messages 数组。

5.4 SendMessage 通信协议

协调器手里最灵的工具是 SendMessage({ to, message })。它的输入极简，本事全在路由上——同一个工具，根据 to 指向的对象状态走出完全不同的路径（SendMessageTool.ts:741 的 call 方法）：

决策树 SendMessage 路由：同一个 to，四种命运 — 绿色是本章路径，灰色第六章展开
SendMessage 的路由按接收方状态分派。对本章重要的是左侧两条绿色路径：运行中排队注入、停止后磁盘复活——它们共同实现了"Worker 可以被随时追加指令"的体验。

两条绿色路径的源码骨架：

// SendMessageTool.call() —— 本进程 Worker 分支（精简）
const task = appState.tasks[agentId]
if (task.status === 'running') {
  // ① 运行中：进 pendingMessages，绝不打断
  queuePendingMessage(agentId, input.message, setAppState)
  return { message: `已排队，将在 ${to} 的下一个工具轮送达` }
}
// ② 已停止（completed / failed / killed）甚至已被驱逐出内存：
//    从磁盘 transcript 复活，带着全部旧上下文继续
const result = await resumeAgentBackground({ agentId, prompt: input.message, ... })

路径①的关键词是"工具轮边界"。Worker 正在跑一个 Bash 命令时收到新指令，指令不会打断命令，而是进入该任务的 pendingMessages 数组；Worker 每轮工具执行结束、准备下一次模型调用时，drainPendingMessages 把攒下的消息全部取走，作为附件拼进下一轮输入（attachments.ts:1085）。这和 5.3 协调器收通知的时机选择是同一个哲学的两个方向：消息永远在回合边界交接，任何 Agent 的任何一轮都不被打断。打断意味着半途而废的工具调用和被撕裂的思路，排队意味着每轮开始时世界是完整的。

路径②回收了第四章埋的伏笔。4.3 说过子 Agent 的 transcript 会落盘（当时的用途是"可查"）；现在它多了一个用途——可复活。resumeAgentBackground 从磁盘读回消息数组，做三重清洗（丢弃未解决的 tool_use、孤儿 thinking 块、空白 assistant 消息——都是中断可能留下的残渣），重建工具池和系统提示词，然后像什么都没发生过一样继续对话。对协调器而言，"Worker 停了"不是句号，是逗号：一个 completed 的调研 Worker，三轮之后仍能被一句 SendMessage 唤回来干实现，它读过的三十个文件还在脑子里。

最值得回味的是图底部那行字：接收方永远以为自己收到了普通 user 消息。Worker 的循环里没有"协调器"这个概念，没有任何针对多 Agent 的特殊代码——它就是第一章那个循环，收消息、干活、回话。整个协调器模式是单方面的：只有中心知道自己在协调，成员只是在过普通的 Agent 生活。这种不对称让系统的每个组件都保持最简单的世界模型，复杂度全部收敛在中心一点。

设计启示

给开发者很多多 Agent 框架为"Agent 间通信"设计了 RPC、话题订阅、共享状态。Claude Code 的答案是：通信 = 往对方的下一轮输入里放一条消息，状态 = 对方自己的 messages 数组。协议瘦到只剩投递语义，因为接收方是一个 LLM，它最擅长的就是理解自然语言消息——传输层不需要表达业务语义，业务语义在消息文本里。

5.5.2 停止、复活与生命周期状态机

把 5.3、5.4 讲过的所有转换拼起来，Worker 的一生是一张完整的状态机：

生命周期 Worker 状态机：终态不是终点 — 只要 transcript 在盘上，就能复活
Worker 状态机的核心特征：所有"结束"都是软的。completed 可以续跑、failed 带着错误现场等修正、killed 也回传部分结果并保留 transcript——真正不可逆的只有磁盘上 transcript 被清理那一刻。

终止路径上有两个容易错过的细节，都体现了"信息不浪费"的原则：

被杀的 Worker 也交作业。stopTask（tasks/stopTask.ts）里有一段耐人寻味的区别对待：后台 Bash 被杀时，"exit code 137" 通知被抑制——那是噪音；但 Agent 任务被杀时通知照发，因为它携带 extractPartialResult 从已产生的消息里抢救出的部分结果——那是 payload。你半路叫停一个调研 Worker，它读过的东西不会白读。

批量终止合并汇报。用户按 ESC 时 killAllRunningAgentTasks 停掉所有 Worker，同时用 markAgentsNotified 把每个任务的 notified 标志直接置位——五个 Worker 不会发五条讣告刷屏，系统发一条聚合消息。去重标志在这里被反向利用：先置位，通知就永远不会发出。

失败处理的策略同样围绕上下文经济学展开：Worker 报告测试失败时，提示词的第一选择是 SendMessage 续跑同一个 Worker——它拥有完整的错误现场（改了什么、报了什么错、试过什么），一句"58 行和 72 行的断言还在失败，改成新的报错文案"就够了；只有当续跑再次失败，才换思路新开或上报用户。对照 5.2.3 的决策表，这正是"修正它自己刚犯的错 → 续跑"那一行。

设计启示

给开发者这个状态机最反直觉的设计是"没有真正的终态"。传统任务系统里 killed 就是死了；这里 kill 只是 abort 掉进程内的循环，身份（agentId）和记忆（transcript）都在盘上，SendMessage 一到就原地复活。把"执行"和"身份+记忆"分离之后，停止变得廉价而无损——这是敢于"随手停、随手续"的底气，也间接鼓励协调器大胆派工：派错了就停，成本近乎为零。

给产品经理ESC 一键全停 + 聚合汇报 + 部分结果保留，这三件事共同回答用户对多 Agent 最深的恐惧："失控了怎么办"。刹车必须比油门显眼，这是自治系统赢得信任的前提。

Agent 间通信——Mailbox

实现目标：任意 Agent 可以给另一个 Agent 发消息，接收方可以阻塞等待，也可以非阻塞轮询。

想象这样一个场景：你开了三个终端窗口，每个窗口里跑着一个独立的 Claude Code——一个当领队，负责分工和验收；两个当队友，各自埋头改一个模块。领队想提醒队友 ada："utils 目录 bob 正在改，你先别碰。"这句话怎么送过去？要知道这是三个互不相识的操作系统进程：不共享内存，不知道彼此的端口和 PID，甚至不知道对方此刻还活着没有。这正是 Claude Code 团队模式（Agent Teams / Swarm）每时每刻要面对的通信问题，也是本章要解决的问题。

第五章的协调器从来不用操心这件事，因为那是"一个进程里的多 Agent"：Coordinator 和 Worker 全都活在同一个 Node 进程中，所谓"发消息"（SendMessage 工具），本质是往同一份内存里的 pendingMessages 数组 push 一项。没读过第五章也不要紧，只需记住这个对比：单进程内传话，就是改一块共享内存。可团队模式里每个成员都是独立进程，跑在 tmux（终端分屏工具）的不同窗格里——共享内存这条路，彻底断了。

两个互不相识的进程要通信，工程上的标准答案是 socket、HTTP、消息队列。Claude Code 选了一个"土"得多的方案：每人一只 JSON 文件当信箱，写信加锁追加，收信定时轮询。土，但仔细掂量会发现它精确匹配了这个场景的每一项需求——本章就从这只文件信箱出发，一路讲到跑在它上面的高层协议：权限跨进程转授、计划审批、优雅关机。你会再次看到第五章出现过的规律：传输层越简单，语义越容易做对；复杂度应该住在协议层，而协议大半是提示词和 JSON 约定。

架构总图 文件即消息总线：两个进程，唯一的共享介质是磁盘 — 没有 socket，没有端口，没有 broker
Mailbox 的全部架构：每个成员一只 JSON 信箱文件，写方加锁追加，收方轮询未读。进程之间零直连——一方崩溃、重启、换机器（只要挂同一块盘），通信语义不受任何影响。

6.1 Mailbox 架构——文件即消息总线

先看数据结构，全部机制围绕它展开（utils/teammateMailbox.ts）：

// 信箱地址：~/.claude/teams/{team}/inboxes/{agent_name}.json
export function getInboxPath(agentName: string, teamName?: string): string {
  return join(getTeamsDir(), sanitize(team), 'inboxes', `${sanitize(agentName)}.json`)
}

// 一封信：发件人 + 正文 + 已读标记，没有了
export type TeammateMessage = {
  from: string          // 发件人名字（自愿声明，见 6.4 的信任模型）
  text: string          // 正文：自然语言，或 JSON 编码的协议消息（6.3）
  timestamp: string
  read: boolean         // 收件方消费后置 true——递送语义的关键
  color?: string        // 发件人的 UI 颜色
  summary?: string      // 5-10 词摘要，给 UI 预览用
}

写入是唯一需要小心的操作：一个团队里多个进程可能同时给同一个人写信，两个 read-modify-write 交错就会丢信。解法是经典的锁文件 + 重试退避：

// writeToMailbox —— 精简（teammateMailbox.ts:134）
await writeFile(inboxPath, '[]', { flag: 'wx' })   // 不存在则创建（原子）
const release = await lockfile.lock(inboxPath, {
  retries: { retries: 10, minTimeout: 5, maxTimeout: 100 },  // 排队等锁，不失败
})
try {
  const messages = await readMailbox(recipientName)   // 拿到锁后重读最新状态
  messages.push({ ...message, read: false })
  await writeFile(inboxPath, jsonStringify(messages, null, 2))
} finally {
  await release()
}

三行注释值得展开成三个设计判断：

为什么是文件，不是 socket / HTTP / 消息队列？把场景的约束列出来，答案几乎是推出来的。第一，通信双方是不同后端上的进程——可能同进程（in-process 队友）、可能是 tmux 窗格、iTerm 窗格，未来还可能是远程会话；socket 需要端口协商和服务发现，文件路径本身就是全局约定好的"地址"。第二，Agent 的回合以秒到分钟计，1 秒的轮询延迟在这个时间尺度上是免费的——为省这 1 秒去引入一个 broker，是拿复杂度换一个没人感知的指标。第三，消息天然要求持久与可审计：进程崩了信还在，重启接着读；用户随时可以 cat 一眼信箱排查问题。第四，零基础设施——不需要用户装 Redis，不需要起 daemon。文件系统是唯一保证存在的中间件。

为什么锁要重试退避，而不是失败报错？注释原话："concurrent callers (multiple Claudes in a swarm) wait for the lock instead of failing immediately"。发信方是模型的工具调用，一次失败对模型而言是一次需要理解和重试的异常；把重试下沉到传输层（最多 10 次、5–100ms 退避），模型看到的世界就永远是"发信必成功"。底层多做一点，上层的心智模型就少一个分支。

为什么拿到锁后要重读？锁保护的是 read-modify-write 全程，而不是 write 一下。等锁期间别人可能已经追加了消息，用等锁前读的旧数组覆盖回去就会丢信。这是所有"文件当数据库"场景的第一课，源码用注释把它钉死在那里。

信箱之外还有一份名册：TeamCreate 工具创建 ~/.claude/teams/{team}/config.json，记录全体成员（名字、agentId、颜色、活跃状态），同时创建一个同名任务列表目录——Team 与 TaskList 一一对应，任务的指派（TaskUpdate 的 owner 字段）和消息的投递共享同一套成员名单。广播（to: '*'）就是读名册、逐个写信箱的语法糖。

设计启示

给开发者选传输层时先问时间尺度。Agent 回合是秒级的，消息延迟 1 秒无感知——此时持久性、可审计性、零部署的价值远大于吞吐和延迟。反过来，如果你的 Agent 需要毫秒级互动（比如协作编辑同一个文件的光标），文件轮询就是错的。没有"最好的消息机制"，只有和交互节奏匹配的消息机制。

值得停一秒的是第①层。进程内队友和领队共享同一份内存、同一个 React 状态树，"当前代码在为谁跑"这个问题没有全局答案，只有调用链上下文能回答。AsyncLocalStorage 在这里扮演的角色，和第四章 SubAgent 的隔离 messages 数组同构：并发的身份必须随执行流走，不能存在共享的地方。

6.2.2 收信的三条通道

本章副标题说"接收方可以阻塞等待，也可以非阻塞轮询"，源码里实际有三条通道，按接收方的状态各司其职：

机制图 三条收信通道：谁在什么状态下用哪条 — 已读标记永远在安全递送之后
三条收信通道覆盖接收方的三种状态：闲着（轮询成轮）、常驻等活（阻塞循环）、干活中（轮边界注入）。无论哪条，已读标记都在消息安全落地之后才写回——这是整个递送语义的支点。

递送语义值得单独讲透，因为它是"文件当消息队列"最容易做错的地方。useInboxPoller 的注释把顺序写得很明白：先递送或可靠入队，再标记已读。如果反过来——先标已读再递送——进程在两步之间崩溃，消息就永远消失了；按现在的顺序，崩溃的后果只是下次轮询重读一遍，即经典的 at-least-once（至少一次）语义。重复递送的代价由消费端的幂等去重抵消：比如权限请求入队前按 tool_use_id 查重（"markMessagesAsRead failed on a prior poll — skip if already queued"）。丢失不可恢复、重复可以过滤，两害相权，分布式系统五十年的老结论，在这里用几十行代码复刻了一遍。

另一个容易错过的细节在阻塞循环里：出队前先全量扫描一遍未读消息找 shutdown 请求，找到就插队处理。注释给了原因：点对点消息可能洪泛（队友之间聊得太热闹），如果严格 FIFO，关机指令会被淹没在队列尾部，想停都停不下来。控制类消息优先于数据类消息——这个取舍在 5.3 的通知队列里是"用户输入优先于系统通知"，方向相反，原则相同：队列必须为"改变队列命运的消息"留插队通道。

设计启示

给开发者注意全书出现第三次的同构：工具结果注入（第一章）、协调器通知注入（第五章）、信箱消息注入（本章），全部是"把内容放进对方下一轮输入"。Mailbox 没有发明新的通信范式，它只是把"对方的输入队列"从进程内存搬到了文件系统——于是通信自动获得了跨进程、持久化、可审计三个属性，而模型侧的世界观完全不用变：还是收到一条消息，还是下一轮看到。

6.3 消息分类——数据面与控制面

信箱里流动的消息其实是两个物种。一种给模型看：自然语言，"我把 auth 模块查完了，结论如下"；一种给代码看：JSON 编码的协议消息，"type: permission_request, tool: Bash, input: …"。收信端的第一件事就是把它们分开（useInboxPoller.ts 的分拣循环）：

分拣器 数据面给模型，控制面给代码 — 分错任何一个方向都是事故
同一只信箱跑两个平面：自然语言走数据面进模型上下文，JSON 协议消息走控制面进代码处理器。分拣的依据只有一个——text 字段能否解析出已知的 type。

为什么这个分离值得用一个守卫函数（isStructuredProtocolMessage）严防死守？源码注释给了正向理由："If getTeammateMailboxAttachments consumes them first, they get bundled as raw text in attachments and never reach their intended handlers"——数据面抢跑，控制面的处理器就永远收不到消息，权限请求石沉大海，Worker 卡死在等待里。反过来想还有一层：让模型读到一坨 {"type":"permission_response",...} 不仅无用，而且有害——模型可能把它当成要回应的内容，生成一段针对 JSON 的"回复"，白烧 token 还污染对话。LLM 系统里的每一字节上下文都该是模型能行动的信息；机器对机器的信令，模型一个字都不该看见。

反方向的分错更隐蔽：把该给模型看的消息拦在代码层。分拣表里有一行专门处理这个微妙情况——shutdown_request 被识别为协议消息，但处理方式是 regularMessages.push(m)：双面投递。因为关机请求的最终决定权在模型（6.4 详述），代码只负责识别和优先级，模型必须读到原文才能判断"我手头的活能不能停"。计划审批请求同理：领队侧自动批准之后，请求原文仍然进入领队模型的上下文——"the approval is already sent, but the model has context about what the teammate is doing"。控制面处理动作，数据面保留知情权，两面不冲突。

数据面的包装格式也顺带看一眼：<teammate-message teammate_id="ada" color="red" summary="...">。和第五章的 <task-notification> 一样，这是"伪装成 user 消息的非用户输入"，靠 XML 标签让模型识别说话的是谁。color 甚至会传进 UI，让领队屏幕上每个队友的发言带自己的颜色——协议为人和模型各服务一半。

设计启示

给开发者"数据面/控制面分离"是网络设备的经典架构，在 LLM 系统里有了新含义：模型的上下文窗口是数据面，代码路径是控制面。设计多 Agent 协议时对每种消息问一句：模型读到它能做什么？能行动的进上下文，不能行动的走代码，需要知情又需要代码动作的双面投递。这个三分法比"全部塞给模型"（浪费+误导）和"全部代码处理"（模型失去决策权）都更精确。

6.4 协议实战——权限瞬移与优雅关机

传输层（6.1）和分拣器（6.3）就位后，真正有产品价值的东西是跑在上面的协议。挑两个最完整的看：一个解决"多进程的权限弹窗该弹在哪"，一个解决"怎么体面地解散团队"。

6.4.1 权限瞬移——跨进程 RPC，传输层是两封信

场景：tmux 窗格里的队友要跑 rm -rf build/，权限系统要求用户确认。弹窗弹在队友自己的窗格里？用户正盯着领队的窗格，根本看不见，队友一卡就是十分钟。Claude Code 的答案是把弹窗瞬移到领队的屏幕上：

时序图 权限请求的跨进程往返 — 一次 RPC，传输层是两封 JSON 信
权限瞬移的完整往返：队友挂起工具调用并注册回调，请求信 1 秒内到达领队并弹出与本地完全一致的权限 UI，用户的决定原路写回，回调 resolve，工具从挂起处继续。整套跨进程 RPC 没有一个 socket。

两个细节把这个协议从"能用"提升到"好用"。其一，领队侧不是弹一个简陋的"允许 ada 执行 Bash？"文本框，而是把请求灌进本地权限系统的标准队列（ToolUseConfirmQueue），复用全部原生 UI——Bash 命令高亮、文件编辑 diff 视图，只多一个"来自 ada"的徽章。跨进程的请求和本地的请求在用户眼里没有任何区别，这是第七章权限系统统一入口的直接红利。其二，响应消息携带 updated_input 和 permission_updates——用户在领队屏幕上不仅能批准，还能修改命令再批准、能勾选"以后都允许"，这些决定会通过 team_permission_update 广播给全队生效。

顺着这条线还能看到信任模型的边界，源码处理得很诚实：信箱文件谁都能写，from 字段是自愿声明，所以关键协议都做了来源校验——计划审批只认 from === 'team-lead'（"prevent teammates from forging approvals"），模式下发同样只认领队。但这层校验防的是混乱（队友互相误发、模型幻觉出一条审批），不是恶意——同一台机器、同一个用户账户下的进程本来就在同一信任域里，真正的安全边界在第七章的权限系统，不在信箱。

6.4.2 优雅关机——对一个 LLM，你不能直接 kill

解散团队时，领队对每个队友发的不是 SIGKILL，是一封请求信，然后是一次三步握手：

优雅关机握手（leader ⇄ teammate，全程走信箱）

① leader → shutdown_request { requestId, reason }
     队友的等待循环里它有插队权（6.2），不会被聊天消息淹没；
     但它走"双面投递"——最终决定权在队友的模型手里。

② 队友模型读到请求，检查手头工作，二选一：
     approveShutdown → shutdown_approved { paneId, backendType }
     rejectShutdown  → shutdown_rejected { reason: "还有未提交的改动" }

③ leader 收到 approved 后才动手（useInboxPoller.ts:677）：
     backend.killPane(paneId)      按队友自报的窗格号杀进程
     removeTeammateFromTeamFile    从名册除名
     unassignTeammateTasks         回收它名下的任务，重新可指派
     聚合一条 teammate_terminated  通知领队模型与 UI

为什么要这么绕？因为被关的是一个正在思考的 LLM，不是一个无状态进程。它可能正拿着改到一半没提交的代码、写到一半的调研结论；直接 kill，这些就全丢了（第五章的 Worker 被杀有部分结果抢救，但那是单进程内存里的便利，跨进程没有）。把"能不能停"设计成一个问句，让队友自己收尾再交回控制权，代价只是一封信的往返，换来的是永远不会杀掉一个手里有活的 Agent。拒绝也不是终点——拒绝理由会出现在领队的上下文里，领队模型可以催、可以等、可以改主意。

③ 里还有一个不起眼但关键的动作：回收任务。Team 与 TaskList 一一对应（6.1），队友退场时它名下的任务自动解除指派，回到可认领状态——否则任务会被一个已经不存在的人永远占着。名册、任务、进程三份状态在关机这一刻同步清理，这是"Team = TaskList" 这个 1:1 设计最实际的回报。

设计启示

给开发者协调器（第五章）与团队（本章）是多 Agent 的两种拓扑，差异全在"成员是什么"：Worker 是一次性的执行器——跑完即死、无身份、被动接活；队友是常驻的对等体——有名字、能互聊、闲时等活、退场要谈判。前者适合可拆解的单一大任务，后者适合边界清晰的长期分工。注意两者共用同一个 SendMessage、同一套"消息进对方下一轮输入"的语义——拓扑变了，通信原语一个没换。

给产品经理"权限弹窗瞬移到领队屏幕"是这一章最值得抄的产品决策：多 Agent 系统的审批必须收敛到用户正在看的那一块屏幕，否则并行度越高，用户越像在玩打地鼠。

第七章

权限与安全

实现目标：不同场景下工具的使用需要不同级别的授权，SubAgent 不能越权操作。

Agent 干活的每一步都是一次工具调用：读文件、跑测试、改代码、删目录。现在问一个绕不开的问题——模型发出了 rm -rf build/，执行，还是不执行？每一步都弹窗问用户，三五个回合之后用户必然烦到点下"全部允许"，安全归零；从来不问，一条幻觉出来的 rm -rf ~ 就是不可逆的灾难。权限系统要做的，是在"烦"和"险"之间造出一个精确的调节器——这是所有 Agent 产品都绕不开的设计题。

Claude Code 的答案是一条判定流水线：每个工具调用在执行前穿过一串顺序固定的关卡——PreToolUse 钩子、deny 规则、工具自查、安全检查、权限模式、allow 规则——在任何一关得出结论就立即离开流水线；走完全程仍无结论的，一律去问用户。无论多复杂的场景，系统最终只输出三个词之一：allow、deny、ask。这条流水线的主干在 utils/permissions/permissions.ts（约 1500 行），但触角伸得很远：Bash 一个工具的命令匹配就有 2600 行（bashPermissions.ts），文件路径判定又是约 1500 行（filesystem.ts）——权限是 Claude Code 里代码密度最高的子系统之一，因为它是唯一一个"判断错一次就可能不可逆"的子系统。

架构总图 权限判定流水线：一次工具调用的旅程 — 否决类关卡永远排在豁免类关卡之前
一次工具调用的完整判定旅程。流水线在任何一关得出 allow/deny 即短路退出；④ 里的三类"豁免免疫"判断即使在 bypassPermissions 模式下也必须弹窗。走完全程仍无结论的调用，兜底一律是 ask——fail-closed。

7.1 判定流水线——顺序即安全语义

先看主干代码的骨架。hasPermissionsToUseToolInner 一共九步，编号是源码注释里的原编号：

// hasPermissionsToUseToolInner —— 精简（permissions.ts:1158）
if (context.abortController.signal.aborted) throw new AbortError()  // 入口先查中止（7.5）

// 1a. 整个工具被 deny 规则禁用             → deny，游戏结束
// 1b. 整个工具有 ask 规则                 → ask（可被沙箱自动放行的 Bash 例外）
// 1c. 问工具自己：tool.checkPermissions(parsedInput, context)
//     Bash 拆子命令逐条判，文件工具判路径（7.2）
// 1d. 工具自查说 deny                     → deny
// 1e. 工具要求人机交互（AskUserQuestion）   → ask，bypass 模式也拦
// 1f. 内容级 ask 规则（Bash(npm publish:*)）→ ask，bypass 模式也拦
// 1g. 安全检查（.git/ .claude/ shell 配置） → ask，bypass 模式也拦

// 2a. bypassPermissions 模式              → allow
// 2b. 整个工具被 allow 规则放行            → allow

// 3. 走到这里还没有结论（passthrough）      → 一律转成 ask
return { ...toolPermissionResult, behavior: 'ask' }

这段代码最重要的信息不在任何一行里，而在行与行的顺序里。三个设计判断值得逐个展开：

为什么 deny（1a）排在 bypass（2a）之前？因为 deny 规则是用户亲手写下的否决票——"这个工具永远不许碰"。bypassPermissions 豁免的是"默认要问"的麻烦，不能豁免"用户明说不行"的禁令。把否决判断放在豁免判断之前，"切到 bypass 模式赶进度"就永远不会误伤你 deny 掉的生产数据库。顺序颠倒一下，整个安全模型就塌了——这就是"顺序即语义"。

为什么 1e/1f/1g 三类 ask 也排在 bypass 之前？它们是三种"豁免免疫"：用户显式配置的内容级 ask 规则（源码注释："This must be respected even in bypass mode, just as deny rules are respected"）；天生需要人回答的交互工具；以及安全检查——对 .git/、.claude/、shell 配置文件这类路径的写入。最后一类保护的其实是权限系统自己：settings.json 里存着规则，钩子脚本开机即运行，改一行，下次启动时整个权限系统就被架空了。看门人可以放行任何人，唯独不能让人改门锁。

为什么兜底（3）是 ask 不是 allow？任何一关都没表态的调用，说明它不在任何已知的安全清单里——未知即危险，fail-closed。第三章 buildTool 见过同款哲学（一句话回顾：工具不声明 isConcurrencySafe 就按"不可并发"处理，宁可慢不可错）。权限这里是：宁可多问一次，不可放过一次。

设计启示

给开发者权限判定的正确形状是一条有序短路链：否决类判断在最前，豁免类判断在中间，兜底分支永远保守。审计一个权限系统不必读完实现，先把它的判定顺序表画出来，找"有没有一条路径能绕过否决"——找到即漏洞。Claude Code 把三类"bypass 也拦"的判断显式排在 bypass 之前，这个顺序本身就是可以被单元测试锁死的安全属性。

7.2 规则语言——Tool(specifier) 与复合命令

流水线里反复出现的"规则"长什么样？它是一门刻意保持简单的小语言，格式只有两种：Tool（管整个工具）和 Tool(content)（管这个工具的某类输入）：

'Read(~/.ssh/**)'              → 文件工具：路径 glob
'Bash(python -c "print\(1\)")' → 括号转义后精确匹配

  regexPattern = regexPattern.slice(0, -3) + '( .*)?'
}

每条规则挂一个行为（allow / ask / deny）和一个来源。来源共八层：五层设置文件（userSettings 用户全局、projectSettings 项目共享、localSettings 项目本地不入库、flagSettings 命令行 --settings、policySettings 管理员托管），加三个运行时源（cliArg、command、session 本次会话）。关键的设计选择是：来源不决定优先级，行为才决定优先级——所有来源的规则摊平后统一按 deny → ask → allow 的顺序查（7.1 的 1a/1b/2b），一条 session 级的 deny 照样压过 userSettings 里的 allow。来源真正决定的是两件事：规则展示时标注"来自哪里"，以及用户点"总是允许"时新规则写回哪个文件。

最难的部分在 Bash。一条 cd src && npm test > out.log 怎么判？答案是拆开来，逐个判，一票否决：

机制图 复合命令的判定：拆分 → 逐个匹配 → 最严的说了算 — 全 allow 才 allow，一个 deny 即 deny
复合命令按连接符拆成子命令逐个判定，聚合时取最严结果。弹窗上用户看到的是"哪几段没过关"，而不是笼统的"这条命令需要审批"——精确到段，用户才只为真正的风险付注意力。

拆分-聚合之外，Bash 匹配还有一批对抗性的细节：判定前剥掉注释行和 timeout 这类"安全包装"（stripSafeWrappers），防止危险命令靠伪装混过前缀匹配；LD_*/DYLD_*/PATH 开头的环境变量赋值被单独盯防（BINARY_HIJACK_VARS），因为它们能把后面任何"无害命令"劫持成任意代码。还有一层防呆：Bash(python:*) 这样的 allow 规则看似具体，实则等于放行任意代码执行——dangerousPatterns.ts 维护了一张解释器/包运行器/远程执行的前缀清单（python、node、npx、bash、ssh……），这类过宽规则在进入自动化程度更高的模式时会被主动剥除。

设计启示

给开发者权限规则是一门小语言，它最重要的设计约束不是表达力，而是必须能从一次对话里长出来：用户点"总是允许"的瞬间，系统要能自动生成一条宽窄合适的规则（精确命令 → 精确规则，带参命令 → prefix:* 规则）。规则语言越花哨，自动生成越容易出错。

给产品经理"看似具体、实则放行一切"的规则（python:*、ssh:*）是权限系统的暗门，值得像 Claude Code 一样维护一张危险前缀清单主动拦截——用户自己很难意识到 npx:* 和 * 是等价的。

7.3 权限模式——改默认值，不改规则

规则管的是"这一条输入"，模式管的是整个会话的姿态。五个档位（PermissionMode.ts）：

default            规则说了算，规则没意见就问用户
acceptEdits        工作目录内的文件写操作免弹窗（filesystem.ts:1366 的快速通道），
                   Bash、目录外写入照常要问
plan               只读调研，动手前先交计划——约束主要在提示词层（见下）
bypassPermissions  规则之外全放行；deny/ask 规则、安全检查、交互工具依然拦（7.1）
dontAsk            "问 = 拒"：无人值守场景，宁可失败也不挂起等一个不存在的人

用户侧切换：Shift+Tab 循环 default → acceptEdits → plan →（bypass 如可用）→ default
                                     （getNextPermissionMode.ts:34）

五个档位里有两个值得拆开看。plan 模式是权限系统里最"薄"的一档：它几乎不改判定流水线，主要靠一条每轮注入的 plan_mode 附件提醒模型"现在是调研阶段，别动手"，加上一个专门的出口——ExitPlanMode 工具把"开始动手"变成一次显式的用户审批。第五章的规律再次出现（一句话回顾：协调器不是一个类，是一份提示词）：约束能放在提示词层就不放在代码层，因为"什么算调研、什么算动手"本身是语义判断。代码层只补两个硬细节：进 plan 前如果在 bypass 模式，isBypassPermissionsModeAvailable 会记住它，批准计划后无缝恢复（permissions.ts:1270）；plan 文件本身永远可写（checkEditableInternalPath）——不然模型连计划都写不出来。

bypassPermissions 则是被看管最严的一档：管理员可以通过 policySettings 让它整体不可用；服务端还有一个远程熔断开关（bypassPermissionsKillswitch.ts），在第一次查询前检查一次，必要时把整个会话降回受控模式。至于 dontAsk 的"问变拒"，实现位置很讲究——它在外层包装函数的最末尾做转换（permissions.ts:503），注释解释了原因："This is done at the end so it can't be bypassed by early returns"。转换放在入口，任何一条提前 return 的路径都可能漏掉它；放在出口，无论走哪条路径都逃不过。

设计启示

给开发者模式和规则是正交的两层：模式改"没规则时的默认答案"，规则改"这一条的例外"，且规则永远赢。这个分层让"临时切 bypass 赶进度"是安全的——你的 deny 规则还在。设计自己的系统时警惕把两层揉在一起的诱惑（"bypass 就是全放行嘛"），那会让每次切模式都变成一次全量安全审计。

给产品经理档位是给用户的信任度旋钮，但旋钮拧到头也要保留几条不可豁免的红线（安全检查、显式 ask 规则）。用户会手滑、会忘记自己在哪个模式——红线是产品替用户兜的底。

7.4 弹窗之后——批准、写回与钩子的边界

流水线输出 ask 之后发生什么？useCanUseTool（useCanUseTool.tsx）接手，按运行形态把这个"问题"路由到三条去路：

机制图 ask 的三条去路与"总是允许"的写回 — 弹窗是入口，规则文件是出口
ask 按运行形态路由：协调器场景先让自动检查代审；swarm 队友把请求转发到领队屏幕（第六章的信箱协议）；本地则进弹窗队列。用户点"总是允许"时，附带的规则建议被持久化——下次同类调用直接在 7.1 的 2b 步放行。

弹窗上的四个选项里藏着两个产品级细节。"修改后允许"：用户可以直接编辑命令再批准，修改后的输入通过 updatedInput 一路替换到工具执行——不必"拒绝、口头纠正、等模型重试"绕一大圈。"总是允许"：ask 决策附带 suggestions（一组 PermissionUpdate），用户勾选即写回对应的设置文件。deny 同样有讲究：拒绝理由（decisionReason）会变成一条给模型看的消息（createPermissionRequestMessage），模型知道"为什么被拒"，才能换一条合规的路走，而不是原地重试。

最后是钩子。PreToolUse 钩子在权限判定之前运行（toolExecution.ts:800），它可以表态 allow/deny/ask，也可以只修改输入。它的权限边界由 resolveHookPermissionDecision（toolHooks.ts:332）一锤定音：

// resolveHookPermissionDecision —— 精简（toolHooks.ts:332）
// 源码注释：hook 'allow' does NOT bypass settings.json deny/ask rules

if (hook.behavior === 'allow') {
  const ruleCheck = await checkRuleBasedPermissions(tool, input, ctx)
  if (ruleCheck === null)            return hook       // 跳过弹窗，直接执行
  if (ruleCheck.behavior === 'deny') return ruleCheck  // deny 规则压过钩子
  return await canUseTool(...)                         // ask 规则：照样弹窗
}
if (hook.behavior === 'deny') return hook              // 钩子的否决立即生效
// 钩子说 ask 或沉默 → 走正常流水线（带上钩子给的理由进弹窗）

注意这个不对称：钩子的 deny 立即生效，allow 却要再过一遍规则。钩子是用户写的自动化脚本，它有资格替用户拒绝（拒绝永远安全），但无权替用户推翻用户自己写的 deny/ask 规则——两个"用户的意志"冲突时，显式配置的规则赢过运行时的脚本。

设计启示

给开发者钩子是自动化的用户，不是超级用户：任何自动批准通道（钩子、AI 代审、CI 机器人）的权限上限必须等于它代替的那个人，且永远低于显式否决。校验方法同 7.1——找"这条通道能不能绕过 deny"。

给产品经理弹窗的价值密度取决于"总是允许"的设计：一次弹窗应该顺手消灭一类未来弹窗，否则你是在训练用户无脑点确认——权限疲劳的终点是权限失效。

7.5 中止信号——Esc 如何传遍整棵 Agent 树

权限管"能不能开始"，中止管"能不能停下"。用户按下 Esc，此刻可能有一个模型请求在流式返回、三个工具在并发执行、一个 SubAgent 在深处调研、还有一个权限弹窗挂着等回答——全都要停，而且停完之后对话还得是合法状态、还能继续聊。Claude Code 用一件标准武器解决：AbortController，但用法有讲究。

机制图 中止信号的树形传播 — 单向下行：父停子必停，子停不连累父
中止信号从回合级 AbortController 单向下行：断流、杀进程、收弹窗、停同步子树；异步 SubAgent 故意不在传播树上。最关键的一步是善后——为悬空的 tool_use 合成配对的 tool_result，保住 API 协议不变量。

整套机制里最值得学的是善后这一步。第一章讲过 API 的硬性协议（一句话回顾：每个 tool_use 块必须有一个配对的 tool_result 块，以 user 角色注入下一轮）。用户在工具跑到一半时按 Esc，消息数组里就留下了没有结果的 tool_use——这个状态是不合法的，直接拿去调 API 会报错。所以 query 循环中止后做的第一件事，是给每个悬空的 tool_use 合成一条 "Interrupted by user" 的 tool_result（query.ts:1015 起的中止处理），再补一条 [Request interrupted by user] 的用户消息。代价是几行合成代码，换来的是：中止不是会话的死刑，只是一个回合的句号——用户下一句话可以直接接着聊。

另一个容易被低估的决定是作用域：AbortController 是回合级的（REPL 每个回合新建一个），不是会话级的。中止是"取消这一次"，不是"关掉这一切"——作用域选错，Esc 就从"后悔药"变成了"自毁按钮"。而同步与异步 SubAgent 的区别对待（共享 vs 独立 controller）延续了第四章的语义（一句话回顾：run_in_background: true 的派发即刻返回 task_id，后台独立运行）：同步派生是父任务的一部分，父停子必停；异步派生是独立的后台任务，它的生死应该由显式的 TaskStop 管理，而不是被父回合的 Esc 连坐。

设计启示

给开发者"取消"要当成协议特性设计，不是异常处理，三条铁律：①信号单向下行（子的失败不该炸掉父）；②每个挂起点都注册响应（模型流、工具进程、权限弹窗——漏一个就有"停不下来"的角落）；③中止后的状态必须仍然合法（合成 tool_result）。市面上"一按停止就只能重开会话"的 Agent，九成是丢了第③条。

第八章

Session 持久化与恢复

实现目标：进程崩溃或用户中断后，能从上次中断点恢复，不丢失工作进度。

写代码写到一半，终端崩了，或者你直接合上电脑走人。第二天敲一句 claude --continue，对话原样回来：聊天记录、改到一半的任务清单、甚至你当时 cd 进去的那个 worktree 目录。凭什么？

第一章埋下的种子在这里收获（一句话回顾：LLM 本身无状态，Agent 的全部状态就是一个只增不减的消息数组）。既然状态只有一个数组，那持久化 = 把数组落盘，恢复 = 把数组读回来——理论上就这么平凡。工程上不平凡的是三个问题：怎么落——每一轮都在写，崩在任何一行都不能丢掉前面的；落成什么形状——用户会撤销、会分叉、会从任意历史点重开，线性列表装不下这些；装不下怎么办——数组只增不减，上下文窗口有硬顶，总有一天塞不进一次 API 调用。

本章顺着这三个问题走：8.1 落盘（append-only JSONL 与消息树），8.2 读回（从叶子回走一条链，顺带恢复整个工作环境），8.3 装不下（Compact：让模型给自己写交接文档）。

架构总图 三层存储：磁盘全量、内存工作集、窗口预算 — 写是 append，读是走链，compact 是换页
Session 系统的全景：内存里的消息数组每轮追加落盘成 JSONL 全量日志；恢复时从树的叶子回走一条链重建数组；数组超出窗口预算时 compact 把工作集换页——磁盘上的历史永远不动。

8.1 转录文件——append-only JSONL 与消息树

先看磁盘上的东西长什么样（utils/sessionStorage.ts，全书最大的单文件之一，约 5100 行）：

~/.claude/projects/-Users-you-myapp/       # 项目目录（cwd 转义成目录名）
  9f3c….jsonl                              # 一个会话 = 一个文件，一行一个 Entry
  9f3c…/subagents/agent-a1b2.jsonl         # 子 Agent 的侧链转录（第四章的隔离，落盘也隔离）

# 消息 Entry：uuid + parentUuid 把消息连成一棵树
{"type":"user",     "uuid":"u-17","parentUuid":"a-16","message":{…}}
{"type":"assistant","uuid":"a-18","parentUuid":"u-17","message":{…}}

# 元数据也是 Entry，混在同一个文件里，同一支笔追加
{"type":"custom-title","sessionId":"9f3c…","customTitle":"修复登录超时"}
{"type":"worktree-state","worktreeSession":{…}}    # 你此刻在哪个 worktree
{"type":"file-history-snapshot", …}                # 文件改动快照（/rewind 的本钱）
{"type":"mode","mode":"normal"}                    # coordinator / normal

三个设计决定，每个都值得停一下。第一，只追加，从不改写。写入口 recordTranscript()（sessionStorage.ts:1408）拿到一批消息后，先按 UUID 对照已落盘集合去重，只把新消息追加到文件尾。好处是崩溃语义简单到极致：进程死在任何时刻，最多废掉正在写的半行 JSON，之前的每一行都完好——不需要 WAL，不需要事务，文件系统的 append 就是事务。去重则让"重复写"无害，调用方不必小心翼翼地记"哪些已经存过了"。

第二，存的是树，不是列表。每条消息带 parentUuid 指向它的前一条。正常聊天时这是一条直线；但用户 /rewind 回某个历史点重新说话时，新消息的 parentUuid 指向那个历史点——树上长出一根新枝，旧枝原封不动地留在文件里。"当前对话"不过是"从某片叶子回走到根的那条链"。撤销、分叉、回放，全都不需要删除任何东西——历史不可变，可变的只是"现在站在哪片叶子上"。

第三，元数据与消息同文件。标题、模式、worktree 状态、文件快照统统作为 Entry 混在同一个 JSONL 里，而不是旁边放一个 meta.json。一个文件就是一个会话的全部事实，复制走一个文件 = 带走整个会话；更重要的是原子性——不存在"消息文件和元数据文件对不上"这种双文件系统的经典事故。

机制图 消息树：历史不可变，分叉只换叶子 — 文件里是一棵树，对话是其中一条链
rewind/fork 不删除历史，只是让新消息指向旧的分叉点。文件永远只追加；树形结构完全由每行的 parentUuid 编码。当前对话是从某片叶子回走到根的一条链。

这就是第一章那句"状态与计算分离"承诺的红利兑现（一句话回顾：模型无状态，状态全在调用方的数组里）：因为状态只是数据，所以它可持久化、可回放、可分叉——换任何一种"状态藏在对象和闭包里"的架构，这三件事每一件都是大工程。

设计启示

给开发者日志式存储三件事一起白拿：持久化、审计、时间旅行——全部来自"只追加、不改写"这一条纪律。给状态建模时先问一句：历史需要不可变吗？需要，就存事件日志而不是最新快照；快照永远可以从日志推出来，反过来不行。

8.2 恢复机制——从叶子回走一条链

恢复是落盘的逆运算，核心函数小得出奇（buildConversationChain，sessionStorage.ts:2069）：

// buildConversationChain —— 精简：从叶子沿 parentUuid 回走到根，再反转
const transcript: TranscriptMessage[] = []
const seen = new Set()
let cur: TranscriptMessage | undefined = leafMessage
while (cur) {
  if (seen.has(cur.uuid)) break        // 环保护：记录事件，返回部分链
  seen.add(cur.uuid)
  transcript.push(cur)
  cur = cur.parentUuid ? messages.get(cur.parentUuid) : undefined
}
transcript.reverse()
// 补遗 pass：并行工具调用会形成 DAG（N 个 tool_use 兄弟消息共享
// 一个 message.id），单亲回走只保得住一支——读侧把兄弟找回来
return recoverOrphanedParallelToolResults(messages, transcript, seen)

存的时候是树，读的时候是链。--continue 取最新的叶子；--resume 弹列表让用户挑（每个会话只读文件尾 16KB 的元数据窗口来显示标题，不必解析整个文件）；--fork-session 则把链复制到一个新 sessionId 下继续——原会话不动，这是会话级的"另存为"。三个入口共享同一条回走逻辑，差别只在"选哪片叶子"和"要不要换身份"。

但只把消息读回来，恢复才完成一半。processResumedConversation（sessionRestore.ts:409）还原的是整个工作现场：任务清单不单独存文件，而是从转录里挖——扫描最后一次 TodoWrite 工具调用的入参（extractTodosFromTranscript），转录本身就是数据库；worktree 状态更妙，恢复时直接 process.chdir(worktreePath)，源码注释点破："process.chdir is the TOCTOU-safe existence check"——切目录这个动作同时兼任存在性检查，目录被删了就抛 ENOENT，顺势把状态改记成"已退出"，不需要"先检查再切换"的竞态两步；此外还有 agent 身份与模型覆盖、成本统计、会话标题，逐项归位。

防御性细节也都是生产事故喂出来的：转录读取有 50MB 硬顶（MAX_TRANSCRIPT_READ_BYTES，注释记着案发编号：会话文件能长到数 GB）；大文件在文件描述符层直接跳过 compact 边界之前的字节——峰值内存按输出大小分配而不是按文件大小（一个 151MB、84% 是过期快照的会话，加载内存从 316MB 降到 155MB）；回走时的 seen 集合防 parentUuid 成环——文件是外部输入，哪怕是自己写的，读的时候也当不可信数据对待。

设计启示

给开发者恢复的完整定义是"用户回来之后，不需要向系统重新解释任何事"。消息数组只是及格线；真正拉开差距的是环境状态——cwd、任务清单、后台任务、半途的计划。列一张"崩溃前用户拥有什么"的清单，逐项回答两个问题：存在哪？怎么恢复？凡是答不上来的项，就是用户将来要骂的地方。另外注意 Claude Code 的选择：todos 这类"可以从消息推导的状态"就不单独存——单一事实源，能推导就不冗余。

8.3 Compact——模型给自己写交接文档

数组只增不减，窗口有硬顶，冲突迟早发生。先看什么时候触发（autoCompact.ts）：

有效窗口   = 模型上下文窗口 − min(该模型最大输出, 20_000)   # 给摘要留产出空间
自动压缩线 = 有效窗口 − 13_000                             # AUTOCOMPACT_BUFFER_TOKENS
警告线     = 压缩线 − 20_000                               # UI 提前亮黄灯
手动硬顶   = 有效窗口 − 3_000                              # /compact 的最后余地

连续失败 3 次熔断                    # 有会话曾连续重试 3272 次，白烧 25 万次 API 调用/天

每个数字都在回答"压缩这件事自己也要花钱"：压缩要调一次模型、产出最多 2 万 token 的摘要，所以阈值必须在真正的硬顶之前留够余量；压缩可能失败（上下文已经大到连"请你总结"这个请求都发不出去），所以有专门的重试策略（掐掉最老的几轮再试）和熔断器。

触发之后，compactConversation()（compact.ts:387）做的事可以概括成一句话：让模型给"下一个自己"写交接文档。压缩提示词要求摘要覆盖九个板块——Primary Request and Intent（用户到底要什么）、Key Technical Concepts、Files and Code Sections（动过哪些文件，附关键代码片段）、Errors and Fixes（犯过什么错、用户怎么纠正的）、Problem Solving、All User Messages（用户原话全录）、Pending Tasks、Current Work、Optional Next Step（下一步，必须引用原文防跑偏）。为什么让模型自己写？因为"这 15 万 token 里什么重要"是语义判断——只有正在干活的模型知道哪句用户反馈是转折点、哪次报错是死胡同。

机制图 Compact 前后：换页，不是清空 — 磁盘全量不动，内存工作集重建
Compact 用一次模型调用把 180K 的历史换成 20K 的交接文档，再按预算重建工作集：重读最近碰过的文件、恢复计划与技能上下文、重播工具清单。转录文件不删任何东西，boundary 只是让"当前对话"从新页开始。

图右侧的灾后重建是最容易被忽略、又最影响体验的一半。摘要写完后上下文被清空，模型对文件系统的"工作记忆"（readFileState）也没了——不重建的话，它醒来第一件事就是把刚才读过的文件挨个重读一遍，压缩省下的 token 立刻还回去。所以 compact 之后自动重注入：最近读过/改过的至多 5 个文件（总预算 50K token、单文件 5K）、进行中的 plan 文件、plan 模式提醒（不然模型压缩完就忘了自己在 plan 模式里）、用过的 skills，以及把工具/Agent/MCP 清单对"空历史"重播一遍——第二章的增量通告机制（一句话回顾：工具清单只在变化时以 delta 附件通告）在空历史上 diff，天然等于全量重播，一行特殊代码都不用加。

全量压缩之外还有两档更轻的手段，构成一个梯度：microcompact 不调模型，直接把老的工具结果原地替换成一行 [Old tool result content cleared]——只清 Read/Bash/Grep 这类可再生的内容（想要随时能重跑），保留最近 N 个；再往下沉是 API 侧的 context editing（clear_thinking / clear_tool_uses 策略），擦除动作直接发生在服务端。三档的成本阶梯很清晰：先擦可再生的（免费），再摘要不可再生的（一次模型调用），实在不行才掐头截断（有损）——每一层都比下一层便宜，也都比下一层保真。

设计启示

给开发者上下文管理的正确心智模型是操作系统的换页：磁盘（转录）保全量，内存（窗口）只装工作集，压缩是把冷页换出、按需换回（重读文件）。两个可直接抄的决定：摘要让模型自己写（重要性是语义判断），压缩后必须重建工作集（否则省下的 token 会被重读吃回去）。

给产品经理压缩应该在阈值前无感发生，留给用户的感知面只有一条状态：上下文余量。警告线、压缩线、硬顶三级设计的本质是：永远不让用户撞上"发不出去"的那堵墙——撞墙一次，用户对长对话的信任就清零一次。
